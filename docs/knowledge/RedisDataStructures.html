Redis核心数据结构

Redis是一款高性能的内存数据结构存储系统，支持多种数据类型，广泛应用于缓存、消息队列、排行榜等场景。单线程QPS可达10万+，支持持久化、主从复制和分布式集群。

🎯

Redis核心特点

  • 基于内存：纯内存操作，速度极快
  • 单线程模型：避免上下文切换，简化并发控制
  • IO多路复用：使用epoll处理高并发连接
  • 多种数据结构：String、Hash、List、Set、ZSet等
  • 持久化机制：支持RDB快照和AOF日志
  • 高可用方案：主从复制、哨兵、集群模式

String 字符串

底层实现：SDS（Simple Dynamic String）

SDS是Redis自定义的动态字符串结构，相比C语言字符串，具有获取长度O(1)、避免缓冲区溢出、减少内存重分配次数等优点。

应用场景

  • 缓存对象：将对象序列化为JSON字符串存储
  • 计数器：使用INCR/INCRBY实现原子自增
  • 分布式锁：使用SETNX实现互斥锁
  • 分布式ID生成：使用INCR生成唯一递增ID
Redis
# 设置值（缓存对象）
SET user:1 '{"name":"张三","age":25}'

# 分布式锁
SETNX lock:order 1              # 不存在才设置
SET lock:order 1 EX 30 NX       # 设置并添加过期时间（推荐）

# 计数器
INCR page:view:home             # 自增
INCRBY article:read:123 5       # 增加指定值

# 批量操作
MGET user:1 user:2 user:3       # 批量获取
String类型的值最大可存储512MB，但实际应用中建议控制单个value大小在10KB以内，避免big key问题。

Hash 哈希

底层实现

小数据量时使用ziplist（压缩列表），大数据量时转换为hashtable（哈希表）。

应用场景

  • 对象存储：比String更灵活，可以只修改某个字段
  • 购物车：用户ID作为key，商品ID作为field，数量作为value
Redis
# 存储用户对象
HSET user:1 name "张三" age 25 email "zhangsan@example.com"
HGET user:1 name                # 获取单个字段
HGETALL user:1                  # 获取所有字段
HMSET user:1 name "李四" age 26 # 设置多个字段

# 购物车实现
HSET cart:user:1 item:1001 2        # 商品1001，数量2
HINCRBY cart:user:1 item:1001 1     # 增加数量
HDEL cart:user:1 item:1001          # 删除商品
HLEN cart:user:1                    # 购物车商品数量
对比项String存储对象Hash存储对象
灵活性修改字段需整体更新可单独修改某个字段
内存占用较大较小（字段复用key空间）
适用场景整体读取，较少修改频繁修改部分字段

List 列表

底层实现：quicklist（ziplist + linkedlist）

Redis 3.2之前使用ziplist或linkedlist，3.2之后使用quicklist，结合了两者的优点：内存占用小且支持快速插入删除。

应用场景

  • 消息队列：生产者-消费者模式
  • 最新列表：最新文章、最新消息
  • 时间线：用户动态时间线
Redis
# 消息队列（生产者-消费者）
LPUSH queue:order '{"orderId":123}'   # 左边推入（生产者）
RPOP queue:order                       # 右边弹出（消费者）
BRPOP queue:order 10                   # 阻塞弹出，10秒超时

# 最新文章列表
LPUSH articles:latest article:100
LTRIM articles:latest 0 99             # 只保留最新100篇
LRANGE articles:latest 0 9             # 获取最新10篇

# 时间线
LPUSH timeline:user:1 tweet:1001
LPUSH timeline:user:1 tweet:1002
使用BRPOP/BLPOP实现阻塞式消息队列时，超时时间建议设置在5-30秒之间，太短会增加网络开销，太长会影响系统响应。
🎲

Set 集合

底层实现

整数集合（intset）或哈希表（hashtable），自动根据数据类型和规模选择最优实现。

核心特性

  • 无序：元素没有顺序
  • 不重复：自动去重
  • 集合运算：支持交集、并集、差集

应用场景

  • 标签系统：给用户打标签
  • 共同关注：使用交集找出共同好友
  • 抽奖：随机抽取不重复用户
Redis
# 用户标签
SADD user:1:tags "90后" "程序员" "游戏"
SMEMBERS user:1:tags              # 获取所有标签
SISMEMBER user:1:tags "程序员"    # 判断是否有某标签

# 共同关注（交集）
SADD user:1:following user:2 user:3 user:4
SADD user:2:following user:3 user:4 user:5
SINTER user:1:following user:2:following  # 共同关注: user:3, user:4
SDIFF user:1:following user:2:following   # 差集: user:2
SUNION user:1:following user:2:following  # 并集: user:2,3,4,5

# 抽奖
SADD lottery:2024 user:1 user:2 user:3
SRANDMEMBER lottery:2024 1         # 随机抽1人（不移除）
SPOP lottery:2024 1                # 随机抽1人（移除）
命令功能示例
SINTER交集SINTER set1 set2
SUNION并集SUNION set1 set2
SDIFF差集SDIFF set1 set2
SRANDMEMBER随机获取（不移除）SRANDMEMBER key count
SPOP随机获取（移除）SPOP key count

ZSet 有序集合

底层实现：跳表（skiplist）+ 哈希表

小数据量使用ziplist，大数据量使用skiplist+dict组合。跳表实现O(logN)的插入和查询，同时支持范围查询。

graph TB subgraph "跳表结构示意" L4["L4: head ----------------------------------------> tail"] L3["L3: head ----------> 30 -------------------------> tail"] L2["L2: head --> 10 --> 30 --> 50 -------------------> tail"] L1["L1: head --> 10 --> 30 --> 50 --> 70 --> 90 --> tail"] end L4 --> L3 --> L2 --> L1

应用场景

  • 排行榜：游戏分数排行、热榜
  • 延时队列：使用score作为执行时间戳
  • 范围查询：按分数范围获取数据
Redis
# 排行榜
ZADD leaderboard:score 100 user:1 200 user:2 150 user:3
ZINCRBY leaderboard:score 50 user:1      # 增加分数
ZREVRANGE leaderboard:score 0 9 WITHSCORES  # 降序前10名
ZRANK leaderboard:score user:1           # 获取排名（升序）
ZREVRANK leaderboard:score user:1        # 获取排名（降序）

# 延时队列（订单超时取消）
# score为订单过期时间戳
ZADD delay:queue 1704067200 order:123
# 消费者定时检查
ZRANGEBYSCORE delay:queue 0 {当前时间戳} LIMIT 0 100
ZREM delay:queue order:123               # 处理后移除
ZRANGEBYSCORE命令非常适合实现延时队列，消费者通过定时轮询获取已到期的任务进行处理。

持久化机制

RDB（Redis Database）

将内存数据以二进制快照的形式保存到磁盘。

  • 优点：文件紧凑、恢复速度快、适合备份
  • 缺点：可能丢失最后一次持久化后的数据
CONFIG
# RDB触发条件
save 900 1      # 900秒内至少1次修改
save 300 10     # 300秒内至少10次修改
save 60 10000   # 60秒内至少10000次修改

# 手动触发
SAVE            # 阻塞持久化
BGSAVE          # 后台持久化（fork子进程）

AOF（Append Only File）

记录所有写命令到日志文件，重启时重新执行命令恢复数据。

  • 优点：数据更安全，最多丢失1秒数据
  • 缺点：文件更大，恢复速度较慢
CONFIG
# AOF配置
appendonly yes
appendfilename "appendonly.aof"

# 刷盘策略
appendfsync always      # 每次写入都刷盘（最安全，最慢）
appendfsync everysec    # 每秒刷盘（推荐）
appendfsync no          # 由操作系统决定（最快，可能丢数据）

# AOF重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

混合持久化（Redis 4.0+）

重写时同时使用RDB和AOF：先写RDB格式的快照，然后追加AOF格式的增量命令。

CONFIG
# 开启混合持久化
aof-use-rdb-preamble yes
特性RDBAOF
数据安全性低（可能丢失分钟级）高（最多丢1秒）
文件大小小（压缩二进制）大（文本命令）
恢复速度快慢
系统资源消耗高（fork时）低（持续写入）
适用场景备份、主从复制数据安全要求高
最佳实践：同时开启RDB和AOF，使用混合持久化，定期备份RDB文件。

缓存问题及解决方案

缓存穿透

问题描述：大量请求查询不存在的数据，绕过缓存直接打到数据库。常见于恶意请求（如id=-1）或查询不存在的用户。

解决方案

方案1：缓存空对象
Java
public Object getWithNullCache(String key) {
    Object value = redis.get(key);
    if (value != null) {
        // 命中缓存（可能是空值标记）
        return "NULL".equals(value) ? null : value;
    }

    // 查询数据库
    value = db.query(key);

    if (value != null) {
        redis.set(key, value, 300);  // 正常过期时间
    } else {
        // 缓存空值，设置较短过期时间
        redis.set(key, "NULL", 60);
    }
    return value;
}
方案2：布隆过滤器

原理：位图 + 多个哈希函数。判断元素可能存在时可能误判，判断元素一定不存在时一定准确。

Java
public Object getWithBloomFilter(String key) {
    // 先检查布隆过滤器
    if (!bloomFilter.mightContain(key)) {
        return null;  // 一定不存在，直接返回
    }

    // 可能存在，继续查询缓存和数据库
    Object value = redis.get(key);
    if (value != null) return value;

    value = db.query(key);
    if (value != null) redis.set(key, value);
    return value;
}

缓存击穿

问题描述：热点Key过期，大量请求同时穿透到数据库。与缓存穿透的区别在于：击穿是Key存在但过期了，穿透是Key不存在。

解决方案

方案1：互斥锁
Java
public Object getWithMutex(String key) {
    Object value = redis.get(key);
    if (value != null) return value;

    String lockKey = "lock:" + key;
    try {
        // 尝试获取锁
        boolean locked = redis.setnx(lockKey, "1", 10);
        if (locked) {
            // 获取锁成功，查询数据库
            value = db.query(key);
            redis.set(key, value, 300);
        } else {
            // 获取锁失败，等待后重试
            Thread.sleep(50);
            return getWithMutex(key);
        }
    } finally {
        redis.del(lockKey);
    }
    return value;
}
方案2：逻辑过期

设置Key永不过期（物理不过期），但在value中维护一个逻辑过期时间，过期后异步刷新。

缓存雪崩

问题描述：大量Key同时过期，或Redis宕机，导致大量请求打到数据库。

解决方案

方案1：随机过期时间
Java
public void setWithRandomExpire(String key, Object value, int baseExpire) {
    // 基础过期时间 + 随机时间（0-300秒）
    int randomExpire = baseExpire + ThreadLocalRandom.current().nextInt(0, 300);
    redis.set(key, value, randomExpire);
}

方案3：熔断降级

使用熔断器（如Hystrix/Sentinel），当缓存或数据库压力过大时，降级返回默认值或从本地缓存读取。

缓存雪崩是最严重的缓存问题，可能导致数据库压力过大而宕机，建议结合多种方案进行防护。
