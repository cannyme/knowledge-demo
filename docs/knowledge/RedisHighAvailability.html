Redis高可用方案

Redis 高可用架构从主从复制到哨兵模式再到集群模式，逐步解决单机故障、自动故障转移和海量数据存储问题。

🔄

Redis高可用架构演进

Redis 高可用架构经历了三个阶段的演进：

1. 主从复制（读写分离）

  • 一主多从架构，主节点负责写，从节点负责读
  • 解决了读性能瓶颈，但不具备自动故障转移能力
  • 主节点故障需要手动切换

2. Sentinel 哨兵（自动故障转移）

  • 在读写分离基础上增加了故障监控和自动切换
  • 解决了高可用问题，但存储容量仍受限于单节点
  • 适用于读多写少、数据量可控的场景

3. Cluster 集群（数据分片）

  • 支持数据分片，突破单机内存限制
  • 内置高可用，自动故障转移
  • 适用于海量数据场景

主从复制原理

主从复制包含全量同步和增量同步两个阶段：

全量同步（首次连接）

  1. 从节点发送 SYNC 命令连接主节点
  2. 主节点执行 BGSAVE，生成 RDB 快照文件
  3. 主节点将 RDB 文件发送给从节点
  4. 从节点加载 RDB 文件完成全量同步
  5. 主节点将新写入的命令缓存到复制缓冲区
  6. RDB 发送完成后，主节点发送缓冲区的写命令给从节点

增量同步（断线重连）

  1. 从节点重连后发送 PSYNC 命令，携带复制偏移量
  2. 如果偏移量在主节点复制积压缓冲区内，则进行增量同步
  3. 主节点将积压缓冲区中的命令发送给从节点
  4. 如果偏移量已不在缓冲区，则重新进行全量同步
sequenceDiagram participant S as Slave participant M as Master S->>M: SYNC / PSYNC offset alt 全量同步 M->>M: BGSAVE 生成 RDB M->>S: 发送 RDB 文件 S->>S: 加载 RDB M->>S: 发送缓冲区命令 else 增量同步 M->>S: 发送积压缓冲区命令 end loop 持续复制 M->>S: 传播写命令 end
CONFIG
# 从节点配置
replicaof 192.168.1.100 6379
replica-read-only yes

# 复制积压缓冲区大小（默认1MB）
repl-backlog-size 64mb

# 主节点至少2个从节点同步才认为写成功
min-replicas-to-write 2
min-replicas-max-lag 10

Sentinel 哨兵机制

Sentinel 是 Redis 的高可用解决方案，提供三大功能：

  • 监控：持续监控主从节点运行状态
  • 通知：节点故障时通知管理员
  • 自动故障转移：主节点故障时自动选举新主节点
graph TD subgraph Sentinel集群 S1[Sentinel 1] S2[Sentinel 2] S3[Sentinel 3] end S1 -.->|监控| M[Master] S2 -.->|监控| M S3 -.->|监控| M S1 -.->|监控| R1[Slave 1] S2 -.->|监控| R1 S3 -.->|监控| R2[Slave 2]

主观下线 vs 客观下线

类型定义触发条件
主观下线（SDOWN）单个 Sentinel 认为节点不可用心跳检测超时（默认 30s）
客观下线（ODOWN）多数 Sentinel 认为节点不可用quorum 个 Sentinel 同意

故障转移流程

  1. Sentinel 检测到主节点客观下线
  2. Sentinel 之间选举 Leader（Raft 算法）
  3. Leader 从从节点中选举新主节点
    • 优先选择复制偏移量最大的（数据最新）
    • 相同偏移量选择 runid 最小的
    • 排除已下线的节点
  4. 其他从节点切换复制到新主节点
  5. 通知客户端新主节点地址
  6. 原主节点恢复后成为从节点
CONFIG
# sentinel.conf
sentinel monitor mymaster 192.168.1.100 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel failover-timeout mymaster 180000
sentinel parallel-syncs mymaster 1

# mymaster: 主节点别名
# 2: 客观下线所需的 Sentinel 数量（quorum）
# 30000: 主观下线判定时间（毫秒）

Cluster 集群

Redis Cluster 是分布式解决方案，通过数据分片实现水平扩展：

16384 个哈希槽

  • Redis Cluster 将数据划分为 16384 个槽（slot）
  • 每个 key 通过 CRC16(key) % 16384 计算所属槽位
  • 每个主节点负责一部分槽位
graph TD subgraph Redis集群 M1[Master 1
槽位 0-5460] M2[Master 2
槽位 5461-10922] M3[Master 3
槽位 10923-16383] M1 --> S1[Slave 1] M2 --> S2[Slave 2] M3 --> S3[Slave 3] end

节点通信（Gossip 协议）

  • 节点之间通过 PING/PONG 消息交换状态信息
  • 每个节点保存完整的集群拓扑信息
  • 不需要中心化协调节点
  • 协议特点：最终一致性，适合大规模集群

槽迁移

当需要扩容或缩容时，进行槽迁移：

  1. 目标节点准备接收槽位
  2. 源节点将槽内 key 逐一迁移到目标节点
  3. 迁移过程中，key 在原节点仍可访问
  4. 迁移完成后，更新槽位映射关系
  5. 广播新的槽位分配信息

Hash Tags

使用 Hash Tags 将相关 key 路由到同一槽位，支持批量操作：

REDIS
# 使用 {} 内的内容计算槽位
user:{123}:profile  → 只用 123 计算槽位
user:{123}:orders   → 与上面同一槽位

# 这样可以批量操作
MGET user:{123}:profile user:{123}:orders
CONFIG
# redis.conf
cluster-enabled yes
cluster-config-file nodes.conf
cluster-node-timeout 5000
cluster-require-full-coverage no

# 集群命令
CLUSTER INFO          # 查看集群信息
CLUSTER NODES         # 查看节点信息
CLUSTER MEET          # 添加节点
CLUSTER ADDSLOTS      # 分配槽位
CLUSTER FAILOVER      # 手动故障转移

缓存问题及解决方案

缓存穿透

问题：查询一个数据库和缓存都不存在的数据，每次请求都打到数据库

场景：恶意攻击使用不存在 ID 查询

解决方案：

  • 布隆过滤器：在缓存前加布隆过滤器，快速判断 key 是否存在
  • 缓存空值：数据库不存在也缓存（设置较短过期时间）
  • 参数校验：对非法参数进行拦截
graph TD A[请求] --> B{布隆过滤器} B -->|可能存在| C[查询缓存] B -->|一定不存在| D[直接返回] C -->|命中| E[返回数据] C -->|未命中| F[查询数据库] F --> G[回写缓存] F -->|不存在| H[缓存空值]

缓存击穿

问题：热点 key 突然过期，大量请求同时打到数据库

场景：秒杀活动中的热门商品缓存过期

解决方案：

  • 互斥锁：只允许一个线程重建缓存，其他线程等待
  • 逻辑过期：缓存不设置 TTL，通过逻辑时间判断是否过期，异步重建
  • 热点 key 永不过期：后台定时刷新热点数据
JAVA
// 互斥锁解决缓存击穿
public String getData(String key) {
    String value = redis.get(key);
    if (value == null) {
        // 获取锁
        if (redis.setnx("lock:" + key, "1", 10)) {
            try {
                value = db.query(key);
                redis.set(key, value, 300);
            } finally {
                redis.del("lock:" + key);
            }
        } else {
            // 其他线程等待后重试
            Thread.sleep(100);
            return getData(key);
        }
    }
    return value;
}

缓存雪崩

问题：大量 key 同时过期，或 Redis 宕机，导致数据库压力骤增

场景：批量导入的数据同时设置相同过期时间

解决方案：

  • 随机过期时间：在基础过期时间上增加随机值
  • 多级缓存：本地缓存（Caffeine）+ 分布式缓存（Redis）
  • 熔断降级：数据库压力过大时直接返回默认值
  • 高可用架构：Redis Cluster 避免单点故障
随机过期时间示例
expire = baseExpire + random(0, 300) // 基础过期时间 + 0~5分钟随机值
问题原因核心解决方案
缓存穿透查询不存在的数据布隆过滤器、缓存空值
缓存击穿热点 key 过期互斥锁、逻辑过期
缓存雪崩大量 key 同时过期随机过期、多级缓存
