JVM内存模型与垃圾回收

深入理解JVM内存结构、垃圾回收机制、GC收集器及调优策略，掌握JVM性能优化核心技术。

🏗️

JVM内存结构

JVM运行时数据区主要分为两大区域：线程共享区域（堆、方法区/元空间）和线程私有区域（虚拟机栈、本地方法栈、程序计数器）。

各区域详解
  • 堆（Heap）：存储对象实例，所有线程共享，是垃圾回收的主要区域
  • 元空间（Metaspace）：存储类元数据、常量池、静态变量（JDK8后从方法区移到本地内存）
  • 虚拟机栈：线程私有，每个方法调用创建一个栈帧，包含局部变量表、操作数栈、动态链接、返回地址
  • 本地方法栈：为Native方法服务
  • 程序计数器：记录当前线程执行的字节码指令地址，唯一不会发生OOM的区域
💡
Metaspace演进

JDK 1.6：字符串常量池在方法区（永久代）
JDK 1.7：字符串常量池移到堆中
JDK 1.8：方法区改为元空间，使用本地内存，无默认上限

🗂️

堆内存详解

堆是JVM内存中最大的一块，用于存储对象实例。现代JVM采用分代收集策略，将堆划分为年轻代和老年代。

┌─────────────────────────────────────────────────────────┐ │ 堆 (Heap) │ ├──────────────────────────┬──────────────────────────────┤ │ 年轻代 (Young) │ 老年代 (Old) │ │ 约占 1/3 │ 约占 2/3 │ ├───────┬────────┬─────────┤ │ │ Eden │ S0 │ S1 │ │ │ 8/10 │ 1/10 │ 1/10 │ │ │ (From) │ (To) │ │ └───────┴────────┴─────────┴──────────────────────────────┘
对象分配流程
  1. 新对象首先在Eden区分配
  2. Eden区满时，触发Minor GC
  3. 存活对象复制到Survivor区（S0或S1），年龄+1
  4. Survivor区对象年龄达到阈值（默认15），晋升到老年代
  5. 大对象（超过阈值）直接进入老年代
内存区域默认比例说明
Eden区80%年轻代新对象分配区域
Survivor S010%年轻代Minor GC后存活对象存放
Survivor S110%年轻代备用Survivor区
老年代约占堆2/3存放长期存活对象
💡
空间分配担保

Minor GC前，JVM会检查老年代最大连续可用空间是否大于年轻代所有对象总空间。如果不足，会触发一次Full GC或等待担保失败后再执行Full GC。

垃圾回收算法

判断对象是否存活

引用计数法（Python使用）：存在循环引用问题
可达性分析（Java使用）：从GC Roots出发遍历引用链，不可达的对象标记为垃圾

GC Roots包括：

  • 虚拟机栈中的引用
  • 方法区静态属性引用
  • 方法区常量引用
  • 本地方法栈JNI引用
  • 同步锁持有的对象
三种基本算法
算法原理优点缺点
标记-清除
Mark-Sweep		 标记存活对象，清除未标记对象 简单，不移动对象 效率低，产生内存碎片
标记-复制
Mark-Copy		 将存活对象复制到另一半空间，清空当前空间 无内存碎片，速度快 浪费一半内存空间
标记-整理
Mark-Compact		 标记存活对象，将它们向一端移动，清理边界外空间 无内存碎片 需要移动对象，效率较低
💡
分代收集策略

新生代：对象朝生夕死，采用标记-复制算法
老年代：对象存活率高，采用标记-清除或标记-整理算法

G1收集器详解

G1（Garbage First）将堆划分为多个大小相等的Region（默认1~32MB），每个Region可以扮演Eden、Survivor或Old角色。

┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐ │ E │ E │ S │ S │ O │ O │ O │ O │ H │ O │ ... └───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘ E: Eden S: Survivor O: Old H: Humongous(大对象)

工作模式：

  • Young GC：Eden区满时触发，回收新生代Region
  • 并发标记：老年代占用率达到阈值时启动
  • Mixed GC：回收新生代和部分老年代Region
  • Full GC：兜底机制（应避免触发）
ZGC收集器

ZGC目标：停顿时间不超过10ms，且不随堆大小增加而增加。核心技术包括：

  • 着色指针（Colored Pointer）：在指针中存储元数据
  • 读屏障（Load Barrier）：运行时检测指针状态
  • 多重映射（Multi-Mapping）：优化内存访问

ZGC支持TB级堆内存，适合对延迟敏感的大型应用。

⚠️
CMS已废弃

CMS在JDK 14中被标记为废弃，JDK 17中正式移除。建议迁移至G1或ZGC收集器。

内存泄漏排查

常见内存泄漏场景
Java ❌ 错误示例 
// 场景1：静态集合持续增长
private static final Map<String, Object> CACHE = new HashMap<>();

// 场景2：ThreadLocal未清理
private static final ThreadLocal<byte[]> THREAD_LOCAL = new ThreadLocal<>();
public void process() {
    THREAD_LOCAL.set(new byte[1024 * 1024]);
    // 忘记调用 THREAD_LOCAL.remove()
}

// 场景3：未关闭的资源
Scanner scanner = new Scanner(new File("data.txt"));
// 忘记关闭，导致内存泄漏
Java ✅ 正确做法 
// 使用WeakHashMap或设置过期时间
private static final Map<String, Object> CACHE = new WeakHashMap<>();

// 使用try-with-resources自动关闭资源
try (Scanner scanner = new Scanner(new File("data.txt"))) {
    // 处理数据
}

// ThreadLocal使用后及时清理
THREAD_LOCAL.set(new byte[1024 * 1024]);
try {
    // 业务逻辑
} finally {
    THREAD_LOCAL.remove();
}

⚠️
引用类型

强引用：永不回收（除非不可达）
软引用（SoftReference）：内存不足时回收，适合缓存
弱引用（WeakReference）：下次GC时回收，适合WeakHashMap
虚引用（PhantomReference）：无法获取对象，用于跟踪GC

调优建议
  • 设置相同Xms和Xmx：避免运行时动态扩容/缩容带来的性能损耗
  • 合理设置新生代大小：过小导致频繁Minor GC，过大会导致老年代紧张
  • 避免大对象：大对象直接进入老年代，增加Full GC频率
  • 监控GC日志：及时发现GC异常（如频繁Full GC）

面试高频考点

Q1: JVM 内存结构有哪几部分？哪些是线程共享的？
区域线程存储内容OOM风险
堆（Heap）共享对象实例、数组✅ Java Heap OOM
元空间（Metaspace）共享类元数据、常量池✅ Metaspace OOM
虚拟机栈私有栈帧（局部变量、操作数栈）✅ StackOverflow
本地方法栈私有Native方法栈帧✅ StackOverflow
程序计数器私有当前字节码指令地址❌ 唯一不会OOM
Q2: Minor GC、Major GC、Full GC 有什么区别？
  • Minor GC：只回收年轻代（Eden+Survivor），频率高，速度快（毫秒级），STW短
  • Major GC：只回收老年代，通常由 Minor GC 触发（空间担保失败），比 Minor GC 慢10倍
  • Full GC：回收整个堆+元空间，STW最长，触发条件：调用System.gc()、老年代空间不足、元空间满、空间担保失败
⚠️
频繁 Full GC 是性能瓶颈的重要信号，排查方向：内存泄漏、对象过大直接进老年代、元空间不足。

Q4: 什么是 STW（Stop-The-World）？如何减少它的影响？

STW 是 GC 时暂停所有用户线程的机制，用于保证 GC 过程中引用关系不变化（一致性快照）。

减少 STW 影响的方法：

  • 选用低停顿收集器（G1、ZGC、Shenandoah）
  • 合理设置堆大小和分代比例，减少 Full GC 频率
  • 避免大对象（超过 Region 一半的对象直接进 Humongous 区）
  • ZGC 目标：STW < 1ms，且与堆大小无关
⚠️

实战陷阱

⚠️
陷阱1：-Xms 和 -Xmx 设置不一致 初始堆（Xms）远小于最大堆（Xmx）时，JVM 会在运行时动态扩堆，这会触发 Full GC 和内存碎片整理，影响响应时间。生产环境建议设置相等，如 -Xms4g -Xmx4g。
⚠️
陷阱2：元空间不设上限导致无限增长 JDK 8+ 元空间默认使用本地内存，没有上限。如果有类加载泄漏（如 CGLib 不断生成代理类），元空间会无限增长最终导致系统级 OOM。必须设置 -XX:MaxMetaspaceSize。
⚠️
陷阱3：finalize() 延迟导致 GC 困难 重写 finalize() 的对象至少需要两次 GC 才能被回收（第一次标记、执行 finalize()，第二次才真正回收）。finalize() 中若又创建强引用，对象可能"复活"。应改用 try-with-resources 或 Cleaner（JDK9+）。
