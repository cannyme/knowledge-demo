适配器模式

将一个类的接口转换成客户希望的另外一个接口，使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。

🎯

模式定义与意图

适配器模式（Adapter Pattern）是一种结构型设计模式，它允许接口不兼容的类相互合作。适配器将一个类的接口转换成客户希望的另外一个接口。

核心思想：通过包装（Wrapper）将不兼容的接口转换为兼容的接口，让原本不能一起工作的类协同工作。

生活中的类比：

  • 电源适配器：将220V交流电转换为设备需要的电压
  • USB转接头：让Type-C接口兼容传统USB设备
  • 翻译官：让说不同语言的人能够交流
🏗️

架构原理

classDiagram class Target { <> +request() } class Adapter { -adaptee: Adaptee +request() } class Adaptee { +specificRequest() } class Client Target <|.. Adapter Adapter --> Adaptee Client --> Target

角色说明：

角色说明
Target目标接口，客户端期望的接口
Adaptee被适配者，已存在的接口
Adapter适配器，将Adaptee接口转为Target接口
Client客户端，使用Target接口
📝

三种实现方式

对象适配器（组合方式）- 推荐

通过组合持有被适配者对象，更灵活，符合"组合优于继承"原则。

Java
// 目标接口
interface MediaPlayer {
    void play(String filename);
}

// 被适配者
class AdvancedMediaPlayer {
    public void playVlc(String filename) {
        System.out.println("播放VLC文件：" + filename);
    }
    public void playMp4(String filename) {
        System.out.println("播放MP4文件：" + filename);
    }
}

// 对象适配器
class MediaAdapter implements MediaPlayer {
    private final AdvancedMediaPlayer advancedPlayer;

    public MediaAdapter(AdvancedMediaPlayer advancedPlayer) {
        this.advancedPlayer = advancedPlayer;
    }

    @Override
    public void play(String filename) {
        if (filename.endsWith(".vlc")) {
            advancedPlayer.playVlc(filename);
        } else if (filename.endsWith(".mp4")) {
            advancedPlayer.playMp4(filename);
        }
    }
}

类适配器（继承方式）

通过继承被适配者类实现适配，耦合度较高，受Java单继承限制。

Java
// 类适配器：通过继承实现
class MediaAdapterByClass extends AdvancedMediaPlayer
                         implements MediaPlayer {
    @Override
    public void play(String filename) {
        if (filename.endsWith(".vlc")) {
            playVlc(filename);  // 调用父类方法
        } else if (filename.endsWith(".mp4")) {
            playMp4(filename);  // 调用父类方法
        }
    }
}
⚠️ 注意事项
类适配器由于使用继承，无法适配Adaptee的子类，且Java单继承限制了扩展性。

接口适配器（缺省适配器）

当接口方法很多，但只需要实现部分方法时使用。通过抽象类提供默认空实现。

Java
// 定义多个方法的接口
interface FileOperation {
    void read();
    void modify();
    void remove();
    void copy();
    void move();
}

// 抽象适配器：提供默认空实现
abstract class FileAdapter implements FileOperation {
    @Override public void read() {}
    @Override public void modify() {}
    @Override public void remove() {}
    @Override public void copy() {}
    @Override public void move() {}
}

// 具体实现：只需覆盖需要的方法
class ReadOnlyFile extends FileAdapter {
    @Override
    public void read() {
        System.out.println("只读文件");
    }
    // 其他方法使用默认空实现
}
方式实现优点缺点
对象适配器组合灵活、可适配子类需要额外引用
类适配器继承简单直接耦合高、单继承限制
接口适配器抽象类减少空实现仅适用于接口方法多的场景
💡

适用场景

  • 遗留系统集成：需要使用已有的类，但其接口与需求不匹配
  • 第三方SDK集成：第三方库接口与项目规范不一致
  • 接口升级：旧版本API需要兼容新版本调用方式
  • 日志框架统一：SLF4J通过适配器支持Log4j、Logback等
  • 数据转换：不同数据格式之间的转换
🔧

JDK中的适配器

1. InputStreamReader

将字节流（InputStream）适配为字符流（Reader）：

Java
// InputStreamReader是字节流到字符流的适配器
InputStream inputStream = new FileInputStream("file.txt");
Reader reader = new InputStreamReader(inputStream, "UTF-8");

// BufferedReader包装Reader提供缓冲功能
BufferedReader br = new BufferedReader(reader);

2. Arrays.asList()

将数组适配为List接口：

Java
// 数组适配为List
String[] array = {"a", "b", "c"};
List<String> list = Arrays.asList(array);  // 返回适配器视图

3. Spring MVC HandlerAdapter

Spring使用适配器模式统一处理不同类型的Controller：

Java
// HandlerAdapter接口
public interface HandlerAdapter {
    boolean supports(Object handler);
    ModelAndView handle(HttpServletRequest request,
                        HttpServletResponse response,
                        Object handler);
}

// 不同类型的Controller使用不同适配器
// - RequestMappingHandlerAdapter
// - HttpRequestHandlerAdapter
// - SimpleControllerHandlerAdapter

优缺点分析

✅ 优点
  • 解耦：目标类和被适配者类解耦，通过适配器间接调用
  • 复用性：复用已有的类，无需修改原有代码
  • 扩展性：可以适配多个被适配者
  • 单一职责：将接口转换逻辑隔离到适配器中
❌ 缺点
  • 增加系统复杂度，需要额外编写适配器类
  • 过多使用适配器会导致代码结构混乱
  • 类适配器受单继承限制
🔄

与其他模式对比

模式目的区别
适配器模式接口转换改变已有接口以兼容目标接口
装饰器模式功能增强不改变接口，动态添加职责
代理模式访问控制不改变接口，控制对对象的访问
外观模式简化接口为子系统提供统一的高层接口
