适配器模式

将一个类的接口转换成客户端期望的另一个接口，使接口不兼容的类可以协同工作。Spring MVC HandlerAdapter、SLF4J、InputStreamReader 均是典型应用。

📖

三种适配器形式

graph LR C[客户端] -- "Target接口" --> A[适配器 Adapter] A -- "调用" --> B[被适配者 Adaptee] style A fill:#f0b429,color:#000 style B fill:#4ecdc4,color:#000
形式实现方式优缺点
对象适配器组合：持有 Adaptee 引用✅ 推荐，灵活，不受单继承限制
类适配器继承 Adaptee + 实现 Target❌ Java 单继承限制，耦合高
接口适配器抽象类提供默认空实现✅ 只需实现关心的方法

代码实现

对象适配器（推荐）

Java
// 目标接口（客户端期望的接口）
interface MediaPlayer {
    void play(String filename);
}

// 被适配者（已存在，接口不兼容）
class AdvancedMediaPlayer {
    public void playVlc(String filename) { System.out.println("播放VLC：" + filename); }
    public void playMp4(String filename) { System.out.println("播放MP4：" + filename); }
    public void playAvi(String filename) { System.out.println("播放AVI：" + filename); }
}

// 对象适配器：持有被适配者，转换接口
class MediaAdapter implements MediaPlayer {
    private final AdvancedMediaPlayer advancedPlayer; // 组合（不是继承）

    public MediaAdapter(AdvancedMediaPlayer player) {
        this.advancedPlayer = player;
    }

    @Override
    public void play(String filename) {
        // 将统一的 play 接口转换为具体方法调用
        if (filename.endsWith(".vlc")) {
            advancedPlayer.playVlc(filename);
        } else if (filename.endsWith(".mp4")) {
            advancedPlayer.playMp4(filename);
        } else if (filename.endsWith(".avi")) {
            advancedPlayer.playAvi(filename);
        } else {
            System.out.println("不支持的格式：" + filename);
        }
    }
}

// 客户端只需要 MediaPlayer 接口
MediaPlayer player = new MediaAdapter(new AdvancedMediaPlayer());
player.play("movie.mp4");  // → 播放MP4：movie.mp4
player.play("music.vlc");  // → 播放VLC：music.vlc

接口适配器（缺省适配器）

Java
// 接口方法很多，实现类只需要部分方法时使用
interface FileOperation {
    void read();
    void write(String content);
    void delete();
    void copy(String dest);
    void move(String dest);
}

// 接口适配器：抽象类提供空实现
abstract class FileAdapter implements FileOperation {
    @Override public void read() {}
    @Override public void write(String c) {}
    @Override public void delete() {}
    @Override public void copy(String d) {}
    @Override public void move(String d) {}
}

// 具体实现：只覆盖关心的方法（类似 MouseAdapter）
class ReadOnlyFile extends FileAdapter {
    @Override
    public void read() {
        System.out.println("只读文件：读取成功");
    }
    // 其他方法保持空实现，不需要逐一处理
}

Spring MVC HandlerAdapter 详解

Spring MVC 中有多种 Handler（Controller）类型，HandlerAdapter 统一了它们的调用方式：

sequenceDiagram participant DS as DispatcherServlet participant HM as HandlerMapping participant HA as HandlerAdapter participant H as Controller DS->>HM: getHandler(request) HM-->>DS: HandlerExecutionChain DS->>HA: supports(handler)? HA-->>DS: true DS->>HA: handle(request, response, handler) HA->>H: invokeHandlerMethod() H-->>HA: ModelAndView HA-->>DS: ModelAndView
Java - HandlerAdapter 接口
// Spring MVC HandlerAdapter 接口（简化）
public interface HandlerAdapter {
    // 判断此适配器是否支持该 Handler
    boolean supports(Object handler);
    // 用统一接口调用不同类型的 Handler
    ModelAndView handle(HttpServletRequest request,
                        HttpServletResponse response,
                        Object handler) throws Exception;
}

// 针对 @RequestMapping 方法的适配器
// RequestMappingHandlerAdapter implements HandlerAdapter

// 针对实现了 Controller 接口的适配器
// SimpleControllerHandlerAdapter implements HandlerAdapter

// 为什么需要 HandlerAdapter？
// 因为不同 Handler 调用方式完全不同：
// - @RequestMapping: 通过反射调用方法
// - Controller 接口: 直接 controller.handleRequest()
// - HttpRequestHandler: handler.handleRequest()
// HandlerAdapter 屏蔽了这些差异，DispatcherServlet 只调用 handle()

// 实际应用：DispatcherServlet 核心逻辑
HandlerExecutionChain chain = getHandler(request);
Object handler = chain.getHandler();
for (HandlerAdapter ha : handlerAdapters) {
    if (ha.supports(handler)) {
        mv = ha.handle(request, response, handler); // 统一接口
        break;
    }
}

面试高频考点

Q1：类适配器 vs 对象适配器的区别？
  • 类适配器（继承）：通过继承 Adaptee + 实现 Target。缺点：Java 单继承限制，且与 Adaptee 耦合强，Adaptee 的变化会影响 Adapter
  • 对象适配器（组合）：持有 Adaptee 引用，更灵活，可以适配 Adaptee 及其子类。推荐使用
  • 原则："组合优于继承"，对象适配器体现了这一原则
Q2：适配器 vs 装饰器 vs 代理的区别？（包装三兄弟）
三者都是"包装"另一个对象，但意图不同：
模式意图接口关系典型场景
适配器接口转换（让不兼容的接口一起工作）转换为不同接口第三方SDK接入、IO流适配
装饰器功能增强（添加新职责）实现相同接口Java IO、咖啡配料
代理控制访问（不暴露真实对象）实现相同接口AOP、权限控制、延迟加载
Q3：Java IO 中的 InputStreamReader 是适配器吗？
是的。InputStream（字节流）和 Reader（字符流）接口不兼容，InputStreamReader 作为适配器，将字节流适配成字符流，同时处理字符编码转换。这是典型的对象适配器：
Java
// InputStreamReader = 适配器
// InputStream      = 被适配者（Adaptee）
// Reader           = 目标接口（Target）
Reader reader = new InputStreamReader(
    new FileInputStream("file.txt"), "UTF-8"
);
// 也是装饰器：BufferedReader 在 Reader 基础上增加缓冲功能
BufferedReader br = new BufferedReader(reader);

实战陷阱

陷阱1：过度使用适配器导致系统复杂
适配器是"补丁"方案，不应替代良好的接口设计。如果自己设计的系统接口不兼容，优先考虑重构接口而非套适配器。适配器最适合整合第三方库或遗留代码。
陷阱2：适配器链过长
A→适配器→B→适配器→C 的调用链会导致性能损耗和调试困难。如果需要多层适配，考虑直接重构。
最佳实践：适配器的命名规范
Spring 中以 Adapter 后缀命名（HandlerAdapter、WebMvcConfigureAdapter），推荐遵循此规范。接口适配器通常以 Abstract 前缀 + Adapter 后缀（AbstractUserDetailsAuthenticationProvider）。
⚖️

优缺点分析

✅ 优点

  • 单一职责：将接口转换逻辑与业务逻辑分离
  • 开闭原则：新增适配器无需修改客户端和被适配者
  • 复用遗留代码：使现有类在新接口下可复用
  • 解耦合：客户端与具体实现类解耦

❌ 缺点

  • 增加复杂度：额外的适配器类使系统更复杂
  • 性能开销：多一层间接调用（通常可忽略）
  • 过多适配器：系统中适配器太多说明设计存在问题

与其他模式的关系

相关模式关系说明
装饰器模式两者都包装对象，但意图不同：适配器转换接口，装饰器增强功能且保持接口不变
代理模式代理保持接口一致，对客户端透明；适配器向客户端暴露不同接口
外观模式外观为子系统提供简化接口；适配器将已有接口转换为另一种接口
桥接模式桥接在设计时建立抽象与实现的分离；适配器在设计后期处理不兼容问题
