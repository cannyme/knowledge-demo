分库分表策略

分库分表是应对海量数据和高并发访问的重要数据库架构方案，通过将数据分散到多个数据库实例和表中，提升系统性能和可扩展性。

❓

为什么需要分库分表

随着业务规模增长，单库单表会面临以下性能瓶颈：

  • 数据量瓶颈：单表数据超过 1000 万行时，查询性能急剧下降，索引维护成本增加
  • 并发量瓶颈：单库连接数有限（MySQL 默认 151），高并发场景下连接池耗尽
  • 存储瓶颈：单机磁盘容量和 IO 能力有限，无法满足海量数据存储需求
  • 单点故障：单库故障会导致整个服务不可用

分库分表方式

垂直拆分（按业务拆分）

垂直拆分为垂直分库和垂直分表两种：

  • 垂直分库：按业务模块将不同表拆分到不同数据库，如用户库、订单库、商品库分离
  • 垂直分表：按字段将单表拆分为多个表，将访问频率不同的字段分离（如基础信息表和详情表）

水平拆分（按数据拆分）

将同一表的数据分散到多个库或表中，每个分片结构相同但数据不同：

  • 水平分库：同一表数据分散到不同数据库实例
  • 水平分表：同一表数据分散到同一库的不同表中

分片策略

范围分片（Range）

按数据范围划分分片，如按时间范围或ID范围：

  • 优点：扩容简单，只需增加新区间；支持范围查询
  • 缺点：可能存在热点问题（如最新数据集中在最后一个分片）
JAVA
// 范围分片示例
public int getShardIndex(long orderId) {
    if (orderId < 1000000) return 0;
    if (orderId < 2000000) return 1;
    if (orderId < 3000000) return 2;
    return 3;
}

哈希分片（Hash）

通过对分片键取模计算分片位置：

  • 优点：数据分布均匀，避免热点
  • 缺点：扩容困难，需要数据迁移
JAVA
// Hash取模分片
public int getShardIndex(long userId, int shardCount) {
    return (int) (userId % shardCount);
}
// userId=100 → 分片0, userId=101 → 分片1

一致性哈希

通过哈希环实现，扩容时只影响相邻节点：

  • 优点：扩容时数据迁移量最小（仅影响相邻节点）
  • 缺点：实现复杂，需要虚拟节点解决数据倾斜

标签分片（Gene/基因法）

将分片基因嵌入ID，保证关联数据在同一分片：

  • 场景：订单和订单明细需要路由到同一分片
  • 实现：将user_id的分片基因嵌入order_id
JAVA
// 基因法：将分片基因嵌入订单ID
public long generateOrderId(long sequence, long userId) {
    long gene = userId % 4;  // 获取分片基因
    return (sequence << 2) | gene;  // 嵌入基因到订单ID
}

public long extractGene(long orderId) {
    return orderId & 0x3;  // 从订单ID提取基因
}

ShardingSphere架构

Apache ShardingSphere 是流行的分库分表中间件，提供两种部署模式：

ShardingJDBC（客户端模式）

  • 以 Jar 包形式嵌入应用程序
  • 性能高，无网络开销
  • 适合 Java 应用，配置简单
graph LR A[应用] --> B[ShardingJDBC] B --> C[数据库0] B --> D[数据库1] B --> E[数据库2]

ShardingProxy（服务端模式）

  • 独立部署的代理服务，支持任意语言
  • 支持连接池管理，集中管控
  • 适合异构语言环境，需要额外运维

分库分表带来的问题

分布式事务

跨分片操作需要保证数据一致性，常用解决方案：

  • XA 事务：强一致性，两阶段提交，性能较差
  • TCC：Try-Confirm-Cancel 柔性事务，业务侵入性高
  • Seata AT 模式：自动补偿，对业务代码侵入小，推荐方案
  • 本地消息表：最终一致性，适合异步场景

跨分片查询

分片键之外的查询需要扫描所有分片：

  • 解决方案：使用全局表（小表广播）、数据冗余、或引入 Elasticsearch
  • 绑定表：关联表使用相同分片键，避免跨分片 JOIN

全局ID生成

分片后需要全局唯一的分布式 ID：

  • UUID：无序，索引性能差
  • 雪花算法（Snowflake）：趋势递增，性能好，推荐使用
  • 数据库号段模式：批量获取，减少数据库访问
  • Redis 自增：利用 INCR 命令

排序分页

跨分片排序分页需要归并各分片结果：

  • 各分片分别排序后，内存归并排序
  • 深分页问题严重，建议限制最大分页或采用游标分页
  • 考虑使用 Elasticsearch 进行复杂查询

最佳实践

分库分表时机判断
单表数据 > 1000万 或 单库连接数不足 或 单机存储不足

优先考虑替代方案：

  • 冷热数据分离：将历史数据归档
  • 读写分离：分担读压力
  • 数据归档：定期迁移历史数据
  • 使用 NoSQL：如 Elasticsearch、MongoDB

分片设计原则：

  • 分片键选择高频查询字段，确保数据分布均匀
  • 避免跨分片查询，尽量将关联数据路由到同一分片
  • 预留扩容空间，考虑一致性哈希减少迁移
  • 监控数据倾斜，及时调整分片策略

数据迁移方案：

  1. 双写阶段：新旧库同时写入，保证数据一致性
  2. 历史数据迁移：使用工具批量迁移
  3. 数据校验：比对校验数据完整性
  4. 灰度切换：逐步切换读流量到新库
  5. 停双写：确认无误后停止旧库写入
