MySQL索引优化

MySQL索引优化的详细解析和最佳实践

🏗️

索引原理

索引是帮助MySQL高效获取数据的数据结构。InnoDB使用B+树作为索引结构，通过索引可以快速定位到数据行，避免全表扫描。

flowchart TD A[SQL查询] --> B[查询优化器] B --> C{使用索引?} C -->|是| D[索引查找] C -->|否| E[全表扫描] D --> F[返回结果] E --> F
💡 索引的本质
索引本质上是一种空间换时间的策略。通过额外的存储空间维护有序的数据结构，将查询的时间复杂度从O(n)降低到O(log n)。
🌳

B+树结构详解

B+树是MySQL InnoDB存储引擎的默认索引结构，具有以下特点：

  • 非叶子节点不存储数据，只存储键值和子节点指针
  • 叶子节点存储所有数据，并通过双向指针连接，便于范围查询
  • 树高度低（通常3-4层），每次查询只需3-4次磁盘IO
  • 数据有序存储，支持高效的范围查找和排序

计算示例

假设每页16KB，主键BIGINT(8字节)，指针6字节：

  • 每个非叶子节点可存：16384/(8+6) ≈ 1170个键
  • 三层B+树可存：1170³ ≈ 16亿条记录
  • 查询只需3次磁盘IO

CREATE TABLE `user` (

InnoDB索引实现

聚簇索引（主键索引）

  • 叶子节点存储完整行数据
  • 一张表只能有一个聚簇索引
  • 数据按主键顺序存储
  • 主键查询效率极高

非聚簇索引（二级索引）

  • 叶子节点存储主键值
  • 一张表可以有多个二级索引
  • 查询需要"回表"操作
  • 通过主键值去聚簇索引查完整数据

💡 回表示例

SELECT * FROM user WHERE name = '张三';

  1. 在idx_name索引中找到 name='张三' 对应的主键id
  2. 根据id去主键索引中查找完整行数据（回表）
✅ 覆盖索引

当查询的列都在索引中时，无需回表，直接从索引返回数据：

SELECT id, name FROM user WHERE name = '张三'; -- 覆盖索引

SELECT * FROM user WHERE name = '张三'; -- 需要回表

⚡

索引优化原则

1. 最左前缀原则

联合索引 idx_name_age(name, age) 的使用情况：

查询条件是否使用索引说明
WHERE name = '张三'✅ 使用使用name字段
WHERE name = '张三' AND age = 25✅ 使用使用name和age字段
WHERE name = '张三' AND age > 20✅ 使用name定位，age范围扫描
WHERE age = 25❌ 不使用不满足最左前缀
WHERE name LIKE '%张'❌ 不使用前缀模糊匹配失效
WHERE name = '张三' OR age = 25❌ 不使用OR破坏索引使用
WHERE name LIKE '张%' AND age = 25✅ 使用name用索引，age可能用到
WHERE name = '张三' ORDER BY age✅ 使用name用索引，排序也用索引

2. 联合索引设计原则

  • 将选择性高（区分度高）的列放前面
  • 考虑查询场景，尽量覆盖更多查询
  • 避免索引列参与计算或使用函数
  • 范围查询字段放后面（范围后的列不走索引）

3. 索引下推（ICP）

MySQL 5.6+ 引入的优化，将WHERE条件的过滤下推到存储引擎层：

sql
-- 索引下推示例
KEY idx_name_age(name, age)
SELECT * FROM user WHERE name LIKE '张%' AND age = 25;

-- 没有ICP：先找到所有name以'张'开头的记录，再回表过滤age
-- 有ICP：在存储引擎层就过滤掉不满足age=25的记录，减少回表

4. 前缀索引

对长字符串列，可以使用前缀索引节省空间：

sql
-- 计算前缀选择性
SELECT COUNT(DISTINCT LEFT(email, 6)) / COUNT(DISTINCT email) FROM user;

-- 创建前缀索引
ALTER TABLE user ADD INDEX idx_email(email(10));

索引失效场景

🚫 1. 对索引列使用函数或运算

❌ WHERE YEAR(create_time) = 2024

✅ WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'

🚫 2. 隐式类型转换

❌ WHERE phone = 13800138000 -- phone是VARCHAR类型

✅ WHERE phone = '13800138000'

🚫 3. LIKE 以通配符开头

❌ WHERE name LIKE '%张%'

✅ WHERE name LIKE '张%'

🚫 4. OR 条件（除非所有列都有索引）

❌ WHERE name = '张三' OR age = 25

✅ 使用UNION替代，或为age也创建索引

🚫 5. NOT IN / NOT EXISTS / <>

❌ WHERE age NOT IN (1, 2, 3)

✅ 考虑使用范围查询或反向思维改写

🚫 6. 联合索引中间列使用范围查询

KEY idx_a_b_c(a, b, c)

❌ WHERE a = 1 AND b > 2 AND c = 3 -- c无法使用索引

✅ WHERE a = 1 AND b = 2 AND c = 3

🚫 7. 索引列允许NULL

NULL值会影响索引统计，建议设置NOT NULL并设置默认值

🚫 8. 索引选择性过低

如性别字段（男/女），区分度太低，建索引效果不佳

SQL优化案例

案例1：大偏移分页优化

❌ 问题SQL

SELECT * FROM user LIMIT 1000000, 10;

需要扫描前1000010行，丢弃前1000000行，效率极低

✅ 优化方案1：子查询+JOIN
SELECT * FROM user u
INNER JOIN (SELECT id FROM user LIMIT 1000000, 10) t
ON u.id = t.id;
-- 先通过覆盖索引快速获取id，再JOIN查询完整数据
✅ 优化方案2：记录上次查询位置
SELECT * FROM user
WHERE id > #{lastId}
ORDER BY id
LIMIT 10;
-- 利用主键索引，避免大偏移量

案例2：SELECT * 导致回表

❌ 问题SQL

SELECT * FROM user WHERE name = '张三';

需要回表查询所有字段

✅ 优化方案
-- 只查询需要的字段
SELECT id, name, age FROM user WHERE name = '张三';

-- 创建覆盖索引
ALTER TABLE user ADD INDEX idx_name_age(name, age);
-- 这样查询可以直接从索引返回，无需回表

案例3：ORDER BY 导致 filesort

❌ 问题SQL

SELECT * FROM user WHERE department_id = 1 ORDER BY create_time DESC;

department_id索引无法用于排序，需要filesort

✅ 优化方案
-- 创建联合索引，满足最左前缀
ALTER TABLE user ADD INDEX idx_dept_time(department_id, create_time);

-- 这样WHERE和ORDER BY都能使用索引，避免filesort

面试高频考点

Q1: B+树索引的结构和查询原理？为什么用 B+树而不是 B 树？
B+树：非叶子节点只存键值（不存数据），叶子节点存完整数据且通过链表相连。优势：①非叶子节点更小，一页能存更多键，树更矮（通常 3-4 层），IO 次数更少；②叶子节点链表支持高效范围查询（顺序IO）；③查询稳定，所有查询都到叶子节点（固定 IO 次数）。B 树非叶子节点存数据，范围查询需回溯，不适合顺序扫描。
Q2: 什么是覆盖索引？什么是回表？如何避免回表？
回表：二级索引查到主键后，再根据主键去聚簇索引查完整行数据，产生额外 IO。覆盖索引：查询字段全部包含在索引中，无需回表，EXPLAIN Extra 显示 "Using index"。避免回表：①将查询列加入联合索引；②SELECT 只查需要的列，避免 SELECT *；③使用主键查询（聚簇索引）。覆盖索引是性能优化的重要手段，大数据量分页查询中效果显著。
Q3: 索引失效的常见场景有哪些？
①最左前缀原则违反：联合索引 (a,b,c) 跳过 a 直接查 b/c；②在索引列上做函数运算或类型转换：WHERE DATE(create_time)='2024-01-01' 或 WHERE id='1'（字符串与数字比较）；③LIKE 以通配符开头：LIKE '%abc'；④使用 OR 连接且 OR 两边非同一索引；⑤范围查询右边的列：联合索引 (a,b,c) 中 a=1 AND b>5 AND c=3，c 索引失效；⑥NOT IN / NOT EXISTS；⑦数据量小时优化器认为全表扫更快。
Q4: 聚簇索引和非聚簇索引的区别？
聚簇索引（Clustered Index）：数据行与索引在同一 B+树中，叶子节点即数据页，InnoDB 主键索引即聚簇索引，每表只有一个。非聚簇索引（Secondary Index）：叶子节点存的是主键值而非完整数据，查询时需回表。MyISAM 的所有索引都是非聚簇索引（数据文件和索引文件分离）。选择主键建议：自增整数（顺序插入，减少页分裂）；避免 UUID（随机写入，B+树频繁页分裂，性能差）。

实战陷阱

⚠️
陷阱1：大数据量深分页性能急剧下降 LIMIT 100000, 10 需要扫描前 100010 行再丢弃前 100000 行，全表IO开销极大。优化方案：①游标分页（WHERE id > last_id LIMIT 10，适合单调递增主键）；②覆盖索引子查询（SELECT * FROM t JOIN (SELECT id FROM t ORDER BY id LIMIT 100000,10) tmp USING(id)）；③业务层限制最大页数。
⚠️
陷阱2：隐式类型转换导致索引失效 字段类型为 VARCHAR 但查询条件用数字（WHERE phone = 13800000000），MySQL 会对字符串列做隐式转换（CAST），导致索引失效，全表扫描。反之（字段为 INT，条件为字符串）索引仍然生效。务必确保查询参数类型与字段类型完全一致。
⚠️
陷阱3：频繁更新的列不适合建索引 每次更新索引列的值，B+树需要删除旧键并插入新键，可能引发页分裂/合并，写性能下降。状态字段（如 order_status，经常从 PENDING 改为 PAID）若区分度低（只有几个值），索引效果差且维护成本高。应评估字段的查询频率、更新频率和区分度，综合判断是否建索引。
