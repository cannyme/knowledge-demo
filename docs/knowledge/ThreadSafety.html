线程安全问题（三大特性）

特性说明解决方案
原子性操作不可分割，要么全部成功，要么全部失败synchronized, Lock, Atomic类
可见性一个线程的修改对其他线程立即可见volatile, synchronized, Lock, final
有序性程序执行顺序与代码顺序一致volatile, synchronized, Lock
⚠️ 问题示例：count++ 不是原子操作
实际执行：1. 读取count值 → 2. 加1 → 3. 写回。多线程下可能读取相同值，导致计数丢失。
Java
// 线程不安全的计数器
class UnsafeCounter {
    private int count = 0;
    public void increment() {
        count++; // 非原子操作！
    }
}

// 两个线程同时执行：
// T1: 读取count=0 → T2: 读取count=0 → T1: 写入count=1 → T2: 写入count=1
// 结果：count=1（期望是2），丢失了1次计数！

volatile关键字

volatile保证可见性和禁止指令重排序，但不保证原子性。

sequenceDiagram participant T1 as Thread 1 participant M as 主内存 participant T2 as Thread 2 T1->>T1: 修改volatile变量 T1->>M: 立即刷新到主内存 T2->>M: 读取最新值 M-->>T2: 保证可见性

适用场景

  • 状态标志位：一个线程修改，其他线程立即可见
  • 双重检查锁：单例模式中的懒加载
  • 独立观察：一个线程写，多个线程读
Java
class VolatileExample {
    // 状态标志：一个线程修改，其他线程立即可见
    private volatile boolean running = true;

    public void stop() {
        running = false; // 对其他线程立即可见
    }

    public void doWork() {
        while (running) {
            // 执行任务
        }
    }
}

// 双重检查锁（DCL）单例
class Singleton {
    private volatile static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton(); // volatile防止指令重排序
                }
            }
        }
        return instance;
    }
}
❌ 错误用法：volatile不能保证原子性，不能用于 count++ 等复合操作

synchronized关键字

synchronized是Java内置的同步机制，可以保证原子性、可见性和有序性。

三种使用方式

Java
class SynchronizedCounter {
    private int count = 0;
    private final Object lock = new Object();

    // 1. 同步方法（锁当前对象this）
    public synchronized void increment() {
        count++;
    }

    // 2. 同步代码块（更灵活，减小锁粒度）
    public void incrementWithBlock() {
        synchronized (this) {
            count++;
        }
    }

    // 3. 使用不同对象作为锁
    public void incrementWithCustomLock() {
        synchronized (lock) {
            count++;
        }
    }

    // 静态同步方法（锁类对象）
    public static synchronized void staticMethod() {
        // 锁的是SynchronizedCounter.class
    }
}

锁升级（JDK 6+优化）

flowchart LR A[无锁] -->|第一个线程访问| B[偏向锁] B -->|第二个线程竞争| C[轻量级锁] C -->|竞争激烈| D[重量级锁]
锁类型说明
偏向锁单线程访问时，在对象头标记线程ID，无需CAS
轻量级锁多线程交替访问，使用CAS自旋，避免线程阻塞
重量级锁竞争激烈，线程阻塞，使用操作系统互斥量
💡 实现原理：基于对象的Monitor（监视器锁），通过monitorenter和monitorexit指令实现

Lock接口

java.util.concurrent.locks.Lock提供更灵活的锁机制，相比synchronized有更多高级特性。

特性synchronizedReentrantLock
获取/释放JVM自动管理手动lock()/unlock()
可中断否是（lockInterruptibly）
超时获取否是（tryLock(timeout)）
公平锁非公平可选公平/非公平
条件变量单个（wait/notify）多个（Condition）
性能JDK6后差异不大高竞争时略优
Java
class ReentrantLockCounter {
    // 公平锁：new ReentrantLock(true)
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock(); // 获取锁
        try {
            count++; // 临界区代码
        } finally {
            lock.unlock(); // 必须释放锁！
        }
    }

    // 尝试获取锁（非阻塞）
    public boolean tryIncrement() {
        if (lock.tryLock()) {
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;
    }

    // 超时获取锁
    public boolean tryIncrementWithTimeout() throws InterruptedException {
        if (lock.tryLock(1, TimeUnit.SECONDS)) {
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;
    }
}

ReadWriteLock（读写锁）

Java
class ReadWriteLockExample {
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();
    private int data = 0;

    // 读操作（多个线程可同时读）
    public int read() {
        readLock.lock();
        try {
            return data;
        } finally {
            readLock.unlock();
        }
    }

    // 写操作（独占）
    public void write(int value) {
        writeLock.lock();
        try {
            data = value;
        } finally {
            writeLock.unlock();
        }
    }
}
读写锁规则：读-读不互斥、读-写互斥、写-写互斥。适合读多写少的场景。

原子类与CAS

原子类使用CAS（Compare-And-Swap）实现无锁并发，性能好且不会阻塞线程。

CAS原理

sequenceDiagram participant T as 线程 participant M as 内存值V participant E as 预期值A participant N as 新值B T->>M: 读取当前值V T->>E: 记录预期值A=V T->>T: 计算新值B T->>M: CAS(V, A, B) alt V == A M->>M: 更新为B M-->>T: 成功 else V != A M-->>T: 失败，重试 end
Java
class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // 原子自增
    }

    public void incrementWithRetry() {
        // 手动CAS操作（带重试）
        int oldValue, newValue;
        do {
            oldValue = count.get();
            newValue = oldValue + 1;
        } while (!count.compareAndSet(oldValue, newValue));
    }

    // 常用方法
    public void methods() {
        count.get();                    // 获取值
        count.set(100);                 // 设置值
        count.getAndIncrement();        // 先获取后自增
        count.incrementAndGet();        // 先自增后获取
        count.addAndGet(5);             // 增加指定值
        count.compareAndSet(0, 1);      // CAS操作
    }
}

原子类族

类型类说明
基本类型AtomicInteger, AtomicLong, AtomicBoolean原子基本类型
引用类型AtomicReference原子引用
数组类型AtomicIntegerArray原子数组
字段更新器AtomicIntegerFieldUpdater原子更新字段
高并发累加器LongAdder, LongAccumulator分段累加，减少竞争
💡 LongAdder：高并发场景下比AtomicLong性能更好，原理是分段累加多个Cell，最后汇总
⚠️ CAS缺点：
1. ABA问题（可用AtomicStampedReference解决）
2. 长时间自旋消耗CPU
3. 只能保证单个变量的原子性

ThreadLocal（线程本地变量）

ThreadLocal为每个线程提供独立的变量副本，线程间互不干扰。

flowchart TD subgraph Thread1["Thread 1"] T1V["ThreadLocal变量副本: user=Alice"] end subgraph Thread2["Thread 2"] T2V["ThreadLocal变量副本: user=Bob"] end subgraph Thread3["Thread 3"] T3V["ThreadLocal变量副本: user=null"] end TL[ThreadLocal对象] -.-> T1V TL -.-> T2V TL -.-> T3V

应用场景

  • 数据库连接：每个线程独立的Connection，避免多线程安全问题
  • 用户会话：存储当前登录用户信息
  • SimpleDateFormat：非线程安全，每个线程一个实例
  • 请求上下文：Spring中存储Request、Response信息
Java
class ThreadLocalExample {
    // 每个线程独立的变量
    private static final ThreadLocal<String> userContext = new ThreadLocal<>();

    // 使用initialValue设置初始值
    private static final ThreadLocal<Integer> requestId =
        ThreadLocal.withInitial(() -> 0);

    public void setUser(String user) {
        userContext.set(user);
    }

    public String getUser() {
        return userContext.get();
    }

    public void process() {
        try {
            setUser("当前用户");
            // 执行业务逻辑
            System.out.println(getUser());
        } finally {
            // ⚠️ 重要：使用完毕必须清理，防止内存泄漏！
            userContext.remove();
        }
    }
}

内存泄漏问题

⚠️ 原因：ThreadLocalMap的key是弱引用（ThreadLocal），value是强引用（存储的值）。
线程池场景下，线程不销毁，value无法回收。
解决：使用后必须调用 remove() 方法清理
Java
// InheritableThreadLocal：子线程可继承父线程的值
private static final InheritableThreadLocal<String> inheritable =
    new InheritableThreadLocal<>();

// 父线程设置值
inheritable.set("父线程的值");

// 子线程可以获取到父线程的值
new Thread(() -> {
    System.out.println(inheritable.get()); // 输出：父线程的值
}).start();

并发容器

ConcurrentHashMap

线程安全的HashMap，采用分段锁（Java 7）或CAS+synchronized（Java 8）实现高并发。

Java
// 线程安全的HashMap
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// 基本操作
map.put("key", 1);
Integer value = map.get("key");

// 原子性复合操作
map.putIfAbsent("key", 2);     // key不存在才put
map.replace("key", 1, 2);      // 旧值等于1时才替换为2

// compute方法（原子性计算）
map.compute("key", (k, v) -> v == null ? 1 : v + 1);

// merge方法（合并）
map.merge("key", 1, Integer::sum); // 存在则相加，不存在则put 1

CopyOnWriteArrayList

写时复制，读操作无锁，写操作加锁并复制新数组。适合读多写少的场景。

Java
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();

// 读操作：无锁
for (String s : list) {
    System.out.println(s);
}

// 写操作：加锁，复制新数组
list.add("new item");
list.remove("old item");
💡 CopyOnWrite原理：写操作时复制一份新数组，修改后替换引用。读操作始终在原数组上进行，无需加锁。
容器特点适用场景
ConcurrentHashMap分段锁/CAS，高并发读写高频读写的Map
CopyOnWriteArrayList写时复制，读无锁读多写少
CopyOnWriteArraySet基于CopyOnWriteArrayList读多写少的Set
ConcurrentLinkedQueueCAS无锁队列高并发队列
BlockingQueue阻塞队列，支持生产者消费者线程协作
