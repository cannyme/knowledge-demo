API网关

API网关是微服务架构的单一入口，负责请求路由、负载均衡、认证鉴权、限流熔断等核心功能，统一处理所有客户端请求并转发到后端服务。

❓

为什么需要API网关

在微服务架构中，服务数量众多且分散，客户端直接与各个服务通信会面临诸多问题：

  • 统一入口：客户端只需知道网关地址，无需了解众多后端服务的具体位置
  • 路由转发：根据请求路径、Header等信息将请求转发到对应的服务
  • 认证鉴权：统一处理身份验证和权限校验，避免每个服务重复实现
  • 限流熔断：防止流量突增压垮后端服务，保障系统稳定性
  • 协议转换：支持HTTP转RPC、WebSocket等协议适配
  • 日志监控：统一记录请求日志、性能指标，便于问题排查

Spring Cloud Gateway

Spring Cloud Gateway是Spring官方推出的API网关组件，基于Spring 5、Spring Boot 2和Project Reactor构建，使用Netty作为运行时，支持高并发非阻塞IO。

路由配置

路由由三部分组成：ID、URI和Predicate（断言）：

YAML
spring:
  cloud:
    gateway:
      routes:
        # 用户服务路由
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/user/**
            - Method=GET,POST
          filters:
            - StripPrefix=1

        # 订单服务路由
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/order/**
            - Header=X-Version, v1
            - Query=token
          filters:
            - AddRequestHeader=X-Source, Gateway

Predicate断言工厂

断言说明示例
Path路径匹配- Path=/api/user/**
Method请求方法匹配- Method=GET,POST
Header请求头匹配- Header=X-Token, \d+
Query查询参数匹配- Query=token
CookieCookie匹配- Cookie=sessionId, .+
Host主机名匹配- Host=**.example.com
After/Before/Between时间匹配- After=2024-01-01T00:00:00+08:00
Weight权重路由- Weight=group1, 80

过滤器（GatewayFilter）

过滤器说明示例
AddRequestHeader添加请求头- AddRequestHeader=X-Request-Id, 123
AddRequestParameter添加请求参数- AddRequestParameter=source, gateway
StripPrefix去除路径前缀- StripPrefix=1 # /api/user/1 → /user/1
RewritePath重写路径- RewritePath=/api/(?<segment>.*), /${segment}
RequestRateLimiter限流- name: RequestRateLimiter
CircuitBreaker熔断- name: CircuitBreaker

限流（RequestRateLimiter）

基于Redis的令牌桶算法实现分布式限流：

YAML
spring:
  cloud:
    gateway:
      routes:
        - id: rate-limited-route
          uri: lb://user-service
          predicates:
            - Path=/api/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100    # 每秒填充100个令牌
                redis-rate-limiter.burstCapacity: 200    # 桶容量200
                key-resolver: "#{@ipKeyResolver}"        # 限流维度
Java
@Bean
public KeyResolver ipKeyResolver() {
    // 基于IP限流
    return exchange -> Mono.just(
        exchange.getRequest().getRemoteAddress().getAddress().getHostAddress()
    );
}

@Bean
public KeyResolver userKeyResolver() {
    // 基于用户限流
    return exchange -> Mono.just(
        exchange.getRequest().getHeaders().getFirst("X-User-Id")
    );
}

自定义过滤器

全局过滤器（GlobalFilter）

所有请求都会经过全局过滤器，适合实现认证、日志等通用功能：

Java
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 白名单放行
        if (isWhiteList(path)) {
            return chain.filter(exchange);
        }

        // 获取Token
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (StringUtils.isEmpty(token)) {
            return unauthorized(exchange);
        }

        // 验证Token
        try {
            Claims claims = JwtUtil.parseToken(token);
            ServerHttpRequest request = exchange.getRequest().mutate()
                .header("X-User-Id", claims.get("userId", String.class))
                .build();
            return chain.filter(exchange.mutate().request(request).build());
        } catch (Exception e) {
            return unauthorized(exchange);
        }
    }

    @Override
    public int getOrder() {
        return -100; // 优先级高，数字越小优先级越高
    }
}

局部过滤器（GatewayFilterFactory）

Java
@Component
public class TimeLogGatewayFilterFactory
        extends AbstractGatewayFilterFactory<Config> {

    public TimeLogGatewayFilterFactory() {
        super(Config.class);
    }

    @Override
    public GatewayFilter apply(Config config) {
        return (exchange, chain) -> {
            long startTime = System.currentTimeMillis();
            return chain.filter(exchange).then(Mono.fromRunnable(() -> {
                long duration = System.currentTimeMillis() - startTime;
                exchange.getResponse().getHeaders()
                    .add("X-Response-Time", duration + "ms");
                if (config.isLogEnabled()) {
                    log.info("请求耗时: {}ms", duration);
                }
            }));
        };
    }

    public static class Config {
        private boolean logEnabled = true;
        // getter/setter
    }
}

高可用方案

  • 网关集群：部署多个网关实例，通过Nginx/SLB进行负载均衡
  • 无状态设计：网关不保存会话状态，请求可在任意实例处理
  • 健康检查：注册中心定期检测网关实例健康状态
  • 动态扩缩容：根据流量自动扩缩容网关实例
⚠️
注意事项 网关作为流量入口，需要做好自身的限流保护，避免成为系统瓶颈。建议单机QPS控制在1万以内。

全局过滤器实战：鉴权 + 限流

java
// 全局鉴权过滤器
@Component
@Order(-100)
public class AuthGlobalFilter implements GlobalFilter {

    private static final List<String> WHITE_LIST =
        List.of("/auth/login", "/auth/register", "/public/**");

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 白名单放行
        if (WHITE_LIST.stream().anyMatch(p -> pathMatcher.match(p, path))) {
            return chain.filter(exchange);
        }

        String token = exchange.getRequest().getHeaders()
            .getFirst(HttpHeaders.AUTHORIZATION);

        if (token == null || !token.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        try {
            Claims claims = jwtUtil.parseToken(token.substring(7));
            // 将用户信息透传到下游
            ServerHttpRequest request = exchange.getRequest().mutate()
                .header("X-User-Id", claims.getSubject())
                .header("X-User-Role", claims.get("role", String.class))
                .build();
            return chain.filter(exchange.mutate().request(request).build());
        } catch (JwtException e) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
    }
}

面试高频考点

Q1: Spring Cloud Gateway 的核心组件和工作流程？
核心组件：Route（路由，包含 id/uri/predicates/filters）、Predicate（断言，匹配请求条件）、Filter（过滤器，处理请求/响应）。工作流程：客户端请求 → Gateway Handler Mapping（匹配 Route）→ Gateway Web Handler（组装过滤器链）→ Pre Filter 链 → 代理请求到下游 → Post Filter 链 → 响应客户端。基于 Spring WebFlux + Reactor Netty，全异步非阻塞。
Q2: Gateway 过滤器的执行顺序？GlobalFilter vs GatewayFilter？
GlobalFilter 全局生效，通过 @Order 或实现 Ordered 接口控制顺序（值越小越先执行）；GatewayFilter 路由级别，在路由配置中按顺序执行。执行顺序：所有 Pre 阶段（GlobalFilter Pre + GatewayFilter Pre）→ 代理请求 → Post 阶段（逆序，GatewayFilter Post + GlobalFilter Post）。注意 Pre 和 Post 是同一 filter 方法中 chain.filter() 前后的代码。
Q3: 网关如何实现限流？常见方案对比？
①内置 RequestRateLimiter（基于 Redis + Lua 令牌桶）：开箱即用，支持按用户/IP/路径限流，需 Redis 依赖；②Sentinel 整合（spring-cloud-alibaba-sentinel-gateway）：功能更强，支持热点参数限流、系统自适应保护、实时监控控制台；③自定义 GlobalFilter + Redis：灵活，可实现复杂策略（如滑动窗口、漏桶）。生产推荐 Sentinel，监控和动态配置能力更强。
Q4: 网关和 Nginx 的职责分工？
Nginx：7 层/4 层代理，处理静态资源、SSL 卸载、负载均衡（到网关集群）、防 DDoS（连接数限制）；性能极高，适合边缘入口。Spring Cloud Gateway：微服务级路由、服务发现集成（lb://）、动态路由、鉴权、限流、熔断、链路追踪透传；能感知注册中心的服务变化。典型部署：外网 → Nginx（集群入口/SSL）→ Gateway（微服务路由）→ 各服务。

实战陷阱

⚠️
陷阱1：修改请求体后 Content-Length 不一致 在 GlobalFilter 中修改请求体（如添加字段）后，若不同步更新 Content-Length header，下游服务会因长度不匹配解析失败。应使用 ModifyRequestBodyGatewayFilterFactory 处理请求体修改，它会自动重新计算长度。
⚠️
陷阱2：跨域配置与过滤器冲突 同时在 Gateway 配置 CORS 和下游服务也配置 CORS 时，响应头中会出现重复的 Access-Control-Allow-Origin，导致浏览器拦截。应统一在 Gateway 层处理跨域（spring.cloud.gateway.globalcors），下游服务去除 CORS 配置。
⚠️
陷阱3：StripPrefix 数量设置错误导致路径错误 路由配置 Path=/api/orders/** 时，若 StripPrefix=1 只去掉 /api，下游收到 /orders/...；若 StripPrefix=2 去掉 /api/orders，下游收到 /...。错误的 StripPrefix 值会导致下游找不到路由（404）。应仔细计算路径层级，并在测试环境验证路径转发结果。
