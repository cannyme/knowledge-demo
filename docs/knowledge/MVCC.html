MVCC多版本并发控制

MVCC.java

通过维护数据的历史版本，实现非阻塞读。读操作读取历史版本，写操作创建新版本，读写互不阻塞。

什么是MVCC

MVCC（Multi-Version Concurrency Control） 即多版本并发控制，是一种数据库并发控制技术。它通过为每个事务创建数据的不同版本，使得读操作和写操作可以并发执行而不会互相阻塞。

核心思想

  • 写操作：创建数据的新版本，旧版本保留在系统中
  • 读操作：根据事务的隔离级别，读取合适版本的数据
  • 版本链：通过链表结构维护同一数据的多个历史版本

MVCC解决的问题

1. 读写冲突

在传统锁机制中，读操作需要加共享锁（S锁），写操作需要加排他锁（X锁）。读写操作会相互阻塞，降低并发性能。MVCC通过读取历史版本，避免了读锁和写锁的冲突。

2. 不可重复读与幻读

在可重复读（RR）隔离级别下，同一个事务内多次查询应该返回相同结果。MVCC通过 Read View 机制确保事务内看到的数据快照是一致的。

MVCC的主要优势：

  • 读操作不阻塞写操作
  • 写操作不阻塞读操作
  • 提高了数据库的并发处理能力
问题 传统锁机制 MVCC方案
读写冲突 读加S锁，写加X锁，相互阻塞 读历史版本，写创建新版本，互不阻塞
不可重复读 读时加锁直到事务结束 使用Read View保证一致性快照
并发性能 较低，锁竞争激烈 较高，读写分离

InnoDB MVCC实现原理

InnoDB通过以下三个核心组件实现MVCC：

1. 隐藏字段（Hidden Fields）

InnoDB在每行数据后面添加三个隐藏字段：

字段名 大小 说明
DB_TRX_ID 6字节 最后插入或更新该行的事务ID
DB_ROLL_PTR 7字节 回滚指针，指向Undo Log中的上一版本
DB_ROW_ID 6字节 行ID，自增（如果表没有主键时使用）

Undo Log用于存储数据的历史版本，当事务需要回滚或读取旧版本数据时使用。

3. Read View（读视图）

Read View是快照读时生成的视图，用于判断数据版本对当前事务的可见性。

判断规则详解

  1. DB_TRX_ID == creator_trx_id：当前事务修改的数据，可见
  2. DB_TRX_ID < min_trx_id：生成Read View时已提交的事务，可见
  3. DB_TRX_ID >= max_trx_id：生成Read View后开启的事务，不可见
  4. min_trx_id <= DB_TRX_ID < max_trx_id：
    • 如果在m_ids中：事务未提交，不可见
    • 如果不在m_ids中：事务已提交，可见

快照读 vs 当前读

快照读（Snapshot Read）

普通的SELECT语句，不加锁，读取历史版本数据。这是MVCC实现非阻塞读的基础。

快照读示例 
-- 普通SELECT，快照读，不加锁
SELECT * FROM user WHERE id = 1;

-- 在RR隔离级别下，多次查询结果一致
BEGIN;
SELECT * FROM user WHERE id = 1;  -- 生成Read View
-- 其他事务修改并提交
SELECT * FROM user WHERE id = 1;  -- 复用Read View，结果不变
COMMIT;

当前读（Current Read）

读取最新版本数据，并加锁。以下操作都属于当前读：

当前读示例 
-- 共享锁（S锁）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;
SELECT * FROM user WHERE id = 1 FOR SHARE;  -- MySQL 8.0+

-- 排他锁（X锁）
SELECT * FROM user WHERE id = 1 FOR UPDATE;

-- DML操作也是当前读
UPDATE user SET name = '王五' WHERE id = 1;
DELETE FROM user WHERE id = 1;
INSERT INTO user VALUES (1, '赵六');
特性 快照读 当前读
读取版本 历史版本 最新版本
是否加锁 不加锁 加锁（S锁或X锁）
性能 高 相对较低
一致性 事务级一致性 实时一致性

MVCC与事务隔离级别

READ UNCOMMITTED

直接读取最新版本数据，不使用MVCC机制。可能导致脏读、不可重复读、幻读。

READ COMMITTED（RC）

每次SELECT都生成新的Read View，因此能看到其他事务已提交的最新数据。

sequenceDiagram participant A as 事务A participant B as 事务B A->>A: BEGIN B->>B: BEGIN A->>A: SELECT (生成Read View R1) B->>B: UPDATE user SET name='李四' A->>A: SELECT (生成Read View R2，仍读旧数据) B->>B: COMMIT A->>A: SELECT (生成Read View R3，读到新数据) Note over A: 不可重复读！

REPEATABLE READ（RR）

第一次SELECT时生成Read View，后续SELECT复用同一个Read View，确保事务内多次读取结果一致。

sequenceDiagram participant A as 事务A participant B as 事务B A->>A: BEGIN B->>B: BEGIN A->>A: SELECT (生成Read View R1) Note right of A: 后续所有SELECT复用R1 B->>B: UPDATE user SET name='李四' A->>A: SELECT (使用R1，读旧版本) B->>B: COMMIT A->>A: SELECT (使用R1，仍读旧版本) Note over A: 可重复读！ A->>A: COMMIT

SERIALIZABLE

所有SELECT都加共享锁，不使用MVCC的快照读机制。完全串行化执行，隔离级别最高但性能最低。

MVCC解决幻读的原理

什么是幻读

幻读是指在同一事务中，两次相同条件的查询返回的行数不同，即"幻影行"的出现。

MVCC如何解决快照读的幻读

在RR隔离级别下，事务第一次执行SELECT时生成Read View。后续所有快照读都复用这个Read View，因此：

  • 即使其他事务插入了新数据并提交，当前事务也看不到这些新数据
  • 因为新数据的DB_TRX_ID >= max_trx_id，对当前事务不可见

当前读的幻读问题

MVCC只能解决快照读的幻读问题，当前读（如SELECT ... FOR UPDATE）的幻读需要通过Next-Key Lock（间隙锁+记录锁）来解决。

Next-Key Lock解决幻读 
-- 事务A
BEGIN;
SELECT * FROM user WHERE id > 5 FOR UPDATE;
-- 对id>5的范围加Next-Key Lock（间隙锁+记录锁）

-- 事务B（被阻塞）
BEGIN;
INSERT INTO user VALUES (6, '新用户');  -- 等待事务A释放锁

-- 事务A提交后，事务B才能继续
COMMIT;

注意：MVCC + Next-Key Lock的组合使得MySQL在RR隔离级别下基本解决了幻读问题。但在某些特殊场景（如当前读和快照读混用）下，仍可能出现幻读现象。

Purge机制（清理过期版本）

为什么需要Purge

随着事务的执行，Undo Log中会积累大量历史版本数据。如果不清理，会导致：

  • 磁盘空间持续增长
  • 版本链过长，查询性能下降
  • 表空间膨胀

Purge的工作原理

Purge线程负责清理不再需要的Undo Log记录：

flowchart LR A[Undo Log
历史版本] --> B{"是否还有
事务引用?"} B -->|是| C[保留] B -->|否| D[Purge清理] C --> E[等待] E --> B style D fill:#3a2e15,stroke:#f0b429,color:#f0b429

判断标准

  • 如果某版本DB_TRX_ID < 系统中活跃的最小事务ID（即没有事务需要读取该版本）
  • 且该版本不是任何其他版本链的必要节点
  • 则可以被Purge清理

长事务的危害

长事务会导致：

  • Undo Log无法及时清理，磁盘空间持续增长
  • 版本链过长，读取操作需要遍历更多版本，性能下降
  • 严重时可能导致表空间满，数据库无法写入

建议：尽量避免长事务，及时提交或回滚。

查看长事务 
-- 查看执行时间超过60秒的事务
SELECT
    trx_id,
    trx_mysql_thread_id,
    trx_state,
    TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS trx_seconds
FROM information_schema.innodb_trx
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 60;

MVCC核心要点回顾：

  • 隐藏字段：DB_TRX_ID、DB_ROLL_PTR、DB_ROW_ID
  • Undo Log：存储历史版本，形成版本链
  • Read View：定义可见性规则，RC每次生成，RR事务开始时生成
  • 快照读：读取历史版本，不加锁
  • 当前读：读取最新版本，加锁
  • Purge：清理过期版本，防止Undo Log无限增长
