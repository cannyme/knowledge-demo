分布式事务

DistributedTransaction.java

在分布式系统中，如何保证多个服务、多个数据库之间的数据一致性，是分布式事务需要解决的核心问题。

CAP定理与BASE理论

CAP定理

分布式系统的三个基本需求，最多只能同时满足两个：

  • C - Consistency（一致性）：所有节点在同一时刻看到相同的数据
  • A - Availability（可用性）：每个请求都能在有限时间内得到响应
  • P - Partition Tolerance（分区容错性）：网络分区时系统仍能正常运行

CAP选择原则
由于网络分区（P）在分布式系统中不可避免，因此实际只能在 CP 和 AP 之间选择：
  • CP：金融交易系统，强一致性优先
  • AP：互联网应用，可用性优先

BASE理论

BASE是对CAP中AP的延伸，强调最终一致性：

  • BA - Basically Available（基本可用）：系统出现不可预知故障时，允许损失部分可用性
  • S - Soft State（软状态）：允许系统存在中间状态，不影响整体可用性
  • E - Eventually Consistent（最终一致性）：所有数据副本经过一段时间后最终达到一致
ACID vs BASE
特性 ACID BASE

适用 传统关系型数据库 NoSQL、分布式系统

1. XA两阶段提交（2PC）

XA是X/Open组织提出的分布式事务规范，基于两阶段提交协议。

2PC流程
sequenceDiagram
    participant TM as 事务管理器(TM)
    participant RM1 as 资源管理器1(RM)
    participant RM2 as 资源管理器2(RM)

    Note over TM,RM2: 阶段一：准备阶段
    TM->>RM1: Prepare
    TM->>RM2: Prepare
    RM1-->>TM: Vote Yes/No
    RM2-->>TM: Vote Yes/No

    Note over TM,RM2: 阶段二：提交阶段
    alt 所有RM返回Yes
        TM->>RM1: Commit
        TM->>RM2: Commit
        RM1-->>TM: ACK
        RM2-->>TM: ACK
    else 任一RM返回No
        TM->>RM1: Rollback
        TM->>RM2: Rollback
        RM1-->>TM: ACK
        RM2-->>TM: ACK
    end
2PC问题
  • 同步阻塞：参与者在等待期间锁定资源
  • 单点故障：协调者宕机导致事务挂起
  • 数据不一致：Commit阶段部分参与者失败

2. TCC（Try-Confirm-Cancel）

TCC是一种补偿型事务，将业务逻辑分为三个阶段：

  • Try：尝试执行，预留资源
  • Confirm：确认执行，真正提交
  • Cancel：取消执行，释放资源
TCC转账示例 
public class TCCAccountService {

    // Try阶段：预留资源
    public boolean tryTransfer(String from, String to, BigDecimal amount) {
        // 1. 检查余额
        Account fromAcc = accountMapper.selectForUpdate(from);
        if (fromAcc.getBalance().compareTo(amount) < 0) {
            throw new BusinessException("余额不足");
        }

        // 2. 冻结金额
        accountMapper.freeze(from, amount);
        // UPDATE account SET balance = balance - ?, frozen = frozen + ?

        // 3. 记录事务日志
        transactionLogMapper.insert(new TransactionLog(
            transactionId, from, to, amount, "TRY"
        ));
        return true;
    }

    // Confirm阶段：确认提交
    public boolean confirmTransfer(String transactionId) {
        TransactionLog log = transactionLogMapper.selectById(transactionId);

        // 幂等性检查
        if ("CONFIRMED".equals(log.getStatus())) {
            return true;
        }

        // 1. 扣减冻结金额
        accountMapper.deductFrozen(log.getFromAccount(), log.getAmount());

        // 2. 增加目标账户余额
        accountMapper.addBalance(log.getToAccount(), log.getAmount());

        // 3. 更新事务状态
        transactionLogMapper.updateStatus(transactionId, "CONFIRMED");
        return true;
    }

    // Cancel阶段：回滚
    public boolean cancelTransfer(String transactionId) {
        TransactionLog log = transactionLogMapper.selectById(transactionId);

        // 幂等性检查
        if ("CANCELLED".equals(log.getStatus())) {
            return true;
        }

        // 1. 解冻金额
        accountMapper.unfreeze(log.getFromAccount(), log.getAmount());

        // 2. 更新事务状态
        transactionLogMapper.updateStatus(transactionId, "CANCELLED");
        return true;
    }
}

3. SAGA模式

Saga将长事务拆分为多个本地事务，每个本地事务执行成功后触发下一个。

SAGA模式流程

    I-->>O: 失败
    O->>O: C1: 取消订单
    Note over O,A: 订单创建失败

4. 本地消息表

将消息和业务操作放在同一事务中，通过定时任务发送消息。

本地消息表实现 
-- 消息表结构
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id VARCHAR(64) UNIQUE,
    target_service VARCHAR(64),
    content TEXT,
    status TINYINT DEFAULT 0,  -- 0待发送 1已发送 2已消费
    retry_count INT DEFAULT 0,
    create_time DATETIME,
    update_time DATETIME
);

// 业务操作 + 保存消息（同一事务）
@Transactional
public void doBusinessAndSaveMessage(Order order) {
    // 1. 执行业务操作
    orderMapper.insert(order);

    // 2. 保存消息（同一事务）
    LocalMessage message = new LocalMessage();
    message.setMessageId(UUID.randomUUID().toString());
    message.setTargetService("inventory-service");
    message.setContent(JSON.toJSONString(order));
    message.setStatus(0);
    localMessageMapper.insert(message);
}

5. 最大努力通知

通过消息通知对方，允许消息丢失，由接收方主动查询补偿。

  • 适用于对一致性要求不高的场景
  • 如支付回调、短信发送等
  • 接收方需要实现幂等和查询接口

三大核心组件
  • TC (Transaction Coordinator)：事务协调器，维护全局事务状态，驱动全局提交或回滚
  • TM (Transaction Manager)：事务管理器，定义全局事务范围，开启/提交/回滚全局事务
  • RM (Resource Manager)：资源管理器，管理分支事务，向TC注册分支并报告状态

Seata AT模式原理

AT模式基于SQL解析生成反向SQL，实现自动补偿。

// 1. 添加依赖
// <dependency>
//     <groupId>com.alibaba.cloud</groupId>
//     <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
// </dependency>

// 2. 配置seata
seata:
  enabled: true
  application-id: order-service
  tx-service-group: my_tx_group
  registry:
    type: nacos
    nacos:
      server-addr: localhost:8848

// 3. 业务代码中使用 @GlobalTransactional
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private InventoryClient inventoryClient;
    @Autowired
    private AccountClient accountClient;

    @GlobalTransactional
    public void createOrder(Order order) {
        // 1. 创建订单
        orderMapper.insert(order);

        // 2. 扣减库存（远程调用）
        inventoryClient.deduct(order.getProductId(), order.getCount());

        // 3. 扣减余额（远程调用）
        accountClient.deduct(order.getUserId(), order.getAmount());

        // 任意一步失败，自动回滚
    }
}

关键注意事项

1. 幂等性设计

所有补偿操作必须是幂等的，因为可能重复执行：

  • 使用唯一事务ID识别重复请求
  • 数据库状态机判断（如status字段）
  • Redis去重缓存
2. 空回滚与悬挂问题
  • 空回滚：Try未执行，Cancel先执行 - 需要记录事务状态
  • 悬挂：Cancel先于Try执行 - Try执行前检查事务状态
3. 监控与告警
  • 监控全局事务执行时间，超过阈值告警
  • 监控回滚比例，异常时触发告警
  • 定期对账，发现并处理不一致

避免分布式事务的设计

最好的分布式事务是没有分布式事务：

  • 业务拆分：将需要事务的操作放在同一服务内
  • 最终一致性：接受异步最终一致，而非强一致
  • 状态机：通过业务状态流转保证最终一致
  • 对账补偿：定期对账，发现差异后补偿
