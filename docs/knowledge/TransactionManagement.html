Spring事务管理

Spring事务管理的详细解析和最佳实践，包括声明式事务、传播行为、隔离级别及常见失效场景

🔒

事务ACID特性

事务是保证数据一致性的核心机制，ACID是事务的四个基本特性：

特性 英文 说明
A Atomicity（原子性） 事务是不可分割的最小单位，要么全部成功，要么全部失败回滚
C Consistency（一致性） 事务前后数据完整性约束不变，如余额不能为负数
I Isolation（隔离性） 并发事务之间互不干扰，一个事务的中间状态对其他事务不可见
D Durability（持久性） 事务提交后永久保存，即使系统崩溃也不会丢失
💡
理解要点Spring事务管理是对数据库事务的封装，通过AOP实现声明式事务，让开发者专注于业务逻辑而无需手动管理事务边界。
🛠️

Spring事务两种方式

方式 实现方式 适用场景 优缺点
声明式事务 @Transactional注解 大多数业务场景（推荐） 简洁、与业务解耦；但粒度控制不够灵活
编程式事务 TransactionTemplate 复杂事务逻辑、需要精细控制 灵活、可控；但代码侵入性强
🏷️

@Transactional 注解属性详解

java
@Transactional(
    // 事务管理器名称（多数据源时指定）
    transactionManager = "transactionManager",

    // 传播行为（默认REQUIRED）
    propagation = Propagation.REQUIRED,

    // 隔离级别（默认使用数据库默认）
    isolation = Isolation.READ_COMMITTED,

    // 超时时间（秒）
    timeout = 30,

    // 是否只读（可优化）
    readOnly = false,

    // 回滚异常（默认RuntimeException）
    rollbackFor = Exception.class,

    // 不回滚异常
    noRollbackFor = BusinessException.class
)
public void createOrder(Order order) {
    // 业务逻辑
}
属性 类型 默认值 说明
transactionManager String "" 指定事务管理器，多数据源时必须指定
propagation Propagation REQUIRED 事务传播行为，见下节详述
isolation Isolation DEFAULT 事务隔离级别，见下节详述
timeout int -1 事务超时时间（秒），-1表示不限制
readOnly boolean false 标记为只读事务，可优化性能（如MySQL不生成undo log）
rollbackFor Class[] {} 指定哪些异常触发回滚（默认RuntimeException）
noRollbackFor Class[] {} 指定哪些异常不触发回滚

事务传播行为（7种）

传播行为定义事务如何传播，即当前方法如何与已有事务交互：

传播行为 说明 使用场景
REQUIRED（默认） 有事务就加入，没有就新建 大多数业务方法，默认即可
SUPPORTS 有事务就加入，没有就以非事务运行 查询方法，事务非必须
MANDATORY 必须在事务中调用，否则抛异常 强制要求有事务的公共方法
REQUIRES_NEW 总是新建事务，挂起当前事务 日志记录、审计等独立操作
NOT_SUPPORTED 以非事务运行，挂起当前事务 无需事务的方法，避免事务开销
NEVER 以非事务运行，有事务则抛异常 严禁在事务中执行的方法
NESTED 有事务就嵌套执行，没有就新建 子操作可独立回滚的场景
📌
REQUIRED vs REQUIRES_NEW 对比

REQUIRED：所有操作在同一个事务中，任何失败都会全部回滚

REQUIRES_NEW：挂起当前事务 -> 新建事务执行 -> 恢复原事务，独立提交/回滚

java
// 主业务方法
@Transactional
public void createOrder(Order order) {
    saveOrder(order);           // 加入当前事务
    deductStock(order);         // 加入当前事务
    saveLog("创建订单");         // 独立事务，即使主事务失败也会记录
}

// 默认 REQUIRED - 加入现有事务
@Transactional
public void deductStock(Order order) {
    // 扣减库存
}

// REQUIRES_NEW - 新建独立事务
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog(String operation) {
    // 保存日志，独立提交
}

// NESTED - 嵌套事务（基于Savepoint）
@Transactional(propagation = Propagation.NESTED)
public void processItem(Item item) {
    // 内部事务回滚不影响外部，但外部回滚会影响内部
}

事务隔离级别

隔离级别决定事务之间相互影响的程度，解决并发事务带来的三种问题：

  • 脏读：读到其他事务未提交的数据
  • 不可重复读：同一事务两次读取结果不同（被其他事务修改）
  • 幻读：同一事务两次读取记录数不同（被其他事务插入/删除）
隔离级别 脏读 不可重复读 幻读 说明
READ_UNCOMMITTED ✗ 可能 ✗ 可能 ✗ 可能 性能最好，数据一致性最差
READ_COMMITTED ✓ 解决 ✗ 可能 ✗ 可能 Oracle默认级别
REPEATABLE_READ ✓ 解决 ✓ 解决 ✓ 解决* MySQL默认（MVCC解决幻读）
SERIALIZABLE ✓ 解决 ✓ 解决 ✓ 解决 完全串行，性能最差
💡
默认选择建议大多数情况下使用数据库默认级别即可。MySQL InnoDB的REPEATABLE_READ通过MVCC机制已能解决幻读问题。
java
// 指定隔离级别
@Transactional(isolation = Isolation.READ_COMMITTED)
public void queryData() {
    // 查询逻辑
}

事务失效场景（6种）

⚠️
❌ 1. 方法不是 public@Transactional 只对 public 方法有效，protected/private 方法不会生成代理
java
// ❌ 错误：非public方法
@Transactional
private void internalMethod() {  // 事务不会生效！
    // ...
}

// ✓ 正确：public方法
@Transactional
public void publicMethod() {
    // ...
}
⚠️
❌ 2. 自调用问题（最常见）同一个类中，A方法调用B方法，B方法的@Transactional失效。原因：Spring AOP基于代理，自调用不走代理。
java
@Service
public class OrderService {

    @Autowired
    private OrderService self;  // 注入自己（解决方案1）

    // 主方法
    public void methodA() {
        // ❌ 自调用，事务失效！
        this.methodB();

        // ✓ 解决方案1：通过注入的代理调用
        self.methodB();

        // ✓ 解决方案2：使用AopContext
        ((OrderService) AopContext.currentProxy()).methodB();
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        // 新事务
    }
}
💡
自调用解决方案
  • 注入自己：@Autowired private OrderService self; 然后通过self调用
  • AopContext：((Service) AopContext.currentProxy()).method()
  • 拆分到另一个类：将方法B移到另一个Service类中
⚠️
❌ 3. 异常被捕获如果捕获异常后没有抛出，事务不会回滚
java
// ❌ 错误：捕获异常未抛出
@Transactional
public void wrongCatch() {
    try {
        // 业务操作
        updateData();
    } catch (Exception e) {
        // 只打印不抛出，事务不会回滚！
        e.printStackTrace();
    }
}

// ✓ 正确：抛出异常或手动标记回滚
@Transactional(rollbackFor = Exception.class)
public void correctCatch() {
    try {
        updateData();
    } catch (Exception e) {
        // 手动标记回滚
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
}
⚠️
❌ 4. 异常类型不对默认只回滚 RuntimeException 和 Error，checked 异常（如 IOException）不会回滚
java
// ❌ 默认不回滚checked异常
@Transactional
public void checkedException() throws IOException {
    // 抛出IOException不会触发回滚
    throw new IOException("IO错误");
}

// ✓ 指定回滚所有异常
@Transactional(rollbackFor = Exception.class)
public void rollbackAll() throws IOException {
    throw new IOException("IO错误");  // 会回滚
}
⚠️
❌ 5. 数据库引擎不支持事务MySQL 的 MyISAM 引擎不支持事务，必须用 InnoDB
sql
-- 查看表使用的引擎
SHOW CREATE TABLE orders;

-- 修改引擎为InnoDB
ALTER TABLE orders ENGINE=InnoDB;
⚠️
❌ 6. 传播行为设置错误如 NOT_SUPPORTED、NEVER 会导致不在事务中执行
java
// NOT_SUPPORTED - 以非事务运行
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void noTransaction() {
    // 此方法不在事务中执行
}

编程式事务

使用 TransactionTemplate 进行细粒度事务控制：

java
@Service
public class OrderService {

    @Autowired
    private TransactionTemplate transactionTemplate;

    // 无返回值
    public void doSomething() {
        transactionTemplate.execute(status -> {
            try {
                // 业务操作
                saveOrder();
                deductStock();
            } catch (Exception e) {
                status.setRollbackOnly();
            }
            return null;
        });
    }

    // 带返回值
    public String doSomethingWithResult() {
        return transactionTemplate.execute(status -> {
            // 业务操作
            saveOrder();
            return "success";
        });
    }

    // 设置事务属性
    public void doWithConfig() {
        transactionTemplate.setPropagationBehavior(
            TransactionDefinition.PROPAGATION_REQUIRES_NEW);
        transactionTemplate.setIsolationLevel(
            TransactionDefinition.ISOLATION_READ_COMMITTED);
        transactionTemplate.setTimeout(30);

        transactionTemplate.execute(status -> {
            // 业务操作
            return null;
        });
    }
}
💡
何时使用编程式事务
  • 需要在事务中执行条件判断（根据条件决定是否回滚）
  • 需要在循环中批量提交（每N条提交一次）
  • 需要获取事务状态（如是否已回滚）
  • 复杂的嵌套事务控制

长事务问题及优化

⚠️
长事务的危害
  • 数据库连接占用时间长，连接池耗尽风险
  • 锁资源占用时间长，影响并发性能
  • 回滚代价大（大量undo log）
  • 主从延迟增加（长事务提交后才同步）
📌
优化方案
  • 缩小事务范围：非必要操作移到事务外
  • REQUIRES_NEW：让独立操作使用单独事务
  • 批量分批：大数据量分批处理，每批独立事务
  • 异步处理：非核心逻辑异步执行
java
@Service
public class OrderService {

    // ❌ 不推荐：事务范围过大
    @Transactional
    public void createOrderBad(Order order) {
        saveOrder(order);           // 核心业务
        deductStock(order);         // 核心业务
        sendNotification(order);    // 非核心：应该移出事务
        updateStatistics(order);    // 非核心：应该移出事务
        writeLog(order);            // 非核心：应该独立事务
    }

    // ✓ 推荐：精简事务范围
    @Transactional
    public void createOrderGood(Order order) {
        // 事务内只保留核心业务
        saveOrder(order);
        deductStock(order);
    }

    // 事务外：发送通知
    public void afterCreateOrder(Order order) {
        createOrderGood(order);
        sendNotification(order);    // 事务外执行
        updateStatistics(order);    // 事务外执行
    }

    // 独立事务：记录日志
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void writeLog(Order order) {
        // 独立提交，不影响主事务
    }

    // 事务提交后执行（事件监听）
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void afterOrderCreated(OrderCreatedEvent event) {
        // 事务提交后异步发送通知
        sendNotification(event.getOrder());
    }
}

分布式事务解决方案

方案原理一致性适用场景
2PC（两阶段提交）协调者准备+提交两阶段强一致对一致性要求极高，允许高延迟
TCC（Try-Confirm-Cancel）业务层补偿实现最终一致金融、支付场景
SAGA长事务拆分为子事务+补偿最终一致跨多服务的长流程
本地消息表消息+定时任务兜底最终一致跨服务数据同步
Seata AT 模式自动生成 undo log 回滚最终一致已有代码改造少
java
// Seata 分布式事务示例（AT 模式，@GlobalTransactional 注解）
@Service
public class OrderService {

    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO dto) {
        // 1. 创建订单（本地事务）
        orderMapper.insert(dto.toOrder());

        // 2. 扣减库存（远程调用，Seata 自动协调）
        inventoryService.deduct(dto.getProductId(), dto.getQty());

        // 3. 扣减账户余额（远程调用）
        accountService.deduct(dto.getUserId(), dto.getAmount());

        // 任何一步失败，Seata TC 协调所有分支事务回滚
    }
}
💡
Seata AT vs TCC AT 模式侵入性低，自动生成 undo log，适合快速接入；TCC 模式性能高，无全局锁，但需要手动实现 Try/Confirm/Cancel 三个接口，开发成本大。

面试高频考点

Q1: @Transactional 有哪些常见失效场景？
①方法非 public；②同类自调用（走 this 而非代理）；③异常被 catch 吞掉未抛出；④抛出 checked 异常但未配置 rollbackFor=Exception.class；⑤数据库引擎不支持事务（MyISAM）；⑥传播行为设为 NOT_SUPPORTED/NEVER；⑦多数据源下未指定正确的 transactionManager。
Q2: 事务传播行为 REQUIRED 和 REQUIRES_NEW 的区别？
REQUIRED：有事务就加入，没有则新建，所有操作在同一个事务中，任何失败都全部回滚。REQUIRES_NEW：总是新建事务，挂起当前事务，两个事务相互独立，子事务提交/回滚不影响父事务（但父事务回滚时，已提交的子事务不会回滚）。典型场景：日志/审计用 REQUIRES_NEW，保证即使业务失败也能记录日志。
Q3: Spring 事务是如何实现的（底层原理）？
Spring 事务基于 AOP 实现。@Transactional 触发 TransactionInterceptor 拦截，使用 PlatformTransactionManager（如 DataSourceTransactionManager）管理事务。底层通过 ThreadLocal 将 Connection 绑定到当前线程，保证同一事务内共用同一个连接。事务提交/回滚时从 ThreadLocal 取出连接操作。
Q4: 长事务有哪些危害？如何优化？
危害：数据库连接被长时间占用（连接池耗尽风险）、锁资源长时间持有（影响并发）、undo log 积压（回滚代价大）、主从延迟增加。优化：①缩小事务范围（将非数据库操作移出事务）②用 @Async 异步化非核心操作③REQUIRES_NEW 让独立操作单独提交④批量操作分批处理⑤使用 @TransactionalEventListener 在事务提交后处理后续逻辑。

实战陷阱

⚠️
陷阱1：readOnly=true 的事务中执行写操作 readOnly=true 告诉数据库驱动这是只读事务（MySQL InnoDB 不生成 undo log，性能更好），如果强行执行 INSERT/UPDATE，会抛出异常（Hibernate 模式）或静默成功（MyBatis 模式但数据可能丢失）。查询方法应正确使用 readOnly=true。
⚠️
陷阱2：在事务方法中调用外部 HTTP 接口 事务方法中调用远程 HTTP 接口，会导致数据库连接在 HTTP 等待期间被长时间持有（长事务）。正确做法：HTTP 调用放在事务外，或使用消息队列解耦，事务只处理本地数据库操作。
⚠️
陷阱3：NESTED 嵌套事务的使用限制 NESTED 基于 JDBC Savepoint 实现，仅 DataSourceTransactionManager 支持。JPA/Hibernate 的 JpaTransactionManager 不支持嵌套事务，会降级为 REQUIRED，若误用可能导致部分回滚失效。建议在确认事务管理器支持后再使用 NESTED。
