服务注册与发现

微服务架构的核心基础设施，解决服务间如何相互找到对方的问题

❓

为什么需要服务发现

在单体架构中，服务调用通过固定的IP和端口完成。但在微服务架构中，服务实例动态变化，需要服务发现机制来管理：

flowchart LR subgraph Before["传统方式：硬编码地址"] A[订单服务] -->|http://192.168.1.100:8080| B[用户服务] style Before fill:rgba(255,123,114,0.1) end subgraph After["服务发现：动态寻址"] C[订单服务] -->|registry.lookup"user-service"| D[(注册中心)] D -->|返回可用实例| E[用户服务实例1] D -->|返回可用实例| F[用户服务实例2] D -->|返回可用实例| G[用户服务实例3] style After fill:rgba(126,231,135,0.1) end Before -.->|演进| After

微服务带来的挑战

  • 服务实例动态扩缩容，IP不固定
  • 服务故障后需要自动剔除
  • 服务间调用关系复杂
  • 需要负载均衡分配流量

服务发现的价值

  • 服务间解耦，无需知道对方地址
  • 自动感知服务上下线
  • 支持动态负载均衡
  • 提供健康检查机制
核心概念：服务注册——服务启动时向注册中心注册自己的地址；服务发现——服务调用时从注册中心获取目标服务的可用地址列表。
🔄

服务注册与发现流程

完整的服务注册与发现流程包含以下步骤：

sequenceDiagram participant P as Provider participant R as Registry participant C as Consumer Note over P,R: 服务注册阶段 P->>P: 服务启动 P->>R: 1. register(ip, port, serviceName) R-->>P: 注册成功 Note over C,R: 服务发现阶段 C->>R: 2. subscribe(serviceName) R-->>C: 3. 返回服务列表 R->>C: 4. 推送服务变更（异步） Note over P,C: 服务调用阶段 C->>P: 5. 直接调用（无需经过注册中心） Note over P,R: 心跳保活 loop 每5秒 P->>R: 6. heartbeat end Note over P,R: 服务下线 P->>R: 7. deregister（优雅关闭） R->>C: 8. 推送服务下线通知
步骤操作说明
1服务注册Provider启动时向注册中心发送注册请求（IP、端口、服务名、元数据）
2服务订阅Consumer启动时向注册中心订阅需要调用的服务
3地址推送注册中心将可用服务地址列表推送给Consumer
4变更通知服务列表变更时，注册中心主动推送通知Consumer
5服务调用Consumer缓存服务地址，调用时直接访问Provider
6心跳保活Provider定时发送心跳，证明自己存活
7服务下线Provider关闭时主动注销，或心跳超时被动剔除
🔀

客户端发现 vs 服务端发现

客户端发现模式：服务消费者自己从注册中心获取服务列表，并选择目标实例：

flowchart LR C[Consumer] -->|1. 查询服务| R[(注册中心)] R -->|2. 返回实例列表| C C -->|3. 选择并调用| P1[Provider 1] C -->|3. 选择并调用| P2[Provider 2] style C fill:rgba(240,180,41,0.2) style R fill:rgba(78,205,196,0.2)

优点

  • 架构简单，无需额外组件
  • Consumer可灵活实现负载均衡
  • 调用延迟低（直连Provider）

缺点

  • Consumer需要集成服务发现SDK
  • 多语言环境需要各自实现
  • 无法统一控制流量
典型实现：Eureka、Nacos、Consul + Spring Cloud LoadBalancer、Dubbo

服务端发现模式：通过负载均衡器统一代理服务调用，Consumer无需感知服务发现细节：

flowchart LR C[Consumer] -->|1. 调用服务| LB[负载均衡器/API网关] LB -->|2. 查询服务| R[(注册中心)] R -->|3. 返回实例| LB LB -->|4. 转发请求| P1[Provider 1] LB -->|4. 转发请求| P2[Provider 2] style LB fill:rgba(240,180,41,0.2) style R fill:rgba(78,205,196,0.2)

优点

  • Consumer无需关注服务发现
  • 可统一控制流量和安全策略
  • 支持异构语言服务

缺点

  • 增加网络跳数，延迟增加
  • 负载均衡器成为单点
  • 需要维护额外的基础设施
典型实现：Kubernetes Service、AWS ELB、Nginx + Consul Template、Spring Cloud Gateway

Eureka架构

Eureka是Netflix开源的服务注册中心，采用AP架构，强调高可用性：

flowchart TB subgraph Server["Eureka Server集群"] S1[Eureka Server 1] <-->|复制| S2[Eureka Server 2] S2 <-->|复制| S3[Eureka Server 3] end subgraph Client["Eureka Client"] P1[Provider] -->|注册| S1 P2[Provider] -->|注册| S2 C1[Consumer] -->|发现| S1 end S1 -.->|自我保护| S1 style Server fill:rgba(240,180,41,0.1) style Client fill:rgba(78,205,196,0.1)
组件职责
Eureka Server服务注册中心，维护服务注册表，提供服务注册与发现接口
Eureka Client服务提供者/消费者，负责服务注册、心跳续约、服务获取
服务注册表内存中的服务实例信息，Server间通过异步复制同步

自我保护机制

当网络分区发生时，Eureka保护服务注册表不被误清理：

flowchart TB A[正常状态] -->|心跳续约| B[服务健康] A -->|网络故障| C[大量服务失联] C -->|15分钟内丢失>85%心跳| D[进入自我保护模式] D -->|停止剔除| E[保留所有注册信息] E -->|网络恢复| A style D fill:rgba(240,180,41,0.3) style E fill:rgba(126,231,135,0.2)
注意：自我保护模式下，Eureka不会剔除任何服务，可能导致Consumer调用到已宕机的服务。需要在网络恢复后及时退出保护模式。
YAML
# Eureka Server配置
eureka:
  server:
    # 关闭自我保护（开发环境）
    enable-self-preservation: false
    # 清理间隔（毫秒）
    eviction-interval-timer-in-ms: 5000
  instance:
    hostname: localhost

# Eureka Client配置
eureka:
  client:
    service-url:
      defaultZone: http://eureka1:8761/eureka,http://eureka2:8762/eureka
    # 注册自己
    register-with-eureka: true
    # 获取注册表
    fetch-registry: true
  instance:
    # 心跳间隔（秒）
    lease-renewal-interval-in-seconds: 30
    # 服务过期时间（秒）
    lease-expiration-duration-in-seconds: 90

Nacos服务发现

Nacos是阿里巴巴开源的动态服务发现、配置管理和服务管理平台，功能全面：

flowchart TB subgraph Nacos["Nacos集群"] N1[Nacos Node 1 Leader] N2[Nacos Node 2 Follower] N3[Nacos Node 3 Follower] N1 <-->|Raft| N2 N2 <-->|Raft| N3 N1 <-->|Raft| N3 end DB[(MySQL)] --> N1 DB --> N2 DB --> N3 P1[Provider] -->|注册| N1 C1[Consumer] -->|订阅| N2 style Nacos fill:rgba(240,180,41,0.1) style DB fill:rgba(126,231,135,0.1)
特性说明
AP/CP切换支持临时实例（AP）和持久实例（CP）两种模式
集群模式基于Raft协议实现数据一致性
数据持久化服务信息持久化到MySQL，避免数据丢失
配置中心集成配置管理功能，支持动态配置推送
元数据支持丰富的服务元数据，用于灰度发布、路由等
YAML
# Nacos服务发现配置
spring:
  application:
    name: user-service
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
        # 命名空间（环境隔离）
        namespace: dev
        # 分组
        group: DEFAULT_GROUP
        # 服务名
        service: ${spring.application.name}
        # 权重
        weight: 1
        # 元数据
        metadata:
          version: 1.0.0
          region: beijing
        # 临时实例（false=持久实例，服务端探测）
        ephemeral: true
Nacos心跳机制：客户端心跳间隔5秒，服务端15秒未收到心跳标记为不健康，30秒未收到则从列表移除。

Consul服务发现

Consul是HashiCorp出品的开源服务网格解决方案，提供服务发现、健康检查、Key/Value存储等功能：

flowchart TB subgraph Consul["Consul集群"] C1[Consul Server Leader] C2[Consul Server] C3[Consul Server] C1 <-->|Raft| C2 C2 <-->|Raft| C3 end A1[Consul Agent] -->|注册| C1 A2[Consul Agent] -->|健康检查| C2 P1[Service] --> A1 P2[Service] --> A2 C1 -->|DNS/HTTP| D[Consumer] style Consul fill:rgba(240,180,41,0.1)
特性说明
CP架构基于Raft协议保证强一致性，优先保证数据正确性
多数据中心原生支持多数据中心部署，适合全球化部署
健康检查支持HTTP/TCP/Script等多种健康检查方式
DNS服务内置DNS服务器，可直接通过DNS解析服务地址
KV存储支持Key/Value存储，可用于配置管理
JSON
// Consul服务定义配置
{
  "service": {
    "name": "user-service",
    "tags": ["v1", "java"],
    "port": 8080,
    "check": {
      "http": "http://localhost:8080/health",
      "interval": "10s",
      "timeout": "5s"
    },
    "meta": {
      "version": "1.0.0",
      "region": "us-east-1"
    }
  }
}
适用场景：Consul适合需要强一致性、多数据中心部署、异构语言集成的场景。

// 优雅下线示例
@Component
public class GracefulShutdown {

    @PreDestroy
    public void shutdown() {
        // 1. 从注册中心注销
        namingService.deregisterInstance(
            serviceName, ip, port);

        // 2. 等待正在处理的请求完成
        try {
            Thread.sleep(5000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }

        // 3. 关闭服务
    }
}

Eureka 适用场景

  • 已有Spring Cloud Netflix项目
  • 追求AP高可用
  • 简单服务发现需求

Nacos 适用场景

  • 新项目首选（推荐）
  • 需要配置中心集成
  • 阿里生态/K8s环境

Consul 适用场景

  • 需要强一致性
  • 多数据中心部署
  • 异构语言集成

ZooKeeper 适用场景

  • Dubbo传统项目
  • 需要强一致性
  • 已有ZK集群
选型建议：新项目优先选择Nacos，功能全面且活跃维护；已有Eureka项目可逐步迁移到Nacos；需要多数据中心场景考虑Consul。
