服务注册与发现

微服务架构的核心基础设施，解决服务间如何相互找到对方的问题

❓

为什么需要服务发现

在单体架构中，服务调用通过固定的IP和端口完成。但在微服务架构中，服务实例动态变化，需要服务发现机制来管理：

flowchart LR subgraph Before["传统方式：硬编码地址"] A[订单服务] -->|http://192.168.1.100:8080| B[用户服务] end subgraph After["服务发现：动态寻址"] C[订单服务] -->|registry.lookup| D[(注册中心)] D -->|返回可用实例| E[用户服务实例1] D -->|返回可用实例| F[用户服务实例2] D -->|返回可用实例| G[用户服务实例3] end Before -.->|演进| After classDef beforeStyle fill:rgba(255,123,114,0.15),stroke:#ff5f56,stroke-width:2px classDef afterStyle fill:rgba(126,231,135,0.15),stroke:#27c93f,stroke-width:2px class A,B beforeStyle class C,D,E,F,G afterStyle

微服务带来的挑战

  • 服务实例动态扩缩容，IP不固定
  • 服务故障后需要自动剔除
  • 服务间调用关系复杂
  • 需要负载均衡分配流量

服务发现的价值

  • 服务间解耦，无需知道对方地址
  • 自动感知服务上下线
  • 支持动态负载均衡
  • 提供健康检查机制
🎯
核心概念 服务注册——服务启动时向注册中心注册自己的地址；服务发现——服务调用时从注册中心获取目标服务的可用地址列表。
🔄

服务注册与发现流程

完整的服务注册与发现流程包含以下步骤：

sequenceDiagram participant P as Provider participant R as Registry participant C as Consumer Note over P,R: 服务注册阶段 P->>P: 服务启动 P->>R: 1. register(ip, port, serviceName) R-->>P: 注册成功 Note over C,R: 服务发现阶段 C->>R: 2. subscribe(serviceName) R-->>C: 3. 返回服务列表 R->>C: 4. 推送服务变更（异步） Note over P,C: 服务调用阶段 C->>P: 5. 直接调用（无需经过注册中心） Note over P,R: 心跳保活 loop 每5秒 P->>R: 6. heartbeat end Note over P,R: 服务下线 P->>R: 7. deregister（优雅关闭） R->>C: 8. 推送服务下线通知
步骤操作说明
1服务注册Provider启动时向注册中心发送注册请求（IP、端口、服务名、元数据）
2服务订阅Consumer启动时向注册中心订阅需要调用的服务
3地址推送注册中心将可用服务地址列表推送给Consumer
4变更通知服务列表变更时，注册中心主动推送通知Consumer
5服务调用Consumer缓存服务地址，调用时直接访问Provider
6心跳保活Provider定时发送心跳，证明自己存活
7服务下线Provider关闭时主动注销，或心跳超时被动剔除

客户端发现 vs 服务端发现

客户端发现模式：服务消费者自己从注册中心获取服务列表，并选择目标实例：

优点

  • 架构简单，无需额外组件
  • Consumer可灵活实现负载均衡
  • 调用延迟低（直连Provider）

缺点

  • Consumer需要集成服务发现SDK
  • 多语言环境需要各自实现
  • 无法统一控制流量
💡
典型实现 Eureka、Nacos、Consul + Spring Cloud LoadBalancer、Dubbo

服务端发现模式：通过负载均衡器统一代理服务调用，Consumer无需感知服务发现细节：

优点

  • Consumer无需关注服务发现
  • 可统一控制流量和安全策略
  • 支持异构语言服务

缺点

  • 增加网络跳数，延迟增加
  • 负载均衡器成为单点
  • 需要维护额外的基础设施

自我保护机制

当网络分区发生时，Eureka保护服务注册表不被误清理：

flowchart TB A[正常状态] -->|心跳续约| B[服务健康] A -->|网络故障| C[大量服务失联] C -->|15分钟内丢失>85%心跳| D[进入自我保护模式] D -->|停止剔除| E[保留所有注册信息] E -->|网络恢复| A classDef warningStyle fill:rgba(240,180,41,0.3),stroke:#f0b429,stroke-width:2px classDef successStyle fill:rgba(126,231,135,0.2),stroke:#27c93f,stroke-width:2px class D warningStyle class E successStyle
⚠️
注意 自我保护模式下，Eureka不会剔除任何服务，可能导致Consumer调用到已宕机的服务。需要在网络恢复后及时退出保护模式。
yaml
# Eureka Server配置
eureka:
  server:
    # 关闭自我保护（开发环境）
    enable-self-preservation: false
    # 清理间隔（毫秒）
    eviction-interval-timer-in-ms: 5000
  instance:
    hostname: localhost

# Eureka Client配置
eureka:
  client:
    service-url:
      defaultZone: http://eureka1:8761/eureka,http://eureka2:8762/eureka
    # 注册自己
    register-with-eureka: true
    # 获取注册表
    fetch-registry: true
  instance:
    # 心跳间隔（秒）
    lease-renewal-interval-in-seconds: 30
    # 服务过期时间（秒）
    lease-expiration-duration-in-seconds: 90

@Autowired

public String callUserService() {
    List<ServiceInstance> instances =
        discoveryClient.getInstances("user-service");

面试高频考点

Q1: 服务注册与发现的核心流程是什么？
①服务提供者启动时向注册中心注册（IP、端口、服务名、元数据）；②注册中心存储服务列表；③服务消费者向注册中心查询服务列表（拉模式）或注册中心推送变更（推模式）；④消费者本地缓存服务列表，通过负载均衡选择实例发起调用；⑤服务提供者定期发送心跳保活，注册中心若长时间未收到心跳则剔除实例。
Q2: CAP 理论与注册中心的选择？Nacos 如何做到 AP/CP 可切换？
CAP：一致性(C)、可用性(A)、分区容错性(P) 三者只能同时满足两个。注册中心一般选 AP（网络分区时仍提供服务列表，允许短暂数据不一致）。Nacos 通过实例类型切换：临时实例（ephemeral=true，默认）走 Distro 协议（AP）；永久实例（ephemeral=false）走 Raft 协议（CP）。临时实例适合微服务，永久实例适合不会心跳的基础设施。
Q3: 服务实例下线时如何保证请求不丢失？
①优雅关机：Spring Boot 配置 server.shutdown=graceful，处理完当前请求再关闭；②预停止钩子：K8s preStop hook 先调用 /actuator/shutdown 或等待若干秒；③注册中心摘除延迟：服务向注册中心注销，但客户端缓存可能尚未刷新，期间仍有请求打到旧实例，需配合重试机制；④健康检查摘除：注册中心停止健康检查后等待 2 个心跳周期再摘除。
Q4: Nacos 集群如何保证高可用？
Nacos 集群采用 Raft 协议（CP 模式下）选主，Leader 负责写，Follower 同步数据。推荐部署 3 或 5 节点（奇数），容忍 (N-1)/2 个节点故障。AP 模式（Distro 协议）下每个节点都可处理请求，数据异步复制，不存在单点问题。客户端配置多个 Nacos 地址（cluster-server-addr），任一节点不可用自动切换。

实战陷阱

⚠️
陷阱1：注册中心保护模式导致服务列表不更新 Eureka 的自我保护模式（eureka.server.enable-self-preservation=true）在心跳丢失超过 85% 时不剔除实例，开发环境频繁重启导致服务列表中有大量已死亡的实例。Nacos 同样有保护阈值配置（protectThreshold），建议开发环境关闭保护，生产环境保留并配合健康检查。
⚠️
陷阱2：服务实例元数据版本不一致导致灰度发布混乱 多版本同时在线时（灰度发布），若负载均衡规则未区分版本，新旧版本实例会被随机调用。应在服务元数据中标记 version 字段，并在负载均衡策略（如 Nacos 权重/标签路由）中正确配置流量分配，避免新旧版本混调导致的兼容性问题。
⚠️
陷阱3：客户端本地缓存导致服务摘除后仍被调用 注册中心摘除实例后，消费者本地缓存（默认 30 秒刷新）可能继续将请求发往已下线实例。解决：①配合熔断/重试（Retry + CircuitBreaker）兜底；②减小缓存刷新间隔；③优先使用 Nacos 推送模式（push 更新比 pull 更及时）。
