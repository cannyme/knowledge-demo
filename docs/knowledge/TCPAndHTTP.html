TCP三次握手

TCP三次握手用于建立可靠的连接，确保双方的发送和接收能力都正常。

sequenceDiagram participant C as Client participant S as Server C->>S: SYN=1, seq=x Note right of C: 请求建立连接
客户端进入SYN_SENT S-->>C: SYN=1, ACK=1, seq=y, ack=x+1 Note left of S: 确认并同意建立
服务端进入SYN_RCVD C->>S: ACK=1, seq=x+1, ack=y+1 Note right of C: 确认服务端的确认
双方进入ESTABLISHED Note over C,S: 连接建立成功，可以传输数据
字段说明
SYNSynchronize，请求建立连接
ACKAcknowledge，确认
seqSequence Number，序列号
ackAcknowledge Number，确认号（期望收到的下一个序号）
为什么是三次握手？
防止已失效的连接请求突然到达服务端。假设两次握手：客户端发送的SYN在网络中滞留，客户端超时重发建立连接后，滞留的SYN到达，服务端误认为是新连接，造成资源浪费。

TCP四次挥手

TCP是全双工通信，每个方向需要单独关闭，因此需要四次挥手。

sequenceDiagram participant C as Client participant S as Server C->>S: FIN=1, seq=u Note right of C: 请求关闭连接
客户端进入FIN_WAIT_1 S-->>C: ACK=1, seq=v, ack=u+1 Note left of S: 确认关闭请求
服务端进入CLOSE_WAIT Note over S: 服务端可能还有数据要发送 S-->>C: FIN=1, ACK=1, seq=w, ack=u+1 Note left of S: 服务端请求关闭
服务端进入LAST_ACK C->>S: ACK=1, seq=u+1, ack=w+1 Note right of C: 确认服务端关闭
客户端进入TIME_WAIT Note over C: 等待2MSL后关闭

状态变化

  • Client: ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
  • Server: ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
为什么TIME_WAIT要等2MSL？
MSL(Maximum Segment Lifetime)是报文最大生存时间。等待2MSL是为了：1. 确保最后一个ACK能到达服务端；2. 等待网络中旧报文消失，防止影响新连接。

TCP vs UDP

特性TCPUDP
连接面向连接无连接

顺序有序无序
流量控制有（滑动窗口）无

TCP可靠传输机制

1. 序列号与确认应答

  • 每个字节都有序号
  • 接收方返回确认号，表示期望收到的下一个序号

2. 超时重传

  • 发送数据后启动定时器
  • 超时未收到ACK则重传
  • RTO(Retransmission Timeout)动态计算

3. 滑动窗口（流量控制）

4. 拥塞控制

算法说明
慢开始拥塞窗口从1开始，每轮翻倍（指数增长）
拥塞避免达到阈值后，每轮加1（线性增长）
快重传收到3个重复ACK立即重传，不等待超时
快恢复拥塞窗口减半，而不是降到1

连接短连接长连接（默认）多路复用基于QUIC

flowchart TD subgraph H11["HTTP/1.1 队头阻塞"] R1[Request 1] --> S1[Response 1] R2[Request 2] -.等待.-> S2[Response 2] R3[Request 3] -.等待.-> S3[Response 3] end subgraph H2["HTTP/2 多路复用"] F1[Frame 1] --> M[合并传输] F2[Frame 2] --> M F3[Frame 3] --> M M --> Stream end
HTTP/1.1 队头阻塞：前一个请求未完成，后一个请求必须等待。管道化不实用，浏览器默认关闭。
HTTP/3 (QUIC)：基于UDP实现可靠传输，解决TCP层面的队头阻塞问题。

HTTPS原理

HTTPS = HTTP + TLS/SSL，通过加密保证通信安全。

sequenceDiagram participant C as Client participant S as Server C->>S: ClientHello
支持的加密套件、随机数 S-->>C: ServerHello
选定的加密套件、随机数 S-->>C: Certificate
服务器证书 S-->>C: ServerHelloDone C->>C: 验证证书 C->>C: 生成预主密钥 C->>S: ClientKeyExchange
加密的预主密钥 C->>S: ChangeCipherSpec C->>S: Finished S->>S: 私钥解密预主密钥 S-->>C: ChangeCipherSpec S-->>C: Finished Note over C,S: 双方生成会话密钥
开始加密通信

密钥交换过程

  1. 客户端生成随机预主密钥(Pre-Master Secret)
  2. 用服务器公钥加密发送
  3. 服务器私钥解密获得预主密钥
  4. 双方用预主密钥+两个随机数生成会话密钥
算法类型算法用途
对称加密AES通信数据加密
非对称加密RSA/ECC密钥交换
哈希算法SHA256完整性校验

跨域问题（CORS）

同源策略：协议+域名+端口必须相同，否则浏览器会阻止请求。

CORS响应头

HTTP
# 允许的来源
Access-Control-Allow-Origin: https://example.com

# 允许的方法
Access-Control-Allow-Methods: GET, POST, PUT, DELETE

# 允许的头部
Access-Control-Allow-Headers: Content-Type, Authorization

# 允许携带Cookie
Access-Control-Allow-Credentials: true

# 预检缓存时间
Access-Control-Max-Age: 86400

跨域解决方案

  • CORS：服务端设置响应头（推荐）
  • JSONP：利用script标签不受同源策略限制（只支持GET）
  • 代理服务器：Nginx反向代理
  • postMessage：用于iframe跨域通信

常见面试题

Q1: 三次握手为什么不是两次？
A: 防止已失效的连接请求突然到达服务端，造成资源浪费。
Q2: 四次挥手为什么TIME_WAIT要等2MSL？
A: 确保最后一个ACK能到达服务端，等待网络中旧报文消失。
Q3: GET和POST的区别？
A: GET幂等、可缓存、参数在URL（长度限制）；POST非幂等、不可缓存、参数在Body。
Q4: HTTPS如何保证安全？
A: 通过TLS协议，使用非对称加密交换密钥，对称加密通信数据。
Q5: HTTP/2相比HTTP/1.1有什么改进？
A: 多路复用、二进制分帧、头部压缩(HPACK)、服务器推送。
