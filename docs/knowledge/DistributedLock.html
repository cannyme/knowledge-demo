分布式锁

DistributedLock.java

在分布式系统中，多个服务实例部署在不同机器上，需要跨进程的锁机制来保证资源访问的互斥性。

为什么需要分布式锁

单机环境下可以使用 synchronized、ReentrantLock 等JVM级别的锁来保证线程安全。但在分布式系统中：

  • 多个服务实例部署在不同机器上
  • 需要跨JVM的互斥机制
  • 多个进程可能同时访问共享资源

分布式锁的核心要求
  • 互斥性：任意时刻只有一个客户端持有锁
  • 防死锁：锁必须能释放（超时释放）
  • 高可用：锁服务稳定可靠
  • 可重入：同一线程可多次获取同一把锁
  • 防误删：只能删除自己加的锁

分布式锁的实现方式对比

实现方式 优点 缺点 适用场景
数据库 实现简单，无需额外组件 性能差，可能死锁 简单场景，已有数据库

Redis分布式锁

基本实现（SET key value NX EX）

Redis SETNX 实现 
public class RedisDistributedLock {

    private static final String LOCK_PREFIX = "lock:";
    private final StringRedisTemplate redisTemplate;

    /**
     * 加锁：使用 SET key value NX EX 原子操作
     * NX - 只有key不存在时才设置
     * EX - 设置过期时间（秒）
     */
    public boolean tryLock(String lockKey, String requestId, long expireSeconds) {
        String key = LOCK_PREFIX + lockKey;
        return Boolean.TRUE.equals(
            redisTemplate.opsForValue()
                .setIfAbsent(key, requestId, expireSeconds, TimeUnit.SECONDS)
        );
    }

    /**
     * 解锁：使用Lua脚本保证原子性
     * 先判断value是否匹配，再删除key
     */
    public boolean releaseLock(String lockKey, String requestId) {
        String key = LOCK_PREFIX + lockKey;
        String luaScript =
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "    return redis.call('del', KEYS[1]) " +
            "else " +
            "    return 0 " +
            "end";

        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(luaScript, Long.class),
            Collections.singletonList(key),
            requestId
        );
        return Long.valueOf(1).equals(result);
    }
}
为什么必须用Lua脚本？
因为"判断value + 删除key"必须是原子操作。否则可能出现：
  1. 线程A判断value相等
  2. 锁过期，线程B获取锁
  3. 线程A删除了B的锁（误删）

锁续期问题（看门狗机制）

问题：业务执行时间 > 锁过期时间，导致锁提前释放

解决：后台线程定期延长锁过期时间

看门狗机制
sequenceDiagram
    participant App as 应用线程
    participant Lock as Redis锁
    participant Dog as 看门狗线程

    App->>Lock: SET lock:order EX 30 NX
    Lock-->>App: OK (获取锁成功)
    Dog->>Dog: 启动，每10秒执行一次
    Note over Dog: 检查锁是否仍被持有
    Dog->>Lock: 如果key存在，EXPIRE 30
    Note over App,Dog: 业务执行中（可能超过30秒）
    Dog->>Lock: 继续续期...
    App->>Lock: DEL lock:order (业务完成)
    Dog->>Dog: 发现锁已释放，停止线程
看门狗实现 
public void tryLockWithWatchdog(String lockKey, String requestId) {
    if (tryLock(lockKey, requestId, 30)) {
        ScheduledExecutorService scheduler =
            Executors.newSingleThreadScheduledExecutor();

        scheduler.scheduleAtFixedRate(() -> {
            if (redisTemplate.hasKey(LOCK_PREFIX + lockKey)) {
                // 续期到30秒
                redisTemplate.expire(
                    LOCK_PREFIX + lockKey,
                    30,
                    TimeUnit.SECONDS
                );
            } else {
                scheduler.shutdown(); // 锁已释放，停止看门狗
            }
        }, 10, 10, TimeUnit.SECONDS); // 每10秒检查一次
    }
}

Redisson使用

Redisson是Redis分布式锁的最佳实践，封装了看门狗等机制

Redisson配置与使用 
// Maven依赖
// <dependency>
//     <groupId>org.redisson</groupId>
//     <artifactId>redisson-spring-boot-starter</artifactId>
//     <version>3.20.0</version>
// </dependency>

@Autowired
private RedissonClient redisson;

public void businessMethod(String orderId) {
    RLock lock = redisson.getLock("lock:order:" + orderId);

    try {
        // 尝试获取锁，最多等待100秒
        // 如果leaseTime=-1，启用看门狗自动续期（默认30秒）
        boolean acquired = lock.tryLock(100, -1, TimeUnit.SECONDS);

        if (acquired) {
            // 执行业务逻辑...
            processOrder(orderId);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        // 只释放自己持有的锁
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
Redisson看门狗配置
  • 默认锁过期时间：30秒（lockWatchdogTimeout）
  • 看门狗检查间隔：10秒（过期时间的1/3）
  • 如果指定了leaseTime，则禁用看门狗

主从延迟问题（RedLock）

问题场景：Redis主从架构下，Master宕机导致锁丢失

  1. 客户端A在Master获取锁
  2. Master宕机，锁数据未同步到Slave
  3. Slave升级为Master
  4. 客户端B也能获取同一把锁

RedLock争议
Redis作者antirez提出RedLock，但分布式系统专家Martin Kleppmann认为：
  • RedLock依赖系统时钟，时钟漂移可能导致问题
  • 如果需要强一致性，建议使用ZooKeeper或etcd
建议：大多数场景Redisson足够，强一致性需求考虑ZooKeeper

ZooKeeper分布式锁

临时顺序节点实现

ZooKeeper分布式锁利用以下特性：

  • 临时节点：会话结束时自动删除，防止死锁
  • 顺序节点：自动编号，实现公平锁
  • Watcher：监听节点变化，实现阻塞等待

Curator实现ZK锁 
// Maven依赖
// <dependency>
//     <groupId>org.apache.curator</groupId>
//     <artifactId>curator-recipes</artifactId>
//     <version>5.5.0</version>
// </dependency>

public class ZookeeperLockDemo {

    private CuratorFramework client;

    public void init() {
        client = CuratorFrameworkFactory.newClient(
            "localhost:2181",
            new ExponentialBackoffRetry(1000, 3)
        );
        client.start();
    }

    public void doWithLock(String lockPath) {
        // 可重入锁（临时顺序节点实现）
        InterProcessMutex lock = new InterProcessMutex(client, lockPath);

        try {
            // 获取锁，最多等待10秒
            if (lock.acquire(10, TimeUnit.SECONDS)) {
                try {
                    // 执行业务逻辑...
                    System.out.println("获取锁成功，执行业务");
                } finally {
                    lock.release();
                }
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
ZooKeeper锁的优点
  • 公平锁：按创建顺序获取锁
  • 避免惊群效应：只监听前一个节点
  • 天然防止死锁：临时节点随会话删除
  • 可重入：InterProcessMutex支持重入

分布式锁注意事项

1. 可重入性

同一线程多次获取同一把锁时应该成功，需要记录线程标识和重入次数

2. 阻塞/非阻塞

  • 阻塞式：获取不到锁时等待，直到获取成功或超时
  • 非阻塞式：获取不到立即返回失败，由业务决定重试策略

3. 锁超时设置

超时时间设置原则
  • 不宜过短：业务未完成锁就释放，失去互斥意义
  • 不宜过长：服务宕机后锁长时间不释放，影响可用性
  • 建议：根据业务最大执行时间 + 缓冲时间，配合看门狗机制

4. 锁粒度控制

细粒度锁示例
不要对所有订单使用同一个锁：lock:order
应该对单个订单加锁：lock:order:{orderId}

选型建议

场景 推荐方案 理由
一般业务场景 Redisson 简单易用，性能好，社区活跃
高可靠性要求 RedLock 或 ZooKeeper 避免Redis主从切换丢锁
需要公平锁 ZooKeeper 天然支持公平锁
简单场景 数据库唯一索引 无需引入新组件
