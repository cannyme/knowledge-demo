代理模式

为目标对象提供一个代理，控制对目标对象的访问。Spring AOP、MyBatis Mapper、RPC 远程调用均是代理模式的核心应用。理解代理是理解 AOP 的关键。

📖

三种代理类型

类型实现方式适用场景性能
静态代理手动编写代理类代理类固定，接口不多最高
JDK 动态代理Proxy.newProxyInstance + InvocationHandler目标类有接口中
CGLIB 代理字节码增强，继承目标类目标类无接口（需非 final）中
💻

代码实现

静态代理 — 手动编写，代理类和目标类实现同一接口

Java
interface UserService {
    void addUser(String name);
    void deleteUser(String name);
}

class UserServiceImpl implements UserService {
    @Override public void addUser(String name) { System.out.println("添加用户：" + name); }
    @Override public void deleteUser(String name) { System.out.println("删除用户：" + name); }
}

// 静态代理：手动包装，每个方法都要写
class UserServiceProxy implements UserService {
    private final UserService target;
    public UserServiceProxy(UserService target) { this.target = target; }

    @Override
    public void addUser(String name) {
        System.out.println("[日志] 调用 addUser 前");
        long start = System.currentTimeMillis();
        target.addUser(name);
        System.out.println("[日志] 耗时：" + (System.currentTimeMillis() - start) + "ms");
    }

    @Override
    public void deleteUser(String name) {
        System.out.println("[日志] 调用 deleteUser 前");
        target.deleteUser(name);
    }
}

// 使用
UserService proxy = new UserServiceProxy(new UserServiceImpl());
proxy.addUser("张三");
⚠️ 静态代理缺点：每新增接口方法，代理类都要同步修改；N 个接口需要 N 个代理类。

JDK 动态代理 — 运行时生成代理，目标类必须实现接口 Spring 默认（有接口时）

Java
// InvocationHandler：统一处理所有方法调用
class LoggingInvocationHandler implements InvocationHandler {
    private final Object target;

    public LoggingInvocationHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("[JDK代理] 方法调用前：" + method.getName());
        long start = System.currentTimeMillis();
        try {
            Object result = method.invoke(target, args); // 反射调用真实方法
            System.out.println("[JDK代理] 调用成功，耗时：" + (System.currentTimeMillis()-start) + "ms");
            return result;
        } catch (InvocationTargetException e) {
            System.out.println("[JDK代理] 异常：" + e.getCause().getMessage());
            throw e.getCause();
        }
    }

    // 工具方法：创建代理对象
    @SuppressWarnings("unchecked")
    public static <T> T createProxy(T target) {
        return (T) Proxy.newProxyInstance(
            target.getClass().getClassLoader(), // ClassLoader
            target.getClass().getInterfaces(),  // 代理哪些接口
            new LoggingInvocationHandler(target) // 调用处理器
        );
    }
}

// 使用
UserService proxy = LoggingInvocationHandler.createProxy(new UserServiceImpl());
proxy.addUser("李四");    // 自动记录日志
proxy.deleteUser("王五"); // 无需逐个方法编写

Proxy.newProxyInstance 三个参数：

  • ClassLoader：用哪个类加载器加载代理类，通常用目标类的 ClassLoader
  • Class[] interfaces：代理类要实现哪些接口（必须是接口）
  • InvocationHandler：所有方法调用都路由到 invoke() 方法

CGLIB 代理 — 继承目标类，无需接口，Spring Boot 2.x 默认 Spring 默认

Java
// 目标类（无需接口）
class OrderService {
    public void createOrder(String orderId) {
        System.out.println("创建订单：" + orderId);
    }

    public final void finalMethod() { // final 方法无法被 CGLIB 代理！
        System.out.println("final 方法不可代理");
    }
}

// CGLIB 方法拦截器（类比 InvocationHandler）
class LoggingMethodInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, Method method, Object[] args,
                            MethodProxy proxy) throws Throwable {
        System.out.println("[CGLIB] 方法前：" + method.getName());
        Object result = proxy.invokeSuper(obj, args); // 调用父类方法（非反射）
        System.out.println("[CGLIB] 方法后：" + method.getName());
        return result;
    }
}

// 创建 CGLIB 代理
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(OrderService.class); // 继承目标类
enhancer.setCallback(new LoggingMethodInterceptor()); // 设置拦截器
OrderService proxy = (OrderService) enhancer.create(); // 生成代理子类

proxy.createOrder("ORD-001"); // 被拦截
// proxy.finalMethod();       // final 方法不被拦截，直接调用
⚠️ CGLIB 限制：不能代理 final 类和 final 方法（无法继承/覆盖）。

保护代理 — 控制访问权限，常用于权限系统

Java
interface Document {
    void read();
    void edit(String content);
    void delete();
}

class RealDocument implements Document {
    private final String name;
    public RealDocument(String name) { this.name = name; }
    @Override public void read() { System.out.println("读取：" + name); }
    @Override public void edit(String c) { System.out.println("编辑：" + name + " → " + c); }
    @Override public void delete() { System.out.println("删除：" + name); }
}

// 保护代理：根据角色控制访问权限
class DocumentProtectionProxy implements Document {
    private final RealDocument doc;
    private final String role; // "guest" | "editor" | "admin"

    public DocumentProtectionProxy(RealDocument doc, String role) {
        this.doc = doc; this.role = role;
    }

    @Override public void read() { doc.read(); } // 所有人可读

    @Override
    public void edit(String content) {
        if ("guest".equals(role)) { System.out.println("权限不足：无法编辑"); return; }
        doc.edit(content);
    }

    @Override
    public void delete() {
        if (!"admin".equals(role)) { System.out.println("权限不足：仅管理员可删除"); return; }
        doc.delete();
    }
}

// 测试
Document guestDoc = new DocumentProtectionProxy(new RealDocument("机密.pdf"), "guest");
guestDoc.read();          // ✅ 成功
guestDoc.edit("内容");    // ❌ 权限不足
guestDoc.delete();        // ❌ 权限不足

Spring AOP 实战应用

Java - @Around 切面（日志+性能监控）
@Aspect
@Component
public class LoggingAspect {

    // 切点：拦截 service 包下所有方法
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceLayer() {}

    // 环绕通知：最强大，可控制方法是否执行
    @Around("serviceLayer()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();

        log.info("[AOP] 调用 {} 参数：{}", methodName, Arrays.toString(args));
        long start = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed(); // 执行真实方法
            log.info("[AOP] {} 成功，耗时：{}ms", methodName, System.currentTimeMillis()-start);
            return result;
        } catch (Exception e) {
            log.error("[AOP] {} 异常：{}", methodName, e.getMessage());
            throw e;
        }
    }

    // 前置通知
    @Before("@annotation(com.example.annotation.RequireLogin)")
    public void checkLogin(JoinPoint jp) {
        // 检查登录状态
        if (!SecurityContext.isLoggedIn()) {
            throw new UnauthorizedException("请先登录");
        }
    }
}

面试高频考点

Q1：JDK 动态代理 vs CGLIB 的区别？Spring 默认用哪个？
维度JDK 动态代理CGLIB
实现原理实现接口，反射调用继承目标类，字节码增强
要求目标类必须有接口目标类不能是 final，方法不能是 final
性能JDK8+ 后差异不大方法调用更快（MethodProxy 非反射）
Spring 默认目标有接口时（Spring Boot 2.0前）Spring Boot 2.0+ 默认全用 CGLIB
Spring Boot 2.x 默认强制使用 CGLIB（proxyTargetClass=true），避免接口代理的一些问题。
Q2：Spring 事务在同类方法调用中为什么会失效？（AOP 代理失效经典题）
Java
@Service
public class OrderService {
    // ❌ 事务失效：methodA 通过 this 调用 methodB，绕过了代理对象
    public void methodA() {
        this.methodB(); // this 是真实对象，不是代理，@Transactional 不生效
    }

    @Transactional
    public void methodB() {
        // 事务注解在此，但通过 this 调用时代理不生效
        orderRepository.save(order);
    }
}

// ✅ 解决方案1：注入自身代理
@Service
public class OrderService {
    @Autowired
    private OrderService self; // 注入代理对象（非 this）

    public void methodA() {
        self.methodB(); // 通过代理对象调用，事务生效
    }

    @Transactional
    public void methodB() { ... }
}

// ✅ 解决方案2：通过 AopContext 获取代理
public void methodA() {
    ((OrderService) AopContext.currentProxy()).methodB();
}
根本原因：Spring AOP 基于代理，只有通过代理对象调用方法，增强逻辑（事务、日志等）才会执行。通过 this 调用绕过了代理。
Q3：哪些情况会导致 AOP 代理失效？
  • 同类方法调用（this 调用）：如上所述
  • private 方法：CGLIB 无法覆盖私有方法，JDK 代理也不支持
  • final 类/方法：CGLIB 无法继承 final 类，无法覆盖 final 方法
  • static 方法：属于类而非实例，代理无法拦截
  • 构造器调用：对象创建时代理还未介入

MyBatis Mapper 接口代理原理

MyBatis 最神奇的地方：只需定义接口，无需实现类，就能执行 SQL。这正是 JDK 动态代理的典型应用。

Java - MyBatis Mapper 代理核心原理
// 用户定义的 Mapper 接口（无实现类）
public interface UserMapper {
    @Select("SELECT * FROM user WHERE id = #{id}")
    User findById(Long id);

    @Insert("INSERT INTO user(name, email) VALUES(#{name}, #{email})")
    int insert(User user);
}

// MyBatis 内部：MapperProxy 实现了 InvocationHandler
// （简化版，展示核心原理）
public class MapperProxy implements InvocationHandler {
    private final SqlSession sqlSession;
    private final Class mapperInterface;

    public MapperProxy(SqlSession sqlSession, Class mapperInterface) {
        this.sqlSession = sqlSession;
        this.mapperInterface = mapperInterface;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 将 Method 转为 SQL 语句 ID（接口全限定名.方法名）
        String statementId = mapperInterface.getName() + "." + method.getName();

        // 根据注解类型决定操作（SELECT/INSERT/UPDATE/DELETE）
        if (method.isAnnotationPresent(Select.class)) {
            return sqlSession.selectOne(statementId, args[0]);
        } else if (method.isAnnotationPresent(Insert.class)) {
            return sqlSession.insert(statementId, args[0]);
        }
        // ...
        return null;
    }
}

// MyBatis 创建 Mapper 代理的方式（MapperProxyFactory）
public class MapperProxyFactory {
    private final Class mapperInterface;

    @SuppressWarnings("unchecked")
    public T newInstance(SqlSession sqlSession) {
        MapperProxy mapperProxy = new MapperProxy<>(sqlSession, mapperInterface);
        // JDK 动态代理：生成实现了 UserMapper 接口的代理对象
        return (T) Proxy.newProxyInstance(
            mapperInterface.getClassLoader(),
            new Class[]{mapperInterface},
            mapperProxy
        );
    }
}

// 使用（Spring 中被自动处理）
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
// mapper 实际是代理对象，调用 findById 时走 MapperProxy.invoke()
User user = mapper.findById(1L);
⚠️

实战陷阱

陷阱1：@Transactional 注解在 private 方法上不生效
Spring AOP 代理基于继承（CGLIB）或接口（JDK），均无法代理 private 方法。@Transactional 加在 private 方法上完全无效，不会报错但也不会开启事务。
陷阱2：同类内方法调用代理失效（经典高频题）
Java
@Service
public class OrderService {
    // ❌ 通过 this 调用，不走代理，@Transactional 失效
    public void placeOrder() {
        this.saveOrder(); // this 是真实对象，非代理
    }

    @Transactional
    public void saveOrder() {
        // 事务注解在这里，但因为通过 this 调用而失效
    }
}

// ✅ 解决方案：注入自身代理
@Service
public class OrderService {
    @Lazy @Autowired
    private OrderService self; // 注入代理对象

    public void placeOrder() {
        self.saveOrder(); // 通过代理调用，事务生效
    }

    @Transactional
    public void saveOrder() { ... }
}
最佳实践：代理的透明性
好的代理应该对客户端完全透明。如果客户端需要感知代理的存在（如调用代理特有方法），说明设计有问题，应重新考虑是否用装饰器更合适。

与其他模式的关系

相关模式关系说明
装饰器模式代理控制对对象的访问（通常透明）；装饰器动态添加功能（客户端主动组装）
适配器模式适配器改变接口；代理保持接口不变，只是控制访问
外观模式外观为复杂子系统提供简化接口；代理与目标对象接口相同，1:1 对应
策略模式代理可以在转发请求前后执行策略（如权限检查、缓存策略）
