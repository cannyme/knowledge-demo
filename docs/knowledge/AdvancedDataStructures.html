跳表（SkipList）

跳表是一种基于链表的多层索引结构，通过随机化的方式构建多层索引，实现近似O(log n)的查找效率。Redis的有序集合（ZSet）就是基于跳表实现的。

结构示意

graph LR subgraph "Level 3" A1[HEAD] --> A2[50] end subgraph "Level 2" B1[HEAD] --> B2[20] --> B3[50] end subgraph "Level 1" C1[HEAD] --> C2[10] --> C3[20] --> C4[30] --> C5[50] end subgraph "Level 0" D1[HEAD] --> D2[5] --> D3[10] --> D4[15] --> D5[20] --> D6[25] --> D7[30] --> D8[40] --> D9[50] --> D10[NIL] end

跳表 vs 平衡树

特性跳表平衡树（AVL/红黑树）
实现难度简单复杂
范围查询高效（底层链表）需要中序遍历
并发友好是（局部锁）否（需要全局锁）
内存占用较高（多层索引）较低
💡 Redis使用跳表原因：实现简单、范围查询高效、并发性能好。

简化版跳表实现

Java
public class SkipList {
    private static final int MAX_LEVEL = 16;
    private final Node head = new Node(-1, MAX_LEVEL);
    private int level = 0;
    private final Random random = new Random();

    static class Node {
        int val;
        Node[] next;
        Node(int val, int level) {
            this.val = val;
            this.next = new Node[level];
        }
    }

    public boolean search(int target) {
        Node curr = head;
        for (int i = level - 1; i >= 0; i--) {
            while (curr.next[i] != null && curr.next[i].val < target) {
                curr = curr.next[i];
            }
        }
        curr = curr.next[0];
        return curr != null && curr.val == target;
    }

    public void add(int val) {
        Node[] update = new Node[MAX_LEVEL];
        Node curr = head;
        for (int i = level - 1; i >= 0; i--) {
            while (curr.next[i] != null && curr.next[i].val < val) {
                curr = curr.next[i];
            }
            update[i] = curr;
        }
        int lvl = randomLevel();
        level = Math.max(level, lvl);
        Node newNode = new Node(val, lvl);
        for (int i = 0; i < lvl; i++) {
            newNode.next[i] = update[i].next[i];
            update[i].next[i] = newNode;
        }
    }

    private int randomLevel() {
        int lvl = 1;
        while (random.nextDouble() < 0.5 && lvl < MAX_LEVEL) {
            lvl++;
        }
        return lvl;
    }
}

布隆过滤器（Bloom Filter）

布隆过滤器是一种空间效率极高的概率型数据结构，用于判断一个元素是否可能存在于集合中。特点是可能存在误报（false positive），但绝不会漏报（false negative）。

原理

graph LR A[元素X] -->|哈希1| B[位数组] A -->|哈希2| B A -->|哈希3| B B -->|判断| C{可能存在?} C -->|是| D[可能存在于集合] C -->|否| E[一定不存在]

误判率公式

误判率 p ≈ (1 - e^(-kn/m))^k

  • n：元素数量
  • m：位数组大小
  • k：哈希函数个数

使用场景

  • 缓存穿透防护：判断请求的数据是否可能存在，避免查询不存在的key
  • URL去重：爬虫系统判断URL是否已抓取
  • 垃圾邮件过滤：判断邮件是否可能是垃圾邮件
Java
// Guava实现布隆过滤器
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1000000,    // 预期插入数量
    0.01        // 期望误判率
);

// 添加元素
bloomFilter.put("user:1001");
bloomFilter.put("user:1002");

// 判断元素可能存在（可能有误判）
if (bloomFilter.mightContain("user:1001")) {
    // 可能存在，需要进一步查询数据库确认
}

if (!bloomFilter.mightContain("user:9999")) {
    // 一定不存在，直接返回
}

LSM树（Log-Structured Merge Tree）

LSM树是一种针对写密集型应用优化的数据结构，通过将随机写转换为顺序写来提高写入性能。广泛应用于LevelDB、RocksDB、HBase等存储系统中。

写入流程

flowchart LR A[写入请求] -->|追加| B[WAL日志] A -->|插入| C[MemTable] C -->|达到阈值| D[Immutable MemTable] D -->|刷盘| E[SSTable L0] E -->|合并| F[SSTable L1] F -->|合并| G[SSTable L2]

核心组件

组件说明
WAL预写日志，保证数据持久性
MemTable内存中的有序数据结构（跳表/红黑树）
SSTable磁盘上的有序字符串表
Compaction合并压缩，清理过期数据

LSM树 vs B+树

特性LSM树B+树
写入性能极高（顺序写）一般（随机写）
读取性能一般（需查询多层）高（直接定位）
空间放大较大（需合并）较小
适用场景写多读少读写均衡

B+树

B+树是B树的变种，是MySQL等数据库最常用的索引结构。所有数据存储在叶子节点，非叶子节点只存储键值作为索引。

结构示意

graph TB A[30 / 60] --> B[10 / 20 / 30] A --> C[30 / 40 / 50 / 60] A --> D[60 / 70 / 80] B --> B1[d1] B --> B2[d2] B --> B3[d3] C --> C1[d4] C --> C2[d5] C --> C3[d6] C --> C4[d7] D --> D1[d8] D --> D2[d9] D --> D3[d10] B1 --- B2 --- B3 --- C1 --- C2 --- C3 --- C4 --- D1 --- D2 --- D3

B+树特点

  • 非叶子节点只存储键值，不存储数据
  • 所有数据存储在叶子节点
  • 叶子节点通过指针连接成链表，支持范围查询
  • 树的高度低（通常3-4层），减少磁盘IO

为什么MySQL使用B+树

  • 范围查询高效：叶子节点链表可直接遍历
  • 单节点存储更多键值：非叶子节点不存数据，树更矮
  • 减少磁盘IO：3层B+树可存储千万级数据
💡 InnoDB索引高度计算：假设页大小16KB，主键8字节，指针6字节。每个页可存储约1170个键值。3层B+树可存储约 1170 * 1170 * 16 ≈ 2000万 条记录。

红黑树

红黑树是一种自平衡二叉搜索树，通过五条规则保证树的平衡，确保最坏情况下查找、插入、删除的时间复杂度都是O(log n)。

红黑树五条规则

  1. 每个节点要么是红色，要么是黑色
  2. 根节点是黑色
  3. 每个叶子节点（NIL）是黑色
  4. 红色节点的子节点必须是黑色（不能有两个连续红节点）
  5. 从任一节点到其每个叶子节点的所有路径都包含相同数目的黑色节点

应用场景

应用说明
Java TreeMap/TreeSet基于红黑树实现的有序Map和Set
Linux CFS调度器完全公平调度器使用红黑树管理进程
EpollLinux事件通知机制使用红黑树管理文件描述符
⚠️ 注意：红黑树的插入和删除操作需要维护平衡，实现复杂。Java中直接使用TreeMap/TreeSet即可，无需自己实现。

一致性哈希

一致性哈希是一种特殊的哈希算法，在分布式系统中用于解决数据分片和数据迁移问题。当节点数量变化时，只需重新分配少量数据。

普通哈希 vs 一致性哈希

graph TB subgraph "普通哈希（节点变化影响大）" A1[key] -->|hash%3| B1[Node0] A1 -->|节点增加| C1[hash%4] C1 --> D1[数据几乎全部重新分配] end subgraph "一致性哈希（节点变化影响小）" A2[key] -->|顺时针找| B2[Node Ring] B2 --> C2[Node1] B2 --> D2[Node2] B2 --> E2[Node3] B2 -->|新增Node4| F2[仅相邻节点数据迁移] end

虚拟节点

为解决数据倾斜问题，引入虚拟节点。每个物理节点对应多个虚拟节点，均匀分布在哈希环上。

graph LR subgraph "哈希环" A[Node1-V1] --> B[Node2-V1] B --> C[Node1-V2] C --> D[Node3-V1] D --> E[Node2-V2] E --> F[Node1-V3] F --> A end

应用场景

  • 分布式缓存：Redis Cluster、Memcached的数据分片
  • 分布式存储：Cassandra、DynamoDB的数据分布
  • 负载均衡：Nginx的一致性哈希负载均衡
