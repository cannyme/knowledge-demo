装饰器模式

动态地给对象添加额外职责，比继承更灵活。Java IO 流、Spring HttpServletRequestWrapper、MyBatis 缓存装饰器均是其经典应用。

📖

模式结构

classDiagram class Component { <<interface>> +operation() } class ConcreteComponent { +operation() } class Decorator { #component: Component; +operation() } class ConcreteDecoratorA { -addedState; +operation() } class ConcreteDecoratorB { +operation(); +addedBehavior() } Component <|.. ConcreteComponent Component <|.. Decorator Decorator o--> Component Decorator <|-- ConcreteDecoratorA Decorator <|-- ConcreteDecoratorB

关键：装饰器与被装饰对象实现同一接口，且持有被装饰对象的引用（组合），形成装饰链。

☕

咖啡配料系统示例

Java
// 组件接口
interface Coffee {
    String getDescription();
    double getCost();
}

// 具体组件：基础咖啡
class SimpleCoffee implements Coffee {
    @Override public String getDescription() { return "黑咖啡"; }
    @Override public double getCost() { return 10.0; }
}

// 抽象装饰器：实现接口 + 持有引用
abstract class CoffeeDecorator implements Coffee {
    protected final Coffee coffee; // 持有被装饰对象
    public CoffeeDecorator(Coffee coffee) { this.coffee = coffee; }
}

// 具体装饰器：牛奶
class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) { super(coffee); }
    @Override public String getDescription() { return coffee.getDescription() + " + 牛奶"; }
    @Override public double getCost() { return coffee.getCost() + 3.0; }
}

// 具体装饰器：糖
class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) { super(coffee); }
    @Override public String getDescription() { return coffee.getDescription() + " + 糖"; }
    @Override public double getCost() { return coffee.getCost() + 1.0; }
}

// 具体装饰器：奶油
class CreamDecorator extends CoffeeDecorator {
    public CreamDecorator(Coffee coffee) { super(coffee); }
    @Override public String getDescription() { return coffee.getDescription() + " + 奶油"; }
    @Override public double getCost() { return coffee.getCost() + 5.0; }
}

// 使用：链式装饰，灵活组合
Coffee coffee = new SimpleCoffee();
System.out.println(coffee.getDescription() + " = " + coffee.getCost()); // 黑咖啡 = 10.0

coffee = new MilkDecorator(coffee);
coffee = new SugarDecorator(coffee);
coffee = new CreamDecorator(coffee);
System.out.println(coffee.getDescription() + " = " + coffee.getCost());
// 黑咖啡 + 牛奶 + 糖 + 奶油 = 19.0

// 链式写法（更简洁）
Coffee fancy = new CreamDecorator(new SugarDecorator(new MilkDecorator(new SimpleCoffee())));

Java IO 装饰器详解

Java IO 是装饰器模式最经典的应用，整个 InputStream/OutputStream 体系都是装饰器链：

classDiagram class InputStream { <<abstract>> +read() int } class FileInputStream { +read() int } class ByteArrayInputStream { +read() int } class FilterInputStream { #in: InputStream; +read() int } class BufferedInputStream { -buf: byte[]; +read() int } class DataInputStream { +readInt() int; +readUTF() String } class GZIPInputStream { +read() int } InputStream <|-- FileInputStream InputStream <|-- ByteArrayInputStream InputStream <|-- FilterInputStream FilterInputStream <|-- BufferedInputStream FilterInputStream <|-- DataInputStream FilterInputStream <|-- GZIPInputStream FilterInputStream o--> InputStream
Java - IO 装饰器使用
// 分层装饰：每层添加不同能力
// FileInputStream：从文件读字节（基础组件）
// GZIPInputStream：解压缩（装饰器）
// BufferedInputStream：缓冲读取（装饰器）
// DataInputStream：读取基本类型（装饰器）

InputStream is = new DataInputStream(
    new BufferedInputStream(        // 第3层：加缓冲，减少系统调用
        new GZIPInputStream(        // 第2层：自动解压
            new FileInputStream("data.gz")  // 第1层：基础读取
        )
    )
);

// 写入示例
OutputStream os = new GZIPOutputStream(
    new BufferedOutputStream(
        new FileOutputStream("output.gz")
    )
);

// 字节流 → 字符流（适配器 + 装饰器）
BufferedReader reader = new BufferedReader(
    new InputStreamReader(           // 适配：字节流→字符流
        new FileInputStream("text.txt"), "UTF-8"
    )
);
String line;
while ((line = reader.readLine()) != null) {
    System.out.println(line);
}

实战：请求体缓存（HttpServletRequestWrapper）

Spring MVC 中 HttpServletRequest 的 InputStream 只能读一次，需要装饰器来实现可重复读：

Java
// 装饰器：缓存请求体，支持多次读取
public class CachedBodyHttpServletRequest extends HttpServletRequestWrapper {
    private final byte[] cachedBody;

    public CachedBodyHttpServletRequest(HttpServletRequest request) throws IOException {
        super(request);
        // 一次性读取请求体并缓存
        this.cachedBody = StreamUtils.copyToByteArray(request.getInputStream());
    }

    @Override
    public ServletInputStream getInputStream() {
        // 每次调用都返回新的流（基于缓存的字节数组）
        return new CachedBodyServletInputStream(this.cachedBody);
    }

    @Override
    public BufferedReader getReader() {
        return new BufferedReader(new InputStreamReader(
            new ByteArrayInputStream(this.cachedBody), StandardCharsets.UTF_8));
    }

    public byte[] getCachedBody() { return this.cachedBody; }
}

// 在过滤器中使用（用装饰器包装原始请求）
@Component
public class RequestBodyCachingFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws Exception {
        // 用装饰器包装，下游可多次读取 body
        CachedBodyHttpServletRequest cachedRequest =
            new CachedBodyHttpServletRequest(request);
        filterChain.doFilter(cachedRequest, response);
    }
}

面试高频考点

Q1：装饰器模式 vs 继承的区别？
  • 继承（静态）：编译时确定，每种组合都需要一个子类，N 个功能组合 = 2^N 个类（类爆炸）
  • 装饰器（动态）：运行时灵活组合，N 个装饰器可产生任意组合，不需要额外类
  • 原则："组合优于继承"，装饰器是组合的典型体现
Q2：装饰器 vs 代理模式的区别？（高频！）
维度装饰器模式代理模式
意图增强功能，添加新职责控制访问，不暴露真实对象
透明性客户端知道在装饰（通常）客户端无感知代理存在
对象来源装饰器从外部接收组件代理通常自己创建目标对象
典型用途Java IO、咖啡配料AOP、权限控制、缓存、日志
Q3：Spring 中哪里用到了装饰器？
  • HttpServletRequestWrapper / HttpServletResponseWrapper：装饰请求/响应，添加缓存、参数过滤等功能
  • BeanWrapper：对 Bean 进行包装，提供属性访问、类型转换能力
  • TransactionAwareCacheDecorator：MyBatis 一级缓存的事务感知装饰器
  • Collections.synchronizedList()：JDK 对 List 的线程安全装饰

MyBatis 缓存装饰器链

MyBatis 二级缓存是装饰器模式的教科书级实现，多个装饰器层层包裹，每层添加不同能力：

Java - MyBatis Cache 装饰器
// Cache 接口（组件接口）
public interface Cache {
    String getId();
    void putObject(Object key, Object value);
    Object getObject(Object key);
    Object removeObject(Object key);
    void clear();
    int getSize();
}

// 基础实现（被装饰的核心组件）
public class PerpetualCache implements Cache {
    private final String id;
    private final Map cache = new HashMap<>();
    // ... 简单 HashMap 存储
}

// SynchronizedCache 装饰器：添加线程安全
public class SynchronizedCache implements Cache {
    private final Cache delegate; // 持有被装饰的 Cache
    public SynchronizedCache(Cache delegate) { this.delegate = delegate; }

    @Override
    public synchronized void putObject(Object key, Object value) {
        delegate.putObject(key, value); // 加锁后委托
    }
    @Override
    public synchronized Object getObject(Object key) {
        return delegate.getObject(key);
    }
}

// LruCache 装饰器：添加 LRU 淘汰策略
public class LruCache implements Cache {
    private final Cache delegate;
    private Map keyMap; // LinkedHashMap 实现 LRU

    public LruCache(Cache delegate) {
        this.delegate = delegate;
        setSize(1024); // 默认容量
    }
    // ...
}

// MyBatis 构建缓存链的方式（CacheBuilder）
Cache cache = new SynchronizedCache(
    new LoggingCache(
        new SerializedCache(
            new ScheduledCache(
                new LruCache(
                    new PerpetualCache("myCache")
                )
            )
        )
    )
);

实战陷阱

陷阱1：装饰器链中忘记调用 super/delegate
Java
// ❌ 错误：忘记调用被装饰对象，导致原有功能丢失
class BrokenDecorator extends CoffeeDecorator {
    @Override
    public double getCost() {
        return 3.0; // 忘记 coffee.getCost()，原始价格丢失！
    }
}

// ✅ 正确：调用被装饰对象后叠加
class CorrectDecorator extends CoffeeDecorator {
    @Override
    public double getCost() {
        return coffee.getCost() + 3.0; // 叠加原价
    }
}
陷阱2：装饰器与继承混用导致类型检查失败
装饰器改变了对象的"外壳"但保留了接口。如果代码用 instanceof 检查具体类型（如 obj instanceof ConcreteComponent），装饰后的对象会失败。应面向接口编程，避免 instanceof 具体类。
最佳实践：用 Collections 工具类的装饰器
JDK 内置了多种装饰器，直接使用即可：Collections.unmodifiableList(list)、Collections.synchronizedMap(map)、Collections.checkedList(list, String.class)。

与其他模式的关系

相关模式关系说明
代理模式结构相似，但意图不同：装饰器增强功能（客户端通常自己组装）；代理控制访问（通常透明于客户端）
适配器模式适配器改变接口；装饰器保持接口不变，仅添加功能
组合模式装饰器只有一个组件，组合有多个；两者都实现相同接口，可结合使用
责任链模式责任链中每个处理者可以终止链；装饰器通常都会委托给下一层
