线程池详解

线程池是Java并发编程的核心组件，通过复用线程降低资源消耗、提高响应速度、便于统一管理。掌握线程池原理与最佳实践是高级Java工程师必备技能。

🎯

为什么要用线程池

在并发编程中，频繁创建和销毁线程会带来显著的性能开销。线程池通过池化技术解决这一问题：

优点说明
降低资源消耗重复利用已创建的线程，减少线程创建和销毁的开销
提高响应速度任务到达时无需等待线程创建，可立即执行
便于管理统一分配、调优和监控线程，防止无限制创建导致系统崩溃
提高稳定性通过拒绝策略和队列机制控制并发量，保护系统稳定性
不使用线程池的风险
无限制创建线程会导致系统资源耗尽，引发OutOfMemoryError，甚至导致服务不可用。每个线程约占用1MB栈空间，10000个线程就需要约10GB内存。
⚙️

线程池核心参数（7个）

ThreadPoolExecutor的完整构造函数包含7个参数，理解这些参数是配置线程池的基础：

参数类型说明
corePoolSizeint核心线程数，常驻线程，即使空闲也保留（除非设置allowCoreThreadTimeOut）
maximumPoolSizeint最大线程数，线程池允许创建的最大线程数量
keepAliveTimelong非核心线程的空闲存活时间
unitTimeUnit时间单位（秒、毫秒等）
workQueueBlockingQueue任务等待队列，存储待执行的任务
threadFactoryThreadFactory线程工厂，用于创建新线程，可设置线程名、守护状态等
handlerRejectedExecutionHandler拒绝策略，当线程数和队列都满时的处理策略
Java
public ThreadPoolExecutor(
    int corePoolSize,                          // 核心线程数
    int maximumPoolSize,                       // 最大线程数
    long keepAliveTime,                        // 非核心线程存活时间
    TimeUnit unit,                             // 时间单位
    BlockingQueue<Runnable> workQueue,         // 任务队列
    ThreadFactory threadFactory,               // 线程工厂
    RejectedExecutionHandler handler           // 拒绝策略
) { ... }
🔄

任务执行流程

当提交一个新任务时，线程池按以下流程处理：

  1. 判断核心线程数：如果当前运行线程数少于corePoolSize，创建新核心线程执行任务
  2. 加入任务队列：如果核心线程已满，将任务加入workQueue等待执行
  3. 创建非核心线程：如果队列已满且线程数小于maximumPoolSize，创建非核心线程执行任务
  4. 执行拒绝策略：如果线程数达到maximumPoolSize且队列已满，触发拒绝策略

执行任务] B -->|否| D{任务队列
未满?} D -->|是| E[任务加入队列
等待执行] D -->|否| F{线程数 < maximumPoolSize?} F -->|是| G[创建非核心线程
执行任务] F -->|否| H[执行拒绝策略]

四种常用线程池

Executors工厂方法提供了四种常用线程池，但生产环境建议直接使用ThreadPoolExecutor手动配置：

1. FixedThreadPool（固定线程数）

核心线程数 = 最大线程数，使用无界队列LinkedBlockingQueue

Java
ExecutorService fixedPool = Executors.newFixedThreadPool(10);
// 等价于：
new ThreadPoolExecutor(
    10, 10, 0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>()  // 无界队列！
);
风险提示
队列无界，任务堆积时可能导致OOM。不推荐直接使用Executors创建。

2. CachedThreadPool（可缓存线程池）

核心线程数为0，最大线程数无限制，空闲线程存活60秒，使用SynchronousQueue

Java
ExecutorService cachedPool = Executors.newCachedThreadPool();
// 等价于：
new ThreadPoolExecutor(
    0, Integer.MAX_VALUE,  // 最大线程数无限制！
    60L, TimeUnit.SECONDS,
    new SynchronousQueue<>()  // 直接交接，不排队
);
风险提示
线程数无限制，高并发时可能创建大量线程导致OOM。

3. SingleThreadExecutor（单线程线程池）

只有一个工作线程，任务按提交顺序执行，使用无界队列

Java
ExecutorService singlePool = Executors.newSingleThreadExecutor();
// 等价于：
new ThreadPoolExecutor(
    1, 1, 0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>()  // 无界队列
);

4. ScheduledThreadPool（定时任务线程池）

支持定时执行、周期执行任务，基于DelayedWorkQueue

Java
ScheduledExecutorService scheduledPool = Executors.newScheduledThreadPool(10);
// 延迟执行
scheduledPool.schedule(task, 5, TimeUnit.SECONDS);
// 固定频率执行
scheduledPool.scheduleAtFixedRate(task, 0, 10, TimeUnit.SECONDS);
// 固定延迟执行
scheduledPool.scheduleWithFixedDelay(task, 0, 10, TimeUnit.SECONDS);
🚫

拒绝策略

当线程数达到maximumPoolSize且任务队列已满时，新任务会触发拒绝策略。ThreadPoolExecutor提供了4种内置策略：

策略类名行为适用场景
中止策略AbortPolicy默认策略，抛出RejectedExecutionException需要感知任务被拒绝的场景
调用者运行策略CallerRunsPolicy由提交任务的线程（调用者）执行任务希望降低任务提交速度，减轻负载
丢弃策略DiscardPolicy静默丢弃任务，不抛异常允许任务丢失的场景
丢弃最老策略DiscardOldestPolicy丢弃队列中最老任务，尝试重新提交优先处理新任务的场景
Java
// 自定义拒绝策略示例
static class CustomRejectedHandler implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 记录日志
        log.warn("Task rejected: {}", r);
        // 存入数据库重试
        retryTaskDao.save(r);
        // 或让调用者执行
        if (!executor.isShutdown()) {
            r.run();
        }
    }
}

线程池状态

ThreadPoolExecutor使用一个AtomicInteger存储线程池状态和线程数量（高3位表示状态，低29位表示线程数）：

状态值说明
RUNNING-1 << COUNT_BITS接收新任务，处理队列任务
SHUTDOWN0 << COUNT_BITS不接收新任务，但处理队列中剩余任务
STOP1 << COUNT_BITS不接收新任务，不处理队列任务，中断正在执行的任务
TIDYING2 << COUNT_BITS所有任务已终止，线程数为0，准备执行terminated()
TERMINATED3 << COUNT_BITSterminated()执行完成
Java
// 状态转换方法
executor.shutdown();      // RUNNING -> SHUTDOWN
executor.shutdownNow();   // RUNNING -> STOP

// 状态检查
executor.isShutdown();    // 是否调用过shutdown/shutdownNow
executor.isTerminated();  // 是否所有任务已完成
📦

任务队列类型

队列选择对线程池行为有重要影响：

队列类型特点注意事项
SynchronousQueue不存储元素，直接交接，可能创建大量线程CachedThreadPool使用
LinkedBlockingQueue无界链表队列，可能导致OOMFixedThreadPool使用，建议指定容量
ArrayBlockingQueue有界数组队列，需指定容量生产环境推荐
PriorityBlockingQueue优先级队列，按优先级取任务需实现Comparable
DelayQueue延迟队列，元素到期才能取出ScheduledThreadPool使用

最佳实践

1. 线程池配置建议

任务类型核心线程数最大线程数队列说明
CPU密集型CPU核心数 + 1核心数 + 1有界队列减少上下文切换
IO密集型CPU核心数 * 2核心数 * 2~4有界队列充分利用IO等待时间
混合型CPU核心数 * (1 + 等待/计算比)适当扩大有界队列根据实际占比调整
公式参考
线程数 = CPU核心数 * (1 + 等待时间/计算时间)。例如IO等待时间是计算时间的2倍，则线程数 = 核心数 * 3

2. 线程池监控

Java
static void monitorThreadPool(ThreadPoolExecutor executor) {
    System.out.println("核心线程数：" + executor.getCorePoolSize());
    System.out.println("最大线程数：" + executor.getMaximumPoolSize());
    System.out.println("当前线程数：" + executor.getPoolSize());
    System.out.println("活跃线程数：" + executor.getActiveCount());
    System.out.println("已完成任务：" + executor.getCompletedTaskCount());
    System.out.println("队列大小：" + executor.getQueue().size());
    System.out.println("队列剩余容量：" + executor.getQueue().remainingCapacity());
}

3. 优雅关闭线程池

Java
static void shutdownGracefully(ExecutorService executor) {
    // 1. 停止接收新任务
    executor.shutdown();
    try {
        // 2. 等待已提交任务完成（最多60秒）
        if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
            // 3. 超时后强制关闭
            executor.shutdownNow();
            // 4. 再次等待
            if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
                System.err.println("线程池未完全关闭");
            }
        }
    } catch (InterruptedException e) {
        // 5. 当前线程被中断，强制关闭
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}

4. 生产环境推荐配置

Java
// CPU密集型线程池示例
static ExecutorService cpuIntensivePool() {
    int coreSize = Runtime.getRuntime().availableProcessors() + 1;
    return new ThreadPoolExecutor(
        coreSize, coreSize, 0L, TimeUnit.MILLISECONDS,
        new LinkedBlockingQueue<>(1000),  // 有界队列防OOM
        new ThreadFactoryBuilder().setNameFormat("cpu-pool-%d").build(),
        new ThreadPoolExecutor.CallerRunsPolicy()  // 调用者执行，降低提交速度
    );
}

// IO密集型线程池示例
static ExecutorService ioIntensivePool() {
    int coreSize = Runtime.getRuntime().availableProcessors() * 2;
    return new ThreadPoolExecutor(
        coreSize, coreSize * 2, 60L, TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(500),
        new ThreadFactoryBuilder().setNameFormat("io-pool-%d").build(),
        new ThreadPoolExecutor.CallerRunsPolicy()
    );
}

线程池配置检查清单

  • ☐ 使用有界队列（ArrayBlockingQueue或指定容量的LinkedBlockingQueue）
  • ☐ 根据任务类型设置合理的核心线程数和最大线程数
  • ☐ 自定义ThreadFactory设置有意义的线程名称，方便问题排查
  • ☐ 选择合适的拒绝策略，CallerRunsPolicy通常是较好的选择
  • ☐ 避免使用Executors的便捷方法创建线程池
  • ☐ 实现线程池监控，关注活跃线程数、队列大小、完成任务数
  • ☐ 确保应用关闭时优雅关闭线程池，避免任务丢失
  • ☐ 对关键业务线程池设置合理的keepAliveTime
Spring Boot推荐
在Spring Boot中可使用ThreadPoolTaskExecutor，它提供了更友好的配置方式，并支持优雅关闭。
