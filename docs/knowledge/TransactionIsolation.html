并发问题定义

问题说明
脏读（Dirty Read）读取到其他事务未提交的数据
不可重复读（Non-repeatable Read）同一事务内两次读取同一数据结果不同
幻读（Phantom Read）同一事务内两次查询，后一次查询到了前一次没有的行

隔离级别详解

读未提交（READ UNCOMMITTED）

最低的隔离级别，事务可以读取到其他事务未提交的数据。

⚠️ 脏读示例
时刻    事务A                        事务B
─────────────────────────────────────────────────────
T1     BEGIN
T2     UPDATE user SET name='李四' WHERE id=1
       (未提交)
T3                                  BEGIN
T4                                  SELECT name FROM user WHERE id=1
                                    → 读到'李四'（脏读！）
T5     ROLLBACK
       (数据回滚为'张三')

问题：事务B读到了未提交的数据，这些数据可能被回滚，导致数据不一致。

适用场景：极少使用，仅用于对数据一致性要求极低的场景。

读已提交（READ COMMITTED）

事务只能读取到其他事务已提交的数据，解决了脏读问题。

⚠️ 不可重复读示例
时刻    事务A                        事务B
─────────────────────────────────────────────────────
T1                                  BEGIN
T2                                  SELECT name FROM user WHERE id=1
                                    → 读到'张三'
T3     BEGIN
T4     UPDATE user SET name='李四' WHERE id=1
T5     COMMIT
T6                                  SELECT name FROM user WHERE id=1
                                    → 读到'李四'（不可重复读！）
T7                                  COMMIT

问题：同一事务内两次读取结果不同，破坏了一致性。

适用场景：Oracle、SQL Server默认级别，适合大多数业务场景。

可重复读（REPEATABLE READ）

MySQL默认隔离级别，通过MVCC机制保证同一事务内多次读取结果一致。

💡 MVCC机制

Multi-Version Concurrency Control（多版本并发控制），通过保存数据的历史版本，让读操作不加锁也能保证一致性。

⚠️ 幻读示例（当前读场景）
时刻    事务A                        事务B
─────────────────────────────────────────────────────
T1     BEGIN
T2     SELECT * FROM user WHERE id>5
       → 2条记录
T3                                  BEGIN
T4                                  INSERT INTO user VALUES(6,'test')
T5                                  COMMIT
T6     SELECT * FROM user WHERE id>5
       → 2条记录（MVCC保证一致性读）
T7     SELECT * FROM user WHERE id>5 FOR UPDATE
       → 3条记录（当前读，幻读！）
T8     UPDATE user SET name='x' WHERE id=6
       → 更新成功（幻读效果）

解决：使用SELECT ... FOR UPDATE加Next-Key Lock防止幻读

串行化（SERIALIZABLE）

最高隔离级别，所有事务串行执行，完全避免并发问题。

实现方式：
  • 读操作加共享锁（S锁）
  • 写操作加排他锁（X锁）
  • 锁的范围扩大到扫描的所有行
⚠️ 性能问题

SERIALIZABLE性能最差，会导致大量锁等待，实际应用中基本不使用。

按粒度分类

锁类型说明命令示例
全局锁锁定整个数据库实例FLUSH TABLES WITH READ LOCK
表级锁锁定整张表LOCK TABLES user READ/WRITE
行级锁锁定单行数据（InnoDB）SELECT ... FOR UPDATE

InnoDB行锁详解

锁类型说明示例
Record Lock记录锁，锁定单条索引记录WHERE id = 1 FOR UPDATE
Gap Lock间隙锁，锁定索引记录之间的间隙WHERE id > 5 AND id < 10
Next-Key Lock临键锁 = Record Lock + Gap LockRR级别默认，防止幻读
SQL
-- 唯一索引等值查询，记录存在
SELECT * FROM user WHERE id = 1 FOR UPDATE;
→ Record Lock on id=1

-- 唯一索引等值查询，记录不存在
SELECT * FROM user WHERE id = 3 FOR UPDATE;  -- 假设id=3不存在
→ Gap Lock on (1, 5)  -- 锁定相邻记录之间的间隙

-- 唯一索引范围查询
SELECT * FROM user WHERE id >= 5 FOR UPDATE;
→ Record Lock on id=5
→ Next-Key Lock on (5, 10]
→ Gap Lock on (10, +∞)

-- 非唯一索引等值查询
-- 假设有普通索引 idx_age，数据：age=20,20,30,30
SELECT * FROM user WHERE age = 20 FOR UPDATE;
→ Next-Key Lock on (-∞, 20]
→ Gap Lock on (20, 30)  -- 锁定到下一个不同值

死锁产生与避免

死锁产生的四个条件

  • 互斥条件：资源一次只能被一个事务占用
  • 请求与保持条件：事务持有资源同时请求新资源
  • 不可剥夺条件：已获得的资源不能被强制释放
  • 循环等待条件：多个事务形成循环等待链
⚠️ 死锁示例
时刻    事务A                        事务B
─────────────────────────────────────────────────────
T1     BEGIN
T2     UPDATE user SET name='A' WHERE id=1
       (持有id=1的锁)
T3                                  BEGIN
T4                                  UPDATE user SET name='B' WHERE id=2
                                     (持有id=2的锁)
T5     UPDATE user SET name='A2' WHERE id=2
       (等待id=2的锁)  ←┐
T6                                  UPDATE user SET name='B2' WHERE id=1
                                     (等待id=1的锁)  ←┘
       → 死锁！双方都持有对方需要的锁

死锁检测与处理

SQL
-- 开启死锁检测（默认ON）
SET GLOBAL innodb_deadlock_detect = ON;

-- 查看最近的死锁信息
SHOW ENGINE INNODB STATUS;

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 查看当前锁
SELECT * FROM performance_schema.data_locks;

-- 锁等待超时配置（默认50秒）
SET GLOBAL innodb_lock_wait_timeout = 50;

避免死锁的建议

💡 最佳实践
  • 固定顺序：按固定顺序访问表和行，避免循环等待
  • 缩短事务：大事务拆小事务，减少锁持有时间
  • 合理索引：避免无索引导致的全表扫描锁
  • 降级隔离：适当降低隔离级别，减少锁竞争
  • 重试机制：捕获死锁异常后重试

隔离级别选择建议

场景建议隔离级别说明
一般OLTP业务READ COMMITTEDOracle默认，避免脏读，性能好
需要一致性读REPEATABLE READMySQL默认，MVCC保证
报表/统计REPEATABLE READ避免统计过程中数据变化
高安全性要求SERIALIZABLE完全串行，性能牺牲
锁优化建议：
  • 合理设计索引，减少锁范围（无索引会锁全表）
  • 避免长事务，及时提交或回滚
  • 尽量使用主键更新，精确锁定
  • 避免在事务中执行耗时操作（RPC调用、复杂计算）
  • SELECT尽量不加锁，除非必要（FOR UPDATE）
