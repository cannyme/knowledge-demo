熔断与限流

熔断与限流的详细解析和最佳实践。熔断器模式当下游服务故障时快速失败，避免级联崩溃；限流控制请求速率，保护系统不被压垮。

❄️

为什么需要熔断

雪崩效应（Cascading Failure）是分布式系统中常见的问题：当某个服务出现故障时，大量请求会阻塞等待，导致线程池耗尽，进而影响上游服务，最终引发整个系统的崩溃。

如上图所示，当第三方支付服务响应缓慢或不可用时：

  • 支付服务线程阻塞等待响应
  • 订单服务线程池被占满，无法处理新请求
  • API网关堆积大量请求，内存耗尽
  • 最终导致整个系统不可用
💡
熔断器的作用 熔断器模式借鉴了电路熔断器的概念，当"电流"（错误率）过大时自动"断开"电路，阻止故障扩散，给故障服务恢复的时间窗口。
🔀

熔断器三种状态

熔断器有三种核心状态，对应不同的请求处理策略：

CLOSED 关闭状态
正常服务状态
请求正常通过，熔断器持续监控错误率。当失败率超过阈值时，切换到 OPEN 状态。
OPEN 打开状态
熔断保护状态
所有请求被快速拒绝（直接返回降级逻辑），不再调用下游服务。经过超时时间后进入 HALF_OPEN 状态。
HALF_OPEN 半开状态
探测恢复状态
放行少量请求试探下游服务是否恢复。如果成功则关闭熔断器，失败则重新打开。
🏗️

状态转换流程

stateDiagram-v2 [*] --> CLOSED: 初始化 CLOSED --> CLOSED: 请求正常通过 CLOSED --> OPEN: 失败率 > 阈值 OPEN --> OPEN: 快速拒绝请求 OPEN --> HALF_OPEN: 超时时间到达 HALF_OPEN --> CLOSED: 探测成功 HALF_OPEN --> OPEN: 探测失败
转换路径 触发条件 行为说明
CLOSED → OPEN 失败率超过阈值（如50%） 记录失败次数，达到阈值后熔断
OPEN → HALF_OPEN 熔断持续时间到达（如10秒） 允许有限请求通过进行探测
HALF_OPEN → CLOSED 连续成功次数达标 服务恢复，熔断器关闭
HALF_OPEN → OPEN 探测请求失败 服务未恢复，重新熔断
⚙️

熔断器核心参数

配置熔断器时需要理解以下5个核心参数：

failureRateThreshold
失败率阈值（默认50%）
当失败率达到此阈值时触发熔断。建议根据业务容忍度设置，关键服务可设为30%-50%。
waitDurationInOpenState
熔断持续时间（默认60秒）
熔断器打开后等待的时间，期间所有请求被拒绝。时间到达后进入半开状态。
slidingWindowSize
滑动窗口大小（默认100）
统计失败率的请求窗口大小。可以是请求数量（COUNT_BASED）或时间（TIME_BASED）。
minimumNumberOfCalls
最小调用次数（默认10）
触发熔断计算的最小请求数。防止启动初期少量失败就触发熔断。
permittedNumberOfCallsInHalfOpenState
半开状态允许请求数（默认10）
HALF_OPEN状态下允许通过的请求数量，用于探测服务是否恢复。
⚠️
参数配置建议 滑动窗口不宜过大（内存占用），也不宜过小（统计不准）。建议生产环境：窗口100-1000，失败率阈值50%，熔断时间10-30秒。

Sentinel熔断规则配置

使用 @SentinelResource 注解声明资源和降级方法：

java
@RestController
public class UserController {

    @SentinelResource(
        value = "getUser",
        blockHandler = "handleBlock",
        fallback = "handleFallback"
    )
    @GetMapping("/user/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.getById(id);
    }

    // 限流/熔断处理（BlockException）
    public User handleBlock(Long id, BlockException ex) {
        return new User(-1L, "服务被熔断", null);
    }

    // 异常降级处理（业务异常）
    public User handleFallback(Long id, Throwable t) {
        return new User(-1L, "服务异常", null);
    }
}

通过代码动态配置熔断规则：

java
// 熔断规则配置
DegradeRule rule = new DegradeRule("getUser")
    // 熔断策略：0-慢调用比例 1-异常比例 2-异常数
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
    // 阈值：慢调用比例阈值（0.0-1.0）
    .setCount(0.5)
    // 熔断持续时间（秒）
    .setTimeWindow(10)
    // 最小请求数
    .setMinRequestAmount(10)
    // 慢调用阈值（RT，毫秒）
    .setSlowRatioThreshold(500)
    // 统计时长（毫秒）
    .setStatIntervalMs(1000);

DegradeRuleManager.loadRules(Collections.singletonList(rule));

Sentinel熔断策略 慢调用比例：响应时间超过阈值的请求占比超过设定值时熔断，适用于响应变慢的场景。
异常比例：异常请求占比超过阈值时熔断，适用于服务出错的场景。
异常数：异常请求数超过阈值时熔断，适用于异常突增的场景。

Hystrix熔断配置

⚠️
重要提示 Netflix Hystrix 已于2018年底停止维护，Spring Cloud 2020+ 版本已移除 Hystrix 依赖。新项目建议使用 Resilience4j 或 Sentinel。

Hystrix 通过命令模式包装远程调用：

java
@Service
public class UserService {

    @HystrixCommand(
        commandKey = "getUser",
        fallbackMethod = "getUserFallback",
        commandProperties = {
            // 熔断触发错误率阈值（默认50）
            @HystrixProperty(name="circuitBreaker.errorThresholdPercentage", value="50"),
            // 熔断持续时间（毫秒，默认5000）
            @HystrixProperty(name="circuitBreaker.sleepWindowInMilliseconds", value="10000"),
            // 触发熔断的最小请求数（默认20）
            @HystrixProperty(name="circuitBreaker.requestVolumeThreshold", value="10"),
            // 是否启用熔断（默认true）
            @HystrixProperty(name="circuitBreaker.enabled", value="true")
        },
        threadPoolProperties = {
            // 线程池大小
            @HystrixProperty(name="coreSize", value="10"),
            // 队列大小
            @HystrixProperty(name="maxQueueSize", value="100")
        }
    )
    public User getUser(Long id) {
        return restTemplate.getForObject(
            "http://user-service/user/{id}",
            User.class,
            id
        );
    }

    public User getUserFallback(Long id) {
        return new User(-1L, "服务降级", null);
    }
}
配置项 默认值 说明
errorThresholdPercentage 50 错误率阈值，达到后触发熔断
sleepWindowInMilliseconds 5000 熔断后休眠时间（毫秒）
requestVolumeThreshold 20 触发熔断的最小请求数
executionTimeoutInMilliseconds 1000 执行超时时间

熔断 vs 降级 vs 限流

这三者是微服务高可用的核心手段，但解决的问题不同：

维度 熔断（Circuit Breaker） 降级（Degradation） 限流（Rate Limiting）
触发条件 下游服务故障/超时，错误率超标 系统负载过高或依赖故障 请求速率超过系统容量
作用位置 调用方（客户端） 服务端或调用方 服务端（入口层）
行为表现 快速失败，阻断故障扩散 提供备用逻辑（简化/缓存数据） 拒绝或排队处理请求
恢复机制 自动探测恢复（半开状态） 手动或自动恢复 请求减少后自动恢复
典型框架 Hystrix、Resilience4j、Sentinel Sentinel、本地缓存 Guava RateLimiter、Sentinel、Nginx
类比 电路保险丝（自动跳闸） 飞机放燃油减重（保命操作） 高速收费站（控制流速）

限流算法详解

限流算法控制请求速率，防止系统被突发流量压垮：

算法 优点 缺点 适用场景
固定窗口 实现简单，内存占用低 存在临界问题（窗口边界突发） 简单限流场景
滑动窗口 精确统计，无临界问题 内存占用较高 精确限流场景
漏桶算法 流量平滑，保护下游 无法应对突发流量 保护下游服务
令牌桶算法 允许突发，平均速率控制 实现稍复杂 API限流（推荐）
滑动日志 最精确的统计 内存占用最高 分布式限流
java
/**
 * 令牌桶算法实现 - 允许一定程度的突发流量
 */
public class TokenBucketRateLimiter {
    private final int capacity;      // 桶容量
    private final int rate;          // 令牌生成速率（个/秒）
    private final AtomicInteger tokens;
    private volatile long lastRefillTime;

    public synchronized boolean tryAcquire(int permits) {
        // 补充令牌
        long now = System.currentTimeMillis();
        int newTokens = (int) ((now - lastRefillTime) * rate / 1000);
        if (newTokens > 0) {
            tokens.set(Math.min(capacity, tokens.get() + newTokens));
            lastRefillTime = now;
        }
        // 尝试获取令牌
        if (tokens.get() >= permits) {
            tokens.addAndGet(-permits);
            return true;
        }
        return false;
    }
}

最佳实践

1. 合理设置阈值
失败率阈值建议50%-70%，避免过于敏感导致频繁熔断。最小请求数建议10-20，防止启动时误触发。
2. 配合降级策略
熔断后必须提供降级逻辑，如返回缓存数据、默认值或友好提示，避免直接抛出异常。
3. 监控与告警
接入Micrometer + Prometheus监控熔断器状态，熔断触发时及时告警，关注半开状态频率。
4. 区分异常类型
业务异常（如参数错误）不应计入熔断统计，只统计超时、连接失败等系统异常。
5. 资源隔离
不同服务使用独立的熔断器实例，避免相互影响。Hystrix 的线程池隔离或 Sentinel 的线程隔离。
6. 限流前置
限流应在网关层或入口层统一处理，熔断在服务调用层，形成双层防护。
💡
生产环境配置示例 失败率阈值: 50% | 熔断时间: 10-30秒 | 滑动窗口: 100 | 最小请求数: 10 | 半开请求数: 5
根据业务特点调整，核心服务可以更保守（阈值30%，熔断时间5秒）。
⚠️
常见陷阱 1. 降级逻辑过于复杂：降级方法应简单快速，避免再次依赖故障服务。
2. 忽略慢调用熔断：响应变慢往往是故障前兆，应配置慢调用比例熔断。
3. 全局异常捕获干扰：确保熔断器能正确捕获异常，不被全局异常处理器拦截。

Resilience4j 完整配置实战

yaml
resilience4j:
  circuitbreaker:
    instances:
      orderService:
        # 滑动窗口大小（基于次数）
        sliding-window-size: 10
        # 失败率阈值（超过则熔断）
        failure-rate-threshold: 50
        # 慢调用率阈值（超过也熔断）
        slow-call-rate-threshold: 80
        # 慢调用时间阈值（ms）
        slow-call-duration-threshold: 2000
        # OPEN 状态持续时间（ms）后进入 HALF-OPEN
        wait-duration-in-open-state: 10000
        # HALF-OPEN 时允许通过的请求数
        permitted-number-of-calls-in-half-open-state: 3
        # 进入熔断前至少需要的请求数
        minimum-number-of-calls: 5

  retry:
    instances:
      orderService:
        max-attempts: 3
        wait-duration: 500ms
        retry-exceptions:
          - java.net.ConnectException
          - java.net.SocketTimeoutException

  bulkhead:
    instances:
      orderService:
        max-concurrent-calls: 10   # 最大并发
        max-wait-duration: 100ms   # 等待时间
java
// 编程式使用 Resilience4j（更灵活）
@Service
public class OrderService {

    private final CircuitBreaker circuitBreaker;
    private final Retry retry;

    public OrderService(CircuitBreakerRegistry cbRegistry,
                        RetryRegistry retryRegistry) {
        this.circuitBreaker = cbRegistry.circuitBreaker("orderService");
        this.retry = retryRegistry.retry("orderService");

        // 监听熔断器状态变化
        circuitBreaker.getEventPublisher()
            .onStateTransition(event ->
                log.warn("熔断器状态变更: {} -> {}",
                    event.getStateTransition().getFromState(),
                    event.getStateTransition().getToState()));
    }

    public OrderDTO getOrder(Long id) {
        // 组合 Retry + CircuitBreaker（先重试再熔断）
        Supplier<OrderDTO> decorated = Decorators
            .ofSupplier(() -> remoteCall(id))
            .withRetry(retry)
            .withCircuitBreaker(circuitBreaker)
            .withFallback(List.of(CallNotPermittedException.class,
                                  ConnectException.class),
                          e -> OrderDTO.empty())
            .decorate();

        return Try.ofSupplier(decorated).getOrElse(OrderDTO.empty());
    }
}

面试高频考点

Q1: 熔断器的三种状态和转换规则？
CLOSED（关闭）：正常状态，请求正常通过，统计失败率；当失败率超过阈值（如 50%）且请求数达到最小值后转 OPEN。OPEN（开启/熔断）：拒绝所有请求，立即返回降级结果；经过等待窗口（如 10s）后转 HALF-OPEN。HALF-OPEN（半开）：放行有限请求（如 3 个）探测下游是否恢复；若成功率达标转回 CLOSED，否则再次转 OPEN。
Q2: 熔断和降级的区别？
熔断（Circuit Breaker）：当下游服务故障率过高时，主动切断调用链路，防止故障扩散（级联失败）。是一种保护机制，保护调用方和下游服务。降级（Fallback）：当调用失败（超时、熔断、异常）时，返回预设的兜底结果（空对象、缓存数据、默认值），保证主流程可用。熔断会触发降级，但降级不一定需要熔断（也可在超时/异常时直接降级）。
Q3: Hystrix 和 Resilience4j 的主要区别？
Hystrix：Netflix 开源，已停止维护（2018）；线程池隔离模式为主，通过线程切换实现异步超时；功能集中在熔断和降级。Resilience4j：轻量级（无依赖），函数式设计；支持 CircuitBreaker、RateLimiter、Retry、Bulkhead、TimeLimiter 模块化组合；信号量隔离模式，更适合响应式编程；与 Spring Boot Actuator 深度集成，监控更友好。新项目应选 Resilience4j。
Q4: 舱壁模式（Bulkhead）是什么？有什么作用？
Bulkhead（舱壁/隔离）来源于船舱设计：将船舱分为多个隔断，一处进水不影响整体。在微服务中，通过线程池隔离（每个服务用独立线程池）或信号量隔离（限制并发数），防止某个下游慢接口耗尽所有线程，保护服务整体可用性。Resilience4j 的 Bulkhead 通过 max-concurrent-calls 限制并发，SemaphoreBulkhead 适合非阻塞，ThreadPoolBulkhead 适合阻塞操作。

实战陷阱

⚠️
陷阱1：熔断阈值设置过低导致误熔断 minimum-number-of-calls 设置过小（如 2），偶发的 1 次失败即触发 50% 失败率达到熔断条件，流量正常时也会频繁熔断。应根据实际 QPS 合理设置滑动窗口和最小请求数，生产建议 minimum-number-of-calls ≥ 10，failure-rate-threshold ≥ 50%。
⚠️
陷阱2：Fallback 中继续调用失败的下游 降级方法中若继续调用同一个失败的远程服务（如"查询失败就再查一次备用接口"），在熔断器 OPEN 时 fallback 中的调用也会立即失败，导致降级本身也失败。Fallback 应直接返回本地缓存、默认值或静态数据，不要发起新的远程调用。
⚠️
陷阱3：注解式 @CircuitBreaker 与 @Retry 顺序问题 当同时使用 @Retry 和 @CircuitBreaker 注解时，执行顺序为 Retry 包裹 CircuitBreaker（即先重试，重试失败才触发熔断计数）。若希望重试不影响熔断计数，应将 CircuitBreaker 包裹 Retry（用编程式 API 组合），确保整体失败才计入熔断窗口。
