数据库连接池详解

ConnectionPool.java

通过复用数据库连接，减少连接创建和销毁的开销，提高系统性能和稳定性。了解连接池原理、配置参数和最佳实践。

为什么要用连接池

数据库连接的创建开销

每次创建数据库连接都需要经过以下步骤：

  1. TCP三次握手：建立与数据库服务器的网络连接（约几十毫秒）
  2. 身份认证：验证用户名密码（约几毫秒）
  3. 初始化资源：分配内存、建立会话等
sequenceDiagram participant App as 应用程序 participant Pool as 连接池 participant DB as 数据库 Note over App,DB: 传统方式（每次新建连接） App->>DB: 创建连接（TCP+认证） DB-->>App: 连接建立 App->>DB: 执行SQL DB-->>App: 返回结果 App->>DB: 关闭连接 Note over App,DB: 使用连接池 Pool->>DB: 预创建连接 DB-->>Pool: 连接就绪 App->>Pool: 获取连接 Pool-->>App: 返回已有连接 App->>DB: 执行SQL DB-->>App: 返回结果 App->>Pool: 归还连接

连接池的优势

优势 说明
性能提升 复用已有连接，避免频繁创建/销毁的开销
资源控制 限制最大连接数，防止数据库连接耗尽
连接管理 自动检测失效连接、超时回收等
监控统计 提供连接使用情况的监控数据
异常处理 连接泄漏检测、自动重连等

连接池核心参数

基本配置参数

参数 说明 建议值
initialSize 初始连接数，应用启动时创建的连接数 5-10
minIdle 最小空闲连接数，连接池保持的最小空闲连接 与max相同或略小
maxActive/maxPoolSize 最大活跃连接数，连接池能创建的最大连接数 10-20（根据负载）
maxWait/connectionTimeout 获取连接的最大等待时间 30秒
idleTimeout 空闲连接超时时间，超过此时间被回收 10分钟
maxLifetime 连接最大存活时间，超过强制关闭 30分钟

验证与检测参数

参数 说明
validationQuery 连接验证SQL，如"SELECT 1"
testOnBorrow 获取连接时验证，建议false（影响性能）
testOnReturn 归还连接时验证，建议false
testWhileIdle 空闲时验证，建议true
timeBetweenEvictionRuns 空闲检测线程运行间隔
leakDetectionThreshold 连接泄漏检测阈值，超过时间未归还打印警告

连接数计算公式：

connections = (CPU核心数 × 2) + 有效磁盘数

例如：4核CPU + 1块SSD = 4×2+1 = 9个连接

注意：这是单机理想值，实际应根据并发量、SQL复杂度调整。

HikariCP配置与原理

为什么HikariCP最快？

1. 字节码优化

使用Javassist在运行时生成代理类，避免反射带来的性能开销。

2. ConcurrentBag设计

┌─────────────────────────────────────────────────────────────┐ │ ConcurrentBag │ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ │ │ sharedList │ │ threadList │ │ waiters │ │ │ │ (共享列表) │ │ (线程本地) │ │ (等待队列) │ │ │ │ CAS操作 │ │ 无锁获取 │ │ 阻塞等待 │ │ │ └─────────────┘ └─────────────┘ └─────────────┘ │ └─────────────────────────────────────────────────────────────┘

3. 快速获取连接流程

  1. 先从ThreadLocal获取（无竞争，最快）
  2. 未命中则从sharedList获取（CAS竞争）
  3. 都没有则等待其他连接归还（阻塞等待）

HikariCP配置示例

Spring Boot + HikariCP配置 
# application.yml
spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/mydb?useSSL=false&serverTimezone=UTC
    username: root
    password: 123456
    hikari:
      # 连接池名称
      pool-name: MyHikariPool

      # 连接池大小配置
      minimum-idle: 10           # 最小空闲连接数
      maximum-pool-size: 20      # 最大连接数

      # 连接超时配置
      connection-timeout: 30000  # 获取连接超时（毫秒）
      idle-timeout: 600000       # 空闲连接超时（毫秒）
      max-lifetime: 1800000      # 连接最大存活时间（毫秒）

      # 连接验证配置
      validation-timeout: 5000   # 连接验证超时
      connection-test-query: SELECT 1  # 连接测试SQL

      # 泄漏检测（开发环境建议开启）
      leak-detection-threshold: 60000  # 60秒未归还打印警告

HikariCP最佳配置建议：

  • minimum-idle 建议与 maximum-pool-size 相同，避免连接频繁创建销毁
  • max-lifetime 应小于数据库的 wait_timeout
  • leak-detection-threshold 只在开发环境开启，生产环境关闭

Druid配置与监控功能

Druid的特点

  • 监控功能强大：内置监控页面，实时查看SQL执行情况
  • SQL防注入：内置WallFilter防止SQL注入攻击
  • SQL执行日志：记录慢SQL、错误SQL等
  • 扩展性好：支持多种Filter扩展

Druid配置示例

Spring Boot + Druid配置 
# application.yml
spring:
  datasource:
    type: com.alibaba.druid.pool.DruidDataSource
    druid:
      # 基本配置
      driver-class-name: com.mysql.cj.jdbc.Driver
      url: jdbc:mysql://localhost:3306/mydb
      username: root
      password: 123456

      # 连接池大小配置
      initial-size: 5       # 初始连接数
      min-idle: 5           # 最小空闲连接
      max-active: 20        # 最大活跃连接
      max-wait: 60000       # 最大等待时间

      # 连接检测配置
      time-between-eviction-runs-millis: 60000   # 检测间隔
      min-evictable-idle-time-millis: 300000      # 最小空闲时间
      validation-query: SELECT 1
      test-while-idle: true
      test-on-borrow: false
      test-on-return: false

      # 预编译语句缓存
      pool-prepared-statements: true
      max-pool-prepared-statement-per-connection-size: 20

      # 监控配置
      stat-view-servlet:
        enabled: true
        url-pattern: /druid/*
        login-username: admin
        login-password: admin
      web-stat-filter:
        enabled: true
        url-pattern: /*

      # 防火墙配置（SQL防注入）
      filters: wall,stat,slf4j
      connection-properties: druid.stat.mergeSql=true;druid.stat.slowSqlMillis=5000

连接池最佳实践

1. 连接数配置

连接数配置建议 
// 计算公式：connections = (CPU核心数 * 2) + 有效磁盘数
// 例如：4核 + 1块SSD = 4*2+1 = 9个连接

// 实际配置应考虑：
// 1. 应用的并发请求数
// 2. 单个请求的执行时间
// 3. 数据库的承载能力

// 高并发场景配置示例：
maximum-pool-size: 50     # 峰值并发量
minimum-idle: 20          # 常规并发量

// 低并发场景配置示例：
maximum-pool-size: 10
minimum-idle: 5

2. 超时配置

参数 建议值 说明
connection-timeout 30秒 根据业务响应时间调整
idle-timeout 10分钟 空闲连接回收时间
max-lifetime 30分钟 必须小于数据库wait_timeout

3. 连接验证配置

推荐配置：

  • test-while-idle = true：空闲时检测连接有效性
  • test-on-borrow = false：获取时不检测（影响性能）
  • test-on-return = false：归还时不检测
  • validation-query = SELECT 1：简单验证SQL

4. 正确的连接使用方式

try-with-resources方式 
// ✅ 正确：使用try-with-resources自动关闭
public void correctUsage() {
    try (Connection conn = dataSource.getConnection();
         PreparedStatement ps = conn.prepareStatement("SELECT * FROM user");
         ResultSet rs = ps.executeQuery()) {

        while (rs.next()) {
            // 处理结果
        }
    } catch (SQLException e) {
        e.printStackTrace();
    }
    // 连接自动归还到连接池
}

// ✅ 正确：Spring事务管理
@Transactional
public void withTransaction() {
    // Spring自动管理连接获取和释放
    userRepository.findById(1L);
}

连接池常见问题

1. 连接泄漏（Connection Leak）

获取连接后未正确归还，导致连接池耗尽。

常见原因

  • 忘记调用close()
  • 异常导致close()未执行
  • 长时间持有连接不释放

解决方案

连接泄漏检测配置 
# HikariCP连接泄漏检测
hikari:
  leak-detection-threshold: 60000  # 60秒未归还打印警告日志

# 日志输出示例：
# APPARENT DEADLOCK: Creating emergency threads for unassigned pending tasks!
# Connection leak detection triggered for connection xxxx

2. 连接池满（Pool Exhausted）

所有连接都被占用，新请求无法获取连接。

排查步骤

  1. 检查是否有连接泄漏
  2. 检查活跃连接数是否持续高位
  3. 检查SQL执行时间是否过长
  4. 检查是否配置了合理的超时时间
监控SQL排查问题 
-- 查看当前连接数
SHOW STATUS LIKE 'Threads_connected';

-- 查看正在执行的SQL
SHOW FULL PROCESSLIST;

-- 查看执行时间长的SQL
SELECT * FROM information_schema.processlist
WHERE command != 'Sleep'
ORDER BY time DESC;

3. 连接数设置不合理

连接数过多：

  • 数据库压力过大
  • 内存消耗增加
  • 上下文切换开销增大

连接数过少：

  • 并发请求等待
  • 应用响应变慢
  • 超时异常增多

4. 常见问题速查表

问题 可能原因 解决方案
获取连接超时 连接池满、连接泄漏 增大连接池、排查泄漏、优化SQL
连接失效异常 连接被数据库关闭 配置test-while-idle、调整max-lifetime
应用启动慢 initialSize过大 减小初始连接数
内存占用高 连接数过多、缓存过大 减少连接数、关闭不必要的缓存
