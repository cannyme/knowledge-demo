双亲委派模型

双亲委派模型是Java类加载的核心机制，确保类加载的安全性和唯一性。

工作流程

flowchart TD Start[收到类加载请求] --> Check{是否已加载?} Check -->|是| Return[返回已加载的类] Check -->|否| Delegate[委派给父加载器] Delegate --> ParentCheck{父加载器能否加载?} ParentCheck -->|能| ParentLoad[父加载器加载] ParentCheck -->|不能| SelfLoad[自己加载] ParentLoad --> Return SelfLoad --> Return

双亲委派的好处

  • 安全性：避免核心类被篡改（用户无法自定义java.lang.String）
  • 避免重复加载：父加载器加载过的类，子加载器不会再加载
Java
// 双亲委派模型核心代码示意
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 检查是否已加载
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        // 2. 委派给父加载器
        if (parent != null) {
            c = parent.loadClass(name, false);
        } else {
            c = findBootstrapClassOrNull(name);
        }
        // 3. 父加载器无法加载，自己加载
        if (c == null) {
            c = findClass(name);
        }
    }
    return c;
}
💡
打破双亲委派的情况：
  • SPI机制（JDBC、JNDI等）：Bootstrap加载器需要加载用户类路径下的实现类
  • OSGi模块化：实现模块间独立的类加载
  • Tomcat类加载：Web应用隔离、热部署需求

初始化时机

主动引用（会触发初始化）

  • new 关键字创建对象
  • 访问静态变量（非final或编译期不确定的final）
  • 调用静态方法
  • 使用反射 Class.forName()
  • 初始化子类时先初始化父类
  • 虚拟机启动时的主类

自定义类加载器

通过继承ClassLoader并重写findClass()方法，可以实现自定义的类加载逻辑。

Java
static class MyClassLoader extends ClassLoader {
    private final String classPath;

    public MyClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // 读取class文件
            byte[] data = loadClassData(name);
            // 将字节码转为Class对象
            return defineClass(name, data, 0, data.length);
        } catch (Exception e) {
            throw new ClassNotFoundException(name);
        }
    }

    private byte[] loadClassData(String name) {
        // 将类名转为文件路径
        String path = classPath + "/" + name.replace('.', '/') + ".class";
        // 读取文件内容
        return Files.readAllBytes(Paths.get(path));
    }
}
💡
应用场景：热部署、代码加密保护、从网络加载类、OSGi模块化等。

WebAppClassLoader 优先自己加载（打破双亲委派），只有加载失败才委派给父加载器，实现不同 Web 应用使用不同版本第三方库互不干扰。

Tomcat 热部署原理

Java
// 热部署核心：销毁旧类加载器，创建新类加载器
// 1. 监测到 class 文件变更
// 2. 销毁旧的 WebAppClassLoader（及其加载的所有类）
// 3. 创建新的 WebAppClassLoader
// 4. 重新加载 Web 应用

// 注意：类加载器本身是 GC Root，只有没有任何引用指向它才能被回收
// 如果静态变量持有线程、连接等资源，可能导致内存泄漏（Permgen/Metaspace OOM）

class HotDeployExample {
    // ❌ 危险：静态变量持有线程，热部署时线程不会销毁
    private static Thread backgroundThread = new Thread(() -> {
        while (true) { /* 不断运行 */ }
    });

    // ✅ 正确：实现 Servlet 的 destroy() 方法，在 Web 应用卸载时清理资源
    public void destroy() {
        backgroundThread.interrupt();
    }
}

面试高频考点

Q1: 双亲委派模型的工作流程？有哪些情况打破了它？

流程：收到类加载请求 → 先检查是否已加载 → 委派给父加载器 → 父加载器不能加载才自己加载。

打破双亲委派的三种情况：

  • JDK 1.2 之前：用户重写 loadClass() 实现自定义逻辑（现在推荐重写 findClass()）
  • SPI 机制（JDBC、JNDI）：Bootstrap 加载的接口需要加载 classpath 下用户实现类，用线程上下文类加载器解决
  • 热部署 / OSGi / Tomcat：应用间类隔离、热替换需求，WebAppClassLoader 优先自己加载
Q2: 类初始化的时机？static 变量和 static 代码块的执行顺序？

触发初始化（主动引用）：new 对象、访问静态变量/方法、反射 Class.forName()、初始化子类先初始化父类、JVM 主类。

不触发初始化（被动引用）：子类引用父类静态变量、数组定义、引用编译期常量（static final）。

Java
class Parent {
    static int A = 1;              // 准备阶段 A=0，初始化阶段 A=1
    static final int B = 2;        // 准备阶段直接 B=2（编译期常量）
    static { System.out.println("Parent static block"); }
}

class Child extends Parent {
    static { System.out.println("Child static block"); }
}

// new Child() 输出：
// Parent static block（先初始化父类）
// Child static block
Q3: 同一个类被两个不同的 ClassLoader 加载，是同一个类吗？

不是。JVM 通过"类加载器 + 全限定类名"共同标识一个类。两个 ClassLoader 分别加载同一个 .class 文件，产生的是两个独立的 Class 对象，互相不兼容。

⚠️
这是 Tomcat 热部署内存泄漏的根本原因：旧类加载器加载的类对象，如果被新类加载器加载的类持有引用，旧类加载器永远无法被 GC 回收。
Q4: 如何实现一个简单的热加载？
Java
// 每次加载都创建新的 ClassLoader，从而强制重新加载类
class HotClassLoader extends ClassLoader {
    private final String classDir;

    public HotClassLoader(String classDir) {
        // 注意：不传 parent，让它自己从文件加载，实现热替换
        super(null);
        this.classDir = classDir;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        String path = classDir + "/" + name.replace('.', '/') + ".class";
        try {
            byte[] bytes = Files.readAllBytes(Paths.get(path));
            return defineClass(name, bytes, 0, bytes.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }
}

// 使用：每次调用时创建新 ClassLoader，实现热加载
public Object createFreshInstance(String className) throws Exception {
    ClassLoader loader = new HotClassLoader("/path/to/classes");
    Class<?> clazz = loader.loadClass(className);
    return clazz.getDeclaredConstructor().newInstance();
}
⚠️

实战陷阱

⚠️
陷阱1：Class.forName() 和 ClassLoader.loadClass() 的区别 Class.forName() 会执行类初始化（运行 static 代码块）；ClassLoader.loadClass() 只加载，不执行初始化。JDBC 注册驱动必须用 Class.forName("com.mysql.jdbc.Driver")，因为驱动注册逻辑在 static 块中。
⚠️
陷阱2：static 变量初始化顺序导致的 NPE static 变量按代码顺序初始化，如果 A 依赖 B 但 B 声明在 A 之后，初始化 A 时 B 还是 null/0，导致 NPE 或错误结果。避免 static 变量之间的循环依赖。
⚠️
陷阱3：Tomcat 热部署后 Metaspace OOM 每次热部署创建新的 WebAppClassLoader，旧的 ClassLoader 若未被回收（ThreadLocal、静态集合持有引用），其加载的所有类对象积压在 Metaspace，最终 OOM。需检查 Listener、Filter 等是否正确清理资源。
