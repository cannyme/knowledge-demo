双亲委派模型

双亲委派模型是Java类加载的核心机制，确保类加载的安全性和唯一性。

工作流程

flowchart TD Start[收到类加载请求] --> Check{是否已加载?} Check -->|是| Return[返回已加载的类] Check -->|否| Delegate[委派给父加载器] Delegate --> ParentCheck{父加载器能否加载?} ParentCheck -->|能| ParentLoad[父加载器加载] ParentCheck -->|不能| SelfLoad[自己加载] ParentLoad --> Return SelfLoad --> Return

双亲委派的好处

  • 安全性：避免核心类被篡改（用户无法自定义java.lang.String）
  • 避免重复加载：父加载器加载过的类，子加载器不会再加载
Java
// 双亲委派模型核心代码示意
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 检查是否已加载
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        // 2. 委派给父加载器
        if (parent != null) {
            c = parent.loadClass(name, false);
        } else {
            c = findBootstrapClassOrNull(name);
        }
        // 3. 父加载器无法加载，自己加载
        if (c == null) {
            c = findClass(name);
        }
    }
    return c;
}
💡 打破双亲委派的情况：
  • SPI机制（JDBC、JNDI等）：Bootstrap加载器需要加载用户类路径下的实现类
  • OSGi模块化：实现模块间独立的类加载
  • Tomcat类加载：Web应用隔离、热部署需求

初始化时机

主动引用（会触发初始化）

  • new 关键字创建对象
  • 访问静态变量（非final或编译期不确定的final）
  • 调用静态方法
  • 使用反射 Class.forName()
  • 初始化子类时先初始化父类
  • 虚拟机启动时的主类

被动引用（不会触发初始化）

场景说明示例
子类引用父类静态变量只初始化父类System.out.println(Child.a)
数组定义类引用不会触发任何类初始化Parent[] arr = new Parent[10]
常量编译期常量进入常量池static final int A = 10

自定义类加载器

通过继承ClassLoader并重写findClass()方法，可以实现自定义的类加载逻辑。

Java
static class MyClassLoader extends ClassLoader {
    private final String classPath;

    public MyClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // 读取class文件
            byte[] data = loadClassData(name);
            // 将字节码转为Class对象
            return defineClass(name, data, 0, data.length);
        } catch (Exception e) {
            throw new ClassNotFoundException(name);
        }
    }

    private byte[] loadClassData(String name) {
        // 将类名转为文件路径
        String path = classPath + "/" + name.replace('.', '/') + ".class";
        // 读取文件内容
        return Files.readAllBytes(Paths.get(path));
    }
}
💡 应用场景：热部署、代码加密保护、从网络加载类、OSGi模块化等。

字符串常量池演进

JDK版本常量池位置特点
JDK 1.6方法区（永久代）intern()复制对象到永久代
JDK 1.7堆内存intern()记录首次出现的引用
JDK 1.8+堆内存方法区改为元空间（本地内存）
Java
public static void stringPoolDemo() {
    // intern示例
    String s3 = new String("b") + new String("c");
    s3.intern();
    String s4 = "bc";
    System.out.println(s3 == s4); // JDK 7+: true
}
⚠️ 注意：频繁使用intern()可能导致常量池溢出（JDK 6），JDK 7+后移至堆中，受最大堆内存限制。
