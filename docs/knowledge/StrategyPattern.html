策略模式

定义一系列算法，将其封装并使之可相互替换。策略模式让算法独立于使用它的客户端变化，是消灭 if-else 的利器，与 Spring 依赖注入结合使用极为优雅。

📖

模式结构

classDiagram class Context { -strategy: Strategy +setStrategy(Strategy) +execute() } class Strategy { <<interface>> +execute() } class ConcreteStrategyA { +execute() } class ConcreteStrategyB { +execute() } class ConcreteStrategyC { +execute() } Context o--> Strategy Strategy <|.. ConcreteStrategyA Strategy <|.. ConcreteStrategyB Strategy <|.. ConcreteStrategyC

三个角色：

  • Strategy（策略接口）：定义算法的公共接口
  • ConcreteStrategy（具体策略）：实现具体算法
  • Context（上下文）：持有策略引用，委托策略执行算法
💻

代码实现

支付策略示例

Java
// 策略接口
interface PaymentStrategy {
    void pay(double amount);
    String getName();
}

// 具体策略：支付宝
class AlipayStrategy implements PaymentStrategy {
    @Override public void pay(double amount) {
        System.out.println("支付宝支付：" + amount + "元");
    }
    @Override public String getName() { return "支付宝"; }
}

// 具体策略：微信
class WechatStrategy implements PaymentStrategy {
    @Override public void pay(double amount) {
        System.out.println("微信支付：" + amount + "元");
    }
    @Override public String getName() { return "微信"; }
}

// 上下文：持有策略，委托执行
class PaymentContext {
    private PaymentStrategy strategy;

    public void setStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;
    }

    public void executePayment(double amount) {
        if (strategy == null) throw new IllegalStateException("未设置支付方式");
        System.out.println("使用" + strategy.getName() + "支付");
        strategy.pay(amount);
    }
}

// 使用：运行时动态切换策略
PaymentContext ctx = new PaymentContext();
ctx.setStrategy(new AlipayStrategy());
ctx.executePayment(100.0);

ctx.setStrategy(new WechatStrategy()); // 策略切换，无需改Context
ctx.executePayment(200.0);

枚举策略 — 避免策略类过多

Java
// 使用枚举实现策略，减少类数量
enum PaymentType implements PaymentStrategy {
    ALIPAY {
        @Override public void pay(double amount) { System.out.println("支付宝：" + amount); }
        @Override public String getName() { return "支付宝"; }
    },
    WECHAT {
        @Override public void pay(double amount) { System.out.println("微信：" + amount); }
        @Override public String getName() { return "微信"; }
    },
    BANK_CARD {
        @Override public void pay(double amount) { System.out.println("银行卡：" + amount); }
        @Override public String getName() { return "银行卡"; }
    };
}

// 使用
PaymentType.ALIPAY.pay(300.0);

Lambda 策略 — 函数式接口简化

Java
// 策略接口是函数式接口时，可直接用 Lambda 替代策略类
@FunctionalInterface
interface SortStrategy {
    void sort(int[] arr);
}

// 无需创建策略类，直接 Lambda
SortStrategy bubbleSort = arr -> {
    for (int i = 0; i < arr.length - 1; i++)
        for (int j = 0; j < arr.length - 1 - i; j++)
            if (arr[j] > arr[j+1]) { int t=arr[j]; arr[j]=arr[j+1]; arr[j+1]=t; }
};

SortStrategy quickSort = arr -> Arrays.sort(arr); // 简单起见

// Map 存储策略
Map<String, SortStrategy> strategies = new HashMap<>();
strategies.put("bubble", bubbleSort);
strategies.put("quick",  quickSort);

// 使用
int[] data = {5, 2, 8, 1};
strategies.get("quick").sort(data);

面试高频考点

Q1：策略模式 vs if-else 的本质区别？
if-else 是过程式思维，把所有逻辑集中在一处，新增分支必须修改代码（违反开闭原则）。策略模式是对象式思维，每种策略封装为独立类，新增策略只需新增类，Context 无需修改。代码从"一大块判断"变成"多个小而内聚的类"。
Q2：策略模式 vs 状态模式的区别？
两者结构几乎相同（都持有接口引用），意图不同：
  • 策略模式：策略由客户端主动选择，算法之间平等可替换，Context 本身无状态变化
  • 状态模式：状态由对象内部自动切换，不同状态下行为不同，状态切换是核心（如订单状态机：待付款→已付款→已发货）
Q3：Spring 中如何优雅地使用策略模式？
Java
// 各策略实现类注册为 Spring Bean，beanName 作为 key
@Component("alipay")
public class AlipayStrategy implements PaymentStrategy { ... }

@Component("wechat")
public class WechatStrategy implements PaymentStrategy { ... }

// Spring 自动将所有 PaymentStrategy Bean 注入 Map（key=beanName）
@Service
public class PaymentService {
    @Autowired
    private Map<String, PaymentStrategy> strategyMap;
    // strategyMap = {"alipay": AlipayStrategy, "wechat": WechatStrategy}

    public void pay(String type, double amount) {
        strategyMap.get(type).pay(amount); // 零 if-else
    }
}
// 新增支付方式：只加一个 @Component 类，Service 代码不动
Q4：开闭原则体现在哪里？
对扩展开放：新增策略只需创建新类实现 Strategy 接口；对修改关闭：Context 类和现有策略类完全不需要修改。满足了"软件实体（类、模块、函数）应对扩展开放，对修改关闭"的原则。
⚖️

优缺点 & 适用场景

✅ 优点

  • 符合开闭原则，易于扩展
  • 消灭 if-else，逻辑清晰
  • 算法可独立测试和复用
  • 可运行时动态切换策略

❌ 缺点

  • 策略类数量可能增多
  • 客户端需了解所有策略差异
  • 策略间不能共享状态

适用场景

  • 支付方式选择（支付宝、微信、银行卡）
  • 打折策略（满减、折扣、会员价、无折扣）
  • 排序算法选择（根据数据规模选快排/归并/插入）
  • 路线规划（步行、驾车、公交）
  • 数据压缩/加密方式选择

实战：电商折扣策略系统

策略模式最经典的实战场景之一：电商平台的多种折扣规则，用策略模式 + Spring 注入彻底消灭 if-else。

Java - 折扣策略完整实现
// Step 1：折扣策略接口
public interface DiscountStrategy {
    /** 计算折扣后价格 */
    BigDecimal calculate(BigDecimal originalPrice, Object context);
    /** 策略名称（用作 Spring beanName 和业务 key）*/
    String getName();
}

// Step 2：具体策略实现

// 无折扣
@Component("NO_DISCOUNT")
public class NoDiscountStrategy implements DiscountStrategy {
    @Override
    public BigDecimal calculate(BigDecimal price, Object ctx) { return price; }
    @Override public String getName() { return "NO_DISCOUNT"; }
}

// 固定折扣（如 8折）
@Component("PERCENTAGE")
public class PercentageDiscountStrategy implements DiscountStrategy {
    @Override
    public BigDecimal calculate(BigDecimal price, Object ctx) {
        double rate = (double) ctx; // ctx 传入折扣率，如 0.8
        return price.multiply(BigDecimal.valueOf(rate)).setScale(2, RoundingMode.HALF_UP);
    }
    @Override public String getName() { return "PERCENTAGE"; }
}

// 满减（满100减20）
@Component("FULL_REDUCTION")
public class FullReductionStrategy implements DiscountStrategy {
    @Override
    public BigDecimal calculate(BigDecimal price, Object ctx) {
        // ctx: Map{threshold:100, reduction:20}
        Map rule = (Map) ctx;
        BigDecimal threshold = rule.get("threshold");
        BigDecimal reduction = rule.get("reduction");
        if (price.compareTo(threshold) >= 0) {
            return price.subtract(reduction);
        }
        return price;
    }
    @Override public String getName() { return "FULL_REDUCTION"; }
}

// 会员折扣（根据会员等级）
@Component("MEMBER")
public class MemberDiscountStrategy implements DiscountStrategy {
    private static final Map RATES = Map.of(
        "SILVER", 0.95, "GOLD", 0.90, "PLATINUM", 0.85
    );
    @Override
    public BigDecimal calculate(BigDecimal price, Object ctx) {
        String level = (String) ctx;
        double rate = RATES.getOrDefault(level, 1.0);
        return price.multiply(BigDecimal.valueOf(rate)).setScale(2, RoundingMode.HALF_UP);
    }
    @Override public String getName() { return "MEMBER"; }
}

// Step 3：策略上下文（Context）
@Service
public class PriceCalculationService {
    @Autowired
    private Map strategyMap;
    // Spring 自动填充：{"NO_DISCOUNT": NoDiscountStrategy, "PERCENTAGE": ...}

    public BigDecimal calculatePrice(String strategyName,
                                     BigDecimal originalPrice,
                                     Object context) {
        DiscountStrategy strategy = strategyMap.get(strategyName);
        if (strategy == null) {
            throw new IllegalArgumentException("未知折扣策略：" + strategyName);
        }
        BigDecimal result = strategy.calculate(originalPrice, context);
        // 保证折扣价不低于0
        return result.max(BigDecimal.ZERO);
    }
}

// 使用示例
BigDecimal price = new BigDecimal("199.00");
// 8折
BigDecimal discounted = priceService.calculatePrice("PERCENTAGE", price, 0.8);
// 满100减20
Map rule = Map.of("threshold", new BigDecimal("100"),
                                       "reduction", new BigDecimal("20"));
BigDecimal fullReduced = priceService.calculatePrice("FULL_REDUCTION", price, rule);
⚠️

实战陷阱

陷阱1：策略模式不能完全消灭所有 if-else
"选择哪个策略"本身可能还需要判断逻辑。策略模式消灭的是算法内部的 if-else，而非"如何选择策略"的 if-else。可以结合工厂模式（或 Spring Map 注入）来消灭策略选择处的 if-else。
陷阱2：策略对象有状态导致并发问题
Java
// ❌ 危险：策略类有可变状态，Spring 单例 Bean 下多线程并发修改
@Component("statefulStrategy")
public class StatefulDiscountStrategy implements DiscountStrategy {
    private int callCount = 0; // 可变状态！Spring 单例，多线程共享，竞态条件

    @Override
    public BigDecimal calculate(BigDecimal price, Object ctx) {
        callCount++; // ❌ 线程不安全
        return price.multiply(BigDecimal.valueOf(0.9));
    }
}

// ✅ 正确：策略类应该是无状态的（Stateless）
@Component("safeStrategy")
public class SafeDiscountStrategy implements DiscountStrategy {
    // 无实例变量，纯函数式，天然线程安全
    @Override
    public BigDecimal calculate(BigDecimal price, Object ctx) {
        return price.multiply(BigDecimal.valueOf(0.9));
    }
}

与其他模式的关系

相关模式关系说明
工厂模式工厂创建策略对象；两者常配合使用（工厂负责实例化，策略负责算法）
状态模式结构几乎相同，但意图不同：策略由客户端选择，状态由对象内部自动切换
模板方法模板方法通过继承改变算法部分；策略通过组合整体替换算法
命令模式命令封装请求为对象，策略封装算法为对象；命令可撤销，策略通常不能
