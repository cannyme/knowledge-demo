工厂模式

将对象的创建与使用分离，由工厂类负责对象创建。Spring BeanFactory、SLF4J LoggerFactory、JDBC DriverManager 均是工厂模式的经典应用。

📖

代码实现

简单工厂 — 集中创建，违反开闭原则（新增产品需修改工厂类）

Java
// 产品接口
interface Payment {
    void pay(double amount);
}

class AlipayPayment implements Payment {
    @Override public void pay(double amount) {
        System.out.println("支付宝支付：" + amount + "元");
    }
}

class WechatPayment implements Payment {
    @Override public void pay(double amount) {
        System.out.println("微信支付：" + amount + "元");
    }
}

// 简单工厂：通过参数决定产品类型
class SimplePaymentFactory {
    public static Payment create(String type) {
        switch (type.toLowerCase()) {
            case "alipay": return new AlipayPayment();
            case "wechat": return new WechatPayment();
            default: throw new IllegalArgumentException("不支持的支付方式：" + type);
        }
    }
}

// 使用
Payment p = SimplePaymentFactory.create("alipay");
p.pay(100.0);
⚠️ 新增支付方式时必须修改 switch，违反开闭原则。

工厂方法 — 每产品对应专属工厂，新增产品只需新增工厂 推荐

Java
// 抽象工厂接口
interface PaymentFactory {
    Payment create();
}

// 每种支付方式各自一个工厂
class AlipayFactory implements PaymentFactory {
    @Override public Payment create() { return new AlipayPayment(); }
}

class WechatFactory implements PaymentFactory {
    @Override public Payment create() { return new WechatPayment(); }
}

// 新增银联：只加新类，不改旧代码（开闭原则）
class UnionPayFactory implements PaymentFactory {
    @Override public Payment create() { return new UnionPayPayment(); }
}

// 使用：面向接口编程，运行时注入具体工厂
PaymentFactory factory = new WechatFactory();
Payment payment = factory.create();
payment.pay(200.0);

抽象工厂 — 创建一族相关产品，整体切换产品族

Java
// 产品族：跨平台 UI 组件
interface Button { void click(); }
interface TextField { void input(String text); }

// Windows 产品族
class WindowsButton implements Button {
    @Override public void click() { System.out.println("Windows 按钮点击"); }
}
class WindowsTextField implements TextField {
    @Override public void input(String t) { System.out.println("Windows 输入：" + t); }
}

// Mac 产品族
class MacButton implements Button {
    @Override public void click() { System.out.println("Mac 按钮点击"); }
}
class MacTextField implements TextField {
    @Override public void input(String t) { System.out.println("Mac 输入：" + t); }
}

// 抽象工厂：生产一整族产品
interface UIFactory {
    Button createButton();
    TextField createTextField();
}

class WindowsUIFactory implements UIFactory {
    @Override public Button createButton() { return new WindowsButton(); }
    @Override public TextField createTextField() { return new WindowsTextField(); }
}

class MacUIFactory implements UIFactory {
    @Override public Button createButton() { return new MacButton(); }
    @Override public TextField createTextField() { return new MacTextField(); }
}

// 使用：切换工厂即切换整个 UI 风格
boolean isWindows = System.getProperty("os.name").contains("Windows");
UIFactory factory = isWindows ? new WindowsUIFactory() : new MacUIFactory();
Button btn = factory.createButton();
TextField tf = factory.createTextField();

面试高频考点

Q1：三种工厂模式的核心区别？
  • 简单工厂：一个工厂类+参数，违反开闭原则，适合产品固定少量
  • 工厂方法：每个产品一个工厂类，扩展时只加新工厂，符合开闭原则
  • 抽象工厂：解决多产品维度问题，扩展产品族容易，扩展产品等级结构难
Q2：BeanFactory 和 FactoryBean 的区别？（超高频陷阱）
对比BeanFactoryFactoryBean
本质Spring IoC 容器接口，管理所有 Bean特殊 Bean，用于生产其他 Bean
作用存储和管理 Bean 的工厂封装复杂对象的创建逻辑
获取产品getBean("name")getBean("factoryBeanName") 返回产品
获取工厂本身N/AgetBean("&factoryBeanName") 加&前缀
Java
// FactoryBean 示例：封装复杂对象的创建（MyBatis SqlSessionFactoryBean 就是这样做的）
@Component
public class MyObjectFactoryBean implements FactoryBean<MyComplexObject> {
    @Override
    public MyComplexObject getObject() throws Exception {
        // 执行复杂初始化（连接池、配置解析等）
        MyComplexObject obj = new MyComplexObject();
        obj.init("complex-config");
        return obj;
    }
    @Override public Class<?> getObjectType() { return MyComplexObject.class; }
    @Override public boolean isSingleton() { return true; }
}

// 获取方式
MyComplexObject obj = ctx.getBean("myObjectFactoryBean");    // 产品对象
MyObjectFactoryBean fb = ctx.getBean("&myObjectFactoryBean"); // FactoryBean 本身
Q3：BeanFactory 与 ApplicationContext 的区别？
  • BeanFactory：基础容器，懒加载（第一次 getBean 时才初始化 Bean）
  • ApplicationContext：BeanFactory 子接口，启动时预实例化所有单例 Bean，额外支持：国际化(i18n)、事件发布、AOP 集成、资源加载
  • 实际开发：几乎全用 ApplicationContext（ClassPathXmlApplicationContext、AnnotationConfigApplicationContext）

实战：Spring 工厂方法消灭 if-else

Java
// Step 1：定义策略接口
public interface PaymentStrategy {
    void pay(double amount);
}

// Step 2：各实现类用 @Component 注册，beanName 即为 key
@Component("alipay")
public class AlipayStrategy implements PaymentStrategy {
    @Override public void pay(double amount) { System.out.println("支付宝：" + amount); }
}

@Component("wechat")
public class WechatStrategy implements PaymentStrategy {
    @Override public void pay(double amount) { System.out.println("微信：" + amount); }
}

// 新增银联：只加这一个类，其他代码不动
@Component("unionpay")
public class UnionPayStrategy implements PaymentStrategy {
    @Override public void pay(double amount) { System.out.println("银联：" + amount); }
}

// Step 3：Spring 自动注入所有实现到 Map（key=beanName）
@Service
public class PaymentService {
    @Autowired
    private Map<String, PaymentStrategy> strategyMap; // Spring 自动填充

    public void pay(String type, double amount) {
        // 完全没有 if-else！对修改关闭，对扩展开放
        PaymentStrategy strategy = strategyMap.get(type);
        if (strategy == null) throw new IllegalArgumentException("不支持：" + type);
        strategy.pay(amount);
    }
}
💡 Spring 会将所有 PaymentStrategy 实现以 beanName 为 key 自动注入 Map，无需任何额外配置。新增支付方式只需加一个带 @Component 的类。

实战陷阱

陷阱1：用 FactoryBean 注册的 Bean，@Autowired 注入的是产品而非工厂
Java
// 注意：@Autowired 注入的是 FactoryBean.getObject() 的返回值
@Autowired
private SqlSessionFactory sqlSessionFactory; // ✅ 注入产品（SqlSessionFactory）

// 若要注入 FactoryBean 本身，用 & 前缀
@Autowired
private FactoryBean factoryBean; // ❌ 报错，需要用 & 前缀获取
陷阱2：简单工厂被误称为"工厂模式"
简单工厂（Static Factory Method）不是 GoF 23 种设计模式之一，只是一种编程惯用法。面试时注意区分：工厂方法模式才是 GoF 设计模式，核心是"将对象创建延迟到子类"。
最佳实践：用 Spring 注入 Map 消灭 if-else 工厂
见上方「实战：Spring 工厂方法消灭 if-else」章节，这是实际项目中最优雅的工厂 + 策略组合写法。
⚖️

优缺点分析

✅ 优点

  • 解耦创建与使用：客户端不依赖具体类
  • 扩展方便：工厂方法新增产品只加类，不改旧代码
  • 单一职责：创建逻辑集中管理
  • 屏蔽复杂性：封装对象创建的复杂初始化（如连接池、配置解析）

❌ 缺点

  • 类数量增多：每个产品一个工厂类（工厂方法）
  • 复杂度增加：抽象工厂在增加产品等级时需大量修改
  • 客户端耦合工厂：客户端仍需知道用哪个工厂

与其他模式的关系

相关模式关系说明
单例模式工厂类本身通常实现为单例；Spring BeanFactory 就是单例的工厂
策略模式工厂决定"创建哪个对象"，策略决定"用哪种算法"；常结合使用（工厂创建策略对象）
模板方法工厂方法模式中，父类定义 createProduct() 钩子，子类覆盖，本质是模板方法的应用
建造者模式工厂关注"创建哪种对象"，Builder 关注"如何逐步构建一个复杂对象"
