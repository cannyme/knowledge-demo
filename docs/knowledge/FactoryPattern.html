工厂模式

将对象的创建与使用分离，由工厂类负责创建对象。工厂模式是创建型设计模式中最常用的模式之一，分为简单工厂、工厂方法和抽象工厂三种形式。

📖

模式定义与意图

定义：工厂模式（Factory Pattern）是一种创建型设计模式，它提供了一种创建对象的最佳方式，而无需向客户端暴露创建逻辑。

核心思想：将对象的创建（new操作）从客户端代码中抽离出来，交给专门的工厂类处理，实现解耦。

三种形式：

  • 简单工厂（Simple Factory）：一个工厂类根据参数创建不同产品（不属于GoF 23种模式）
  • 工厂方法（Factory Method）：定义创建对象的接口，让子类决定实例化哪个类
  • 抽象工厂（Abstract Factory）：创建一系列相关或依赖对象的接口（产品族）

适用场景

以下场景适合使用工厂模式：

  • 日志框架：SLF4J、Log4j等日志框架通过工厂创建具体日志实现
  • 数据库连接：不同数据库（MySQL、Oracle、PostgreSQL）的Connection创建
  • 支付系统：支持多种支付方式（支付宝、微信、银行卡）
  • Spring框架：BeanFactory负责创建和管理Bean对象
  • 跨平台UI组件：不同操作系统创建对应的UI组件

代码实现

简单工厂（Simple Factory） - 一个工厂类创建所有产品，通过参数决定创建哪种产品

Java
// 产品接口
interface Payment {
    void pay(double amount);
}

// 具体产品
class AlipayPayment implements Payment {
    @Override
    public void pay(double amount) {
        System.out.println("支付宝支付：" + amount + "元");
    }
}

class WechatPayment implements Payment {
    @Override
    public void pay(double amount) {
        System.out.println("微信支付：" + amount + "元");
    }
}

class BankCardPayment implements Payment {
    @Override
    public void pay(double amount) {
        System.out.println("银行卡支付：" + amount + "元");
    }
}

// 简单工厂
class SimplePaymentFactory {
    public static Payment create(String type) {
        switch (type.toLowerCase()) {
            case "alipay": return new AlipayPayment();
            case "wechat": return new WechatPayment();
            case "bank": return new BankCardPayment();
            default: throw new IllegalArgumentException("不支持的支付方式");
        }
    }
}

// 使用
Payment payment = SimplePaymentFactory.create("alipay");
payment.pay(100.0);

特点：简单直观，但违反开闭原则（新增产品需修改工厂类）

工厂方法（Factory Method） - 每个产品有对应的工厂，新增产品只需新增工厂类

Java
// 工厂接口
interface PaymentFactory {
    Payment create();
}

// 具体工厂：支付宝
class AlipayFactory implements PaymentFactory {
    @Override
    public Payment create() {
        return new AlipayPayment();
    }
}

// 具体工厂：微信
class WechatFactory implements PaymentFactory {
    @Override
    public Payment create() {
        return new WechatPayment();
    }
}

// 具体工厂：银行卡
class BankCardFactory implements PaymentFactory {
    @Override
    public Payment create() {
        return new BankCardPayment();
    }
}

// 使用
PaymentFactory factory = new WechatFactory();
Payment payment = factory.create();
payment.pay(200.0);

特点：符合开闭原则，但客户端需要知道具体工厂类

抽象工厂（Abstract Factory） - 创建一系列相关对象（产品族）

Java
// 产品族：家具
interface Chair {
    void sit();
}

interface Table {
    void put(String item);
}

// 现代风格产品
class ModernChair implements Chair {
    @Override
    public void sit() {
        System.out.println("坐在现代风格的椅子上");
    }
}

class ModernTable implements Table {
    @Override
    public void put(String item) {
        System.out.println("在现代风格的桌子上放：" + item);
    }
}

// 古典风格产品
class ClassicalChair implements Chair {
    @Override
    public void sit() {
        System.out.println("坐在古典风格的椅子上");
    }
}

class ClassicalTable implements Table {
    @Override
    public void put(String item) {
        System.out.println("在古典风格的桌子上放：" + item);
    }
}

// 抽象工厂接口
interface FurnitureFactory {
    Chair createChair();
    Table createTable();
}

// 现代风格工厂
class ModernFurnitureFactory implements FurnitureFactory {
    @Override
    public Chair createChair() {
        return new ModernChair();
    }

    @Override
    public Table createTable() {
        return new ModernTable();
    }
}

// 古典风格工厂
class ClassicalFurnitureFactory implements FurnitureFactory {
    @Override
    public Chair createChair() {
        return new ClassicalChair();
    }

    @Override
    public Table createTable() {
        return new ClassicalTable();
    }
}

// 使用
FurnitureFactory factory = new ModernFurnitureFactory();
Chair chair = factory.createChair();
Table table = factory.createTable();
chair.sit();
table.put("书籍");

产品族概念：现代风格家具族 = 现代椅子 + 现代桌子；古典风格家具族 = 古典椅子 + 古典桌子

三种工厂模式对比

特性 简单工厂 工厂方法 抽象工厂
产品结构 单一产品 单一产品 产品族（多个相关产品）
扩展性 需修改代码 新增工厂类 新增工厂类
复杂度 低 中 高
使用场景 产品种类少且固定 产品频繁扩展 需要创建产品族
⚖️

优缺点分析

优点

  • 解耦：将对象的创建和使用分离，降低耦合度
  • 可扩展：新增产品只需新增工厂类（工厂方法/抽象工厂）
  • 可维护：集中管理对象的创建逻辑
  • 复用：工厂可以被多个客户端复用

缺点

  • 类数量增加：每个产品对应一个工厂类，增加系统复杂度
  • 理解难度：引入新的抽象层，增加理解成本
  • 简单工厂缺陷：新增产品需要修改工厂类，违反开闭原则

与其他模式的关系

相关模式 关系说明
单例模式 工厂类通常设计为单例，确保只有一个工厂实例存在
策略模式 工厂模式负责创建策略对象，策略模式负责切换算法
依赖注入 Spring的IoC容器是工厂模式的升华，通过配置创建Bean
模板方法 工厂方法模式常使用模板方法定义创建流程

Spring中的工厂：BeanFactory和ApplicationContext是工厂模式的典型应用，通过配置文件或注解定义Bean的创建方式，实现了完全的解耦。
