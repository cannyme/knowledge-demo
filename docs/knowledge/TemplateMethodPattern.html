模板方法模式

在父类中定义算法骨架，将某些步骤延迟到子类实现。子类可以在不改变算法结构的前提下，重定义特定步骤。Spring JdbcTemplate、HttpServlet、JUnit 均是其经典应用。

📖

模式定义与结构

核心思想：算法骨架固定在父类（final 修饰），可变步骤抽象为 abstract 方法由子类实现，可选步骤作为钩子方法（hook，提供空默认实现）由子类按需覆盖。

classDiagram class AbstractClass { +templateMethod()$ +primitiveOp1()* +primitiveOp2()* +hook() +concreteOp() } class ConcreteClassA { +primitiveOp1() +primitiveOp2() } class ConcreteClassB { +primitiveOp1() +primitiveOp2() +hook() } AbstractClass <|-- ConcreteClassA AbstractClass <|-- ConcreteClassB
方法类型修饰符说明
模板方法final定义算法骨架，禁止子类重写
抽象步骤abstract protected子类必须实现的可变步骤
钩子方法protected（空实现）子类可选择性覆盖，默认什么都不做
具体方法protected/private通用逻辑，子类直接复用

代码实现

数据处理器模板 — 骨架固定（验证→解析→处理→保存→日志），处理逻辑由子类决定

Java
// 抽象模板类
abstract class DataProcessor {
    // 模板方法：final 禁止子类修改算法骨架
    public final void process(String data) {
        if (!validate(data)) {           // 步骤1：验证（具体方法，通用逻辑）
            System.out.println("数据验证失败");
            return;
        }
        Object parsed = parse(data);     // 步骤2：解析（具体方法）
        Object result = doProcess(parsed); // 步骤3：处理（抽象方法，子类必须实现）
        save(result);                    // 步骤4：保存（具体方法）
        afterProcess();                  // 步骤5：后置处理（钩子方法，子类可选）
    }

    // 具体方法：通用验证逻辑
    protected boolean validate(String data) {
        return data != null && !data.isEmpty();
    }

    // 具体方法：通用解析逻辑
    protected Object parse(String data) {
        System.out.println("解析数据：" + data);
        return data.trim();
    }

    // 抽象方法：子类必须实现各自的处理逻辑
    protected abstract Object doProcess(Object data);

    // 具体方法：通用保存逻辑
    protected void save(Object result) {
        System.out.println("保存结果：" + result);
    }

    // 钩子方法：默认空实现，子类按需覆盖
    protected void afterProcess() {}
}

// JSON 处理器：覆盖 doProcess + 钩子
class JsonDataProcessor extends DataProcessor {
    @Override
    protected Object doProcess(Object data) {
        System.out.println("JSON 序列化：" + data);
        return "{\"result\": \"" + data + "\"}";
    }

    @Override
    protected void afterProcess() {
        System.out.println("[JSON] 处理完成，发送回调通知");
    }
}

// XML 处理器：还覆盖了 parse（特殊解析逻辑）
class XmlDataProcessor extends DataProcessor {
    @Override
    protected Object parse(String data) {
        System.out.println("XML 特殊解析：" + data);
        return data.toUpperCase(); // 覆盖父类解析逻辑
    }

    @Override
    protected Object doProcess(Object data) {
        System.out.println("XML 封装：" + data);
        return "" + data + "";
    }
}

// 使用
DataProcessor jsonProcessor = new JsonDataProcessor();
jsonProcessor.process("hello world");
// 输出：解析数据→JSON序列化→保存结果→发送回调通知

DataProcessor xmlProcessor = new XmlDataProcessor();
xmlProcessor.process("hello world");
// 输出：XML特殊解析→XML封装→保存结果

订单处理模板 — 实际项目中最常见的模板方法应用场景

Java
abstract class OrderProcessor {
    // 模板方法：订单处理骨架（final 保护）
    public final void processOrder(Order order) {
        validateOrder(order);             // 1. 校验订单
        double price = calculatePrice(order); // 2. 计算价格（子类可覆盖）
        order.setTotalPrice(price);
        deductStock(order);               // 3. 扣减库存
        boolean paid = doPayment(order);  // 4. 支付（抽象，子类实现）
        if (!paid) { rollback(order); return; }
        createShipping(order);            // 5. 创建物流
        sendNotification(order);          // 6. 发送通知（钩子）
    }

    protected void validateOrder(Order order) {
        if (order.getItems() == null || order.getItems().isEmpty())
            throw new IllegalArgumentException("订单商品为空");
    }

    // 子类可覆盖以实现折扣等逻辑
    protected double calculatePrice(Order order) {
        return order.getItems().stream().mapToDouble(Item::getPrice).sum();
    }

    protected void deductStock(Order order) {
        System.out.println("扣减库存");
    }

    // 抽象：不同类型订单支付方式不同
    protected abstract boolean doPayment(Order order);

    protected void rollback(Order order) {
        System.out.println("支付失败，回滚库存");
    }

    protected void createShipping(Order order) {
        System.out.println("创建物流单");
    }

    // 钩子：默认不发通知，VIP 订单覆盖
    protected void sendNotification(Order order) {}
}

// 普通订单
class NormalOrderProcessor extends OrderProcessor {
    @Override
    protected boolean doPayment(Order order) {
        System.out.println("普通支付：" + order.getTotalPrice() + " 元");
        return true;
    }
}

// VIP 订单：覆盖价格计算 + 支付 + 通知钩子
class VipOrderProcessor extends OrderProcessor {
    @Override
    protected double calculatePrice(Order order) {
        return super.calculatePrice(order) * 0.9; // 9折
    }

    @Override
    protected boolean doPayment(Order order) {
        System.out.println("VIP支付（已优惠）：" + order.getTotalPrice() + " 元");
        return true;
    }

    @Override
    protected void sendNotification(Order order) {
        System.out.println("发送 VIP 专属短信通知");
    }
}

回调方式（组合替代继承） — Java 8+ 用 Lambda 替代子类，更灵活，Spring JdbcTemplate 就是此方式

Java
// 模板类：用函数式接口代替抽象方法
class DataProcessorTemplate {
    // 模板方法：骨架固定，可变步骤通过 Lambda 传入
    public  T process(String data,
                         Function processor,   // 可变：处理逻辑
                         Consumer afterHook) {          // 可选钩子
        // 固定步骤1：验证
        if (data == null || data.isEmpty()) return null;
        // 固定步骤2：调用可变处理逻辑
        T result = processor.apply(data);
        // 固定步骤3：保存
        System.out.println("保存：" + result);
        // 可选步骤4：执行钩子
        if (afterHook != null) afterHook.accept(result);
        return result;
    }
}

// 使用：无需写子类，直接 Lambda
DataProcessorTemplate template = new DataProcessorTemplate();

// JSON 处理
template.process("hello",
    data -> "{\"msg\": \"" + data + "\"}",   // 处理逻辑
    result -> System.out.println("JSON 回调：" + result) // 钩子
);

// Spring JdbcTemplate 就是这种思路
// jdbcTemplate.query(sql, (rs, rowNum) -> new User(rs.getString("name")));
// ↑ 骨架（获取连接、执行SQL、关闭连接）固定，RowMapper 是可变的"处理步骤"
继承 vs 回调：继承方式（传统模板方法）适合步骤较多、子类需要复用多个固定步骤的场景；回调方式（Lambda）更轻量，适合可变步骤少且简单的场景，避免为每种变体创建子类。

框架中的经典应用

1. HttpServlet — 最经典的模板方法

Java - HttpServlet 核心
// HttpServlet（简化版）是模板方法的教科书实现
public abstract class HttpServlet extends GenericServlet {
    // 模板方法：根据 HTTP 方法路由到不同的钩子
    protected void service(HttpServletRequest req, HttpServletResponse resp) {
        String method = req.getMethod();
        if ("GET".equals(method)) {
            doGet(req, resp);    // 钩子方法
        } else if ("POST".equals(method)) {
            doPost(req, resp);   // 钩子方法
        } else if ("PUT".equals(method)) {
            doPut(req, resp);
        } else if ("DELETE".equals(method)) {
            doDelete(req, resp);
        }
        // ...
    }

    // 所有钩子方法默认返回 405 Method Not Allowed
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        resp.sendError(HttpServletResponse.SC_METHOD_NOT_ALLOWED);
    }
    protected void doPost(HttpServletRequest req, HttpServletResponse resp) {
        resp.sendError(HttpServletResponse.SC_METHOD_NOT_ALLOWED);
    }
}

// 开发者只需覆盖需要的方法（其余钩子保持默认 405）
public class UserServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws IOException {
        resp.getWriter().write("GET /users");
    }

    @Override
    protected void doPost(HttpServletRequest req, HttpServletResponse resp)
            throws IOException {
        resp.getWriter().write("POST /users - create user");
    }
}

2. Spring JdbcTemplate — 回调方式的模板方法

Java - JdbcTemplate 骨架
// JdbcTemplate 模板方法（简化版）
public class JdbcTemplate {
    // 模板方法：获取连接→执行SQL→映射结果→释放连接
    public  T query(String sql, RowMapper rowMapper, Object... args) {
        Connection conn = null;
        PreparedStatement ps = null;
        ResultSet rs = null;
        try {
            conn = dataSource.getConnection();          // 固定：获取连接
            ps = conn.prepareStatement(sql);            // 固定：创建语句
            for (int i = 0; i < args.length; i++) {
                ps.setObject(i + 1, args[i]);
            }
            rs = ps.executeQuery();                     // 固定：执行查询
            return rowMapper.mapRow(rs, 0);             // 可变：结果映射（回调）
        } catch (SQLException e) {
            throw new DataAccessException(e);
        } finally {
            JdbcUtils.closeResultSet(rs);               // 固定：释放资源
            JdbcUtils.closeStatement(ps);
            JdbcUtils.closeConnection(conn);
        }
    }
}

// 使用：只需提供"结果映射"这一步的逻辑
User user = jdbcTemplate.query(
    "SELECT * FROM user WHERE id = ?",
    (rs, rowNum) -> {                   // RowMapper Lambda（可变步骤）
        User u = new User();
        u.setId(rs.getLong("id"));
        u.setName(rs.getString("name"));
        u.setEmail(rs.getString("email"));
        return u;
    },
    userId
);

3. AbstractQueuedSynchronizer（AQS）— 并发框架的模板方法

Java - AQS 模板方法
// AQS 是 ReentrantLock、Semaphore、CountDownLatch 的基础
// 它定义了加锁/释放锁的骨架，子类只需实现 tryAcquire/tryRelease
public abstract class AbstractQueuedSynchronizer {
    // 模板方法：排他锁获取骨架
    public final void acquire(int arg) {
        if (!tryAcquire(arg))          // 子类实现：尝试获取锁
            acquireQueued(addWaiter(Node.EXCLUSIVE), arg); // 骨架：入队等待
    }

    // 模板方法：排他锁释放骨架
    public final boolean release(int arg) {
        if (tryRelease(arg)) {         // 子类实现：尝试释放锁
            unparkSuccessor(head);     // 骨架：唤醒后继节点
            return true;
        }
        return false;
    }

    // 抽象钩子：子类必须实现
    protected boolean tryAcquire(int arg) {
        throw new UnsupportedOperationException();
    }
    protected boolean tryRelease(int arg) {
        throw new UnsupportedOperationException();
    }
}

// ReentrantLock 只需实现 tryAcquire/tryRelease
class NonfairSync extends AbstractQueuedSynchronizer {
    @Override
    protected boolean tryAcquire(int acquires) {
        // 非公平锁：直接 CAS 抢锁
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(Thread.currentThread());
            return true;
        }
        return false;
    }
}

面试高频考点

Q1：模板方法模式 vs 策略模式的核心区别？
维度模板方法策略模式
实现方式继承（子类重写步骤）组合（注入不同算法对象）
粒度改变算法中的某些步骤替换整个算法
运行时切换不可以（编译时确定）可以（运行时 setStrategy）
耦合度强（子类依赖父类）弱（接口依赖）
适用场景骨架固定、步骤可变算法族完全可替换
记忆口诀：模板方法是"继承+部分覆盖"，策略模式是"组合+整体替换"。
Q2：为什么模板方法要用 final 修饰？不加会怎样？
final 的作用是保护算法骨架不被子类篡改。如果不加 final，子类可以覆盖模板方法，完全改变算法的执行顺序（如跳过验证步骤），破坏了"骨架固定"的设计意图，也违反了里氏替换原则中"子类不应改变父类的行为契约"的精神。
Q3：什么是好莱坞原则？和模板方法的关系？
好莱坞原则："Don't call us, we'll call you"（不要打电话给我们，我们会打给你）。在模板方法中体现为：父类调用子类的方法，而不是子类调用父类。子类（低层组件）实现步骤，父类（高层组件）控制调用时机和顺序。这是一种"反向控制"，避免了低层组件主动依赖高层组件。
Q4：JUnit 测试框架用了模板方法吗？
是的。JUnit 3 中 TestCase 是抽象类，runTest() 是模板方法，执行骨架为：setUp() → runTest() → tearDown()。开发者继承 TestCase 后，覆盖 setUp()/tearDown()（钩子）和各 test* 方法（抽象步骤）。JUnit 4/5 改用注解（@BeforeEach/@AfterEach），本质是回调方式的模板方法。

实战陷阱

陷阱1：Java 单继承限制扩展性
模板方法依赖继承，而 Java 只支持单继承。如果子类已经继承了其他类，就无法使用模板方法模式。解决方案：改用回调方式（接口 + Lambda），或用组合代替继承。
陷阱2：钩子方法设计粒度不当
钩子方法太多：子类负担重，每次扩展都要翻看父类；太少：灵活性不够，子类无法定制关键步骤。原则：只把真正需要定制的步骤暴露为钩子，其余用具体方法封装。
陷阱3：子类覆盖了不该覆盖的具体方法
Java
// 父类具体方法（本意：通用逻辑，子类无需改）
protected void save(Object result) {
    database.save(result); // 通用保存逻辑
}

// ❌ 子类错误覆盖了具体方法，破坏了模板的预期行为
class BrokenProcessor extends DataProcessor {
    @Override
    protected void save(Object result) {
        // 忘记调用 super.save()，数据丢失！
        System.out.println("只打印不保存");
    }
}
// 解决：将不希望被覆盖的具体方法改为 private 或 final

优缺点分析

✅ 优点

  • 代码复用：公共逻辑在父类集中管理，子类零重复
  • 开闭原则：新增变体只需新增子类，不改父类骨架
  • 反向控制：父类控制流程，子类只实现步骤
  • 强制规范：final 保证所有子类遵循统一流程

❌ 缺点

  • 继承耦合：子类与父类强耦合，父类变化影响所有子类
  • 单继承限制：Java 无法多继承，扩展受限
  • 类数量增多：每种变体一个子类
  • 违反里氏替换：父类修改可能破坏子类行为

与其他模式的关系

相关模式关系说明
策略模式两者都用于封装变化的行为：模板方法用继承改变部分步骤；策略用组合替换整个算法
工厂方法工厂方法是模板方法的特例：模板方法中有一步是"创建对象"，该步骤由子类实现，就是工厂方法模式

钩子方法 vs 观察者模板方法的钩子是父类主动调用子类方法（继承）；观察者是主题通知观察者（组合）
