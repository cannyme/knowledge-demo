Kafka核心概念与使用

Apache Kafka是一个分布式流处理平台，具有高性能、高吞吐、可持久化、可横向扩展等特点。广泛应用于日志收集、流处理、事件溯源等大数据场景，单机吞吐量可达100万/秒

🎯

生产者配置

ACK机制 - 消息可靠性配置

acks配置行为可靠性性能
acks=0不等待确认，发送即认为成功最低（可能丢失）最高
acks=1等待Leader确认（默认）中等（Leader宕机可能丢失）中等
acks=all等待所有ISR确认最高最低
Java
// ACK配置
props.put(ProducerConfig.ACKS_CONFIG, "all");           // 最高可靠性
props.put(ProducerConfig.RETRIES_CONFIG, 3);            // 发送失败重试次数
props.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 1); // 保证顺序

@Configuration
public class KafkaProducerConfig {
    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        config.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        config.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        config.put(ProducerConfig.ACKS_CONFIG, "all");
        config.put(ProducerConfig.RETRIES_CONFIG, 3);
        config.put(ProducerConfig.BATCH_SIZE_CONFIG, 16384);
        config.put(ProducerConfig.LINGER_MS_CONFIG, 5);
        config.put(ProducerConfig.BUFFER_MEMORY_CONFIG, 33554432);
        return new DefaultKafkaProducerFactory<>(config);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}

发送消息示例

Java
@Service
public class OrderProducer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    // 同步发送
    public void sendSync(String topic, String key, String message) {
        try {
            SendResult<String, String> result = kafkaTemplate.send(topic, key, message).get();
            RecordMetadata metadata = result.getRecordMetadata();
            System.out.println("发送成功: " + metadata.partition() + "-" + metadata.offset());
        } catch (Exception e) {
            System.err.println("发送失败: " + e.getMessage());
        }
    }

    // 异步发送
    public void sendAsync(String topic, String message) {
        CompletableFuture<SendResult<String, String>> future = kafkaTemplate.send(topic, message);
        future.whenComplete((result, ex) -> {
            if (ex == null) {
                System.out.println("发送成功");
            } else {
                System.err.println("发送失败: " + ex.getMessage());
            }
        });
    }

    // 发送到指定分区
    public void sendToPartition(String topic, int partition, String key, String message) {
        kafkaTemplate.send(topic, partition, key, message);
    }
}

消费者组与分区分配

消费者组工作原理

flowchart TD subgraph "Consumer Group A" C1[Consumer 1] --> P0[Partition 0] C2[Consumer 2] --> P1[Partition 1] C3[Consumer 3] --> P2[Partition 2] end subgraph "Consumer Group B" C4[Consumer 4] --> P0 C4 --> P1 C4 --> P2 end style C1 fill:#3a2e15,stroke:#f0b429,color:#f0b429 style C2 fill:#3a2e15,stroke:#f0b429,color:#f0b429 style C3 fill:#3a2e15,stroke:#f0b429,color:#f0b429 style C4 fill:#0d3d3a,stroke:#4ecdc4,color:#4ecdc4
💡 消费者组规则：1) 一个分区只能被同组一个消费者消费 2) 消费者数 > 分区数时，有消费者空闲 3) 消费者故障时触发Rebalance（再均衡）

Rebalance再均衡

当消费者组成员变化时，Kafka会重新分配分区给消费者。

触发条件影响优化方案
消费者加入/离开组消费暂停合理设置session.timeout.ms
消费者超时Offset可能丢失合理设置heartbeat.interval.ms
Topic分区数变化重复消费使用静态成员（group.instance.id）
Java
@Configuration
@EnableKafka
public class KafkaConsumerConfig {
    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        config.put(ConsumerConfig.GROUP_ID_CONFIG, "order-group");
        config.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        config.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        config.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest");
        config.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, false);
        // 避免频繁Rebalance
        config.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG, 10000);
        config.put(ConsumerConfig.HEARTBEAT_INTERVAL_MS_CONFIG, 3000);
        return new DefaultKafkaConsumerFactory<>(config);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        factory.setConcurrency(3);
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
        return factory;
    }
}

Offset管理

自动提交（不推荐生产环境使用）

Java
// 自动提交配置
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, true);
props.put(ConsumerConfig.AUTO_COMMIT_INTERVAL_MS_CONFIG, 5000); // 每5秒提交

@KafkaListener(topics = "orders", groupId = "order-group")
public void consume(String message) {
    // 处理消息
    processMessage(message);
    // Offset自动提交，可能在处理失败前已提交，导致消息丢失
}
⚠️ 风险：自动提交可能导致消息丢失。如果消费处理失败但Offset已自动提交，消息将不会被重新消费。

手动提交（推荐）

Java
// 手动提交配置
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, false);

// 手动同步提交
@KafkaListener(topics = "orders", groupId = "order-group")
public void consumeWithSyncAck(ConsumerRecord<String, String> record, Acknowledgment ack) {
    try {
        processMessage(record.value());
        ack.acknowledge(); // 业务处理成功后提交
    } catch (Exception e) {
        // 不提交，消息会重新投递
        log.error("处理失败", e);
    }
}

// 批量消费手动提交
@KafkaListener(topics = "orders", groupId = "order-group",
               containerFactory = "batchFactory")
public void consumeBatch(List<ConsumerRecord<String, String>> records, Acknowledgment ack) {
    try {
        for (ConsumerRecord<String, String> record : records) {
            processMessage(record.value());
        }
        ack.acknowledge(); // 批量提交
    } catch (Exception e) {
        log.error("批量处理失败", e);
    }
}
提交方式优点缺点适用场景
自动提交实现简单可能丢失消息对丢失不敏感的场景
手动同步提交不会丢失消息阻塞，影响吞吐一般业务场景
手动异步提交性能好可能重复消费高性能要求场景

分区策略

生产者发送消息时，需要决定消息发送到哪个分区。Kafka提供了多种分区策略。

策略实现方式特点适用场景
RoundRobin
轮询		依次发送到每个分区负载均衡，但可能打乱消息顺序无顺序要求的场景
Range
范围		按key的范围分配相同前缀的key进入同一分区需要局部有序
Sticky
粘性		尽可能保持上次分配减少Rebalance影响消费者频繁变化的场景
自定义实现Partitioner接口完全自定义逻辑特殊业务需求

分区策略配置

Java
// 指定分区发送
kafkaTemplate.send(topic, partition, key, message);

// 使用key的hash选择分区（默认）
kafkaTemplate.send(topic, key, message);

// 自定义分区器
props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG, CustomPartitioner.class);

// 自定义分区器实现
public class CustomPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        // 自定义分区逻辑：例如按用户ID分区
        return Math.abs(key.hashCode()) % numPartitions;
    }
}

ISR机制（副本同步）

ISR（In-Sync Replicas）同步副本集合

flowchart LR subgraph "Partition 0" L[Leader
Broker 1
读写入口] F1[Follower
Broker 2
同步数据] F2[Follower
Broker 3
同步数据] end L -.->|同步复制| F1 L -.->|同步复制| F2 ISR["ISR = [Leader, Follower1, Follower2]"] style L fill:#3a2e15,stroke:#f0b429,color:#f0b429 style ISR fill:#0d3d3a,stroke:#4ecdc4,color:#4ecdc4
概念说明
Leader主副本，负责读写操作
Follower从副本，从Leader同步数据
ISR与Leader保持同步的副本集合
AR所有副本（Assigned Replicas）
OSR落后副本（Out-of-Sync Replicas）

副本同步流程

sequenceDiagram Producer->>Leader: 发送消息 Leader->>Follower1: 复制请求 Leader->>Follower2: 复制请求 Follower1-->>Leader: 确认 Follower2-->>Leader: 确认 Leader-->>Producer: 确认（acks=all）

关键配置参数

Properties
# Broker配置
replica.lag.time.max.ms=10000    # Follower最大落后时间，超过则移出ISR
min.insync.replicas=2            # 最小ISR副本数

# Topic配置
replication.factor=3             # 副本因子

# Producer配置
acks=all                         # 等待所有ISR确认
retries=3                        # 发送失败重试次数
💡 ISR机制保证：只有当消息被ISR中的所有副本确认后，才认为消息已提交。即使Leader宕机，也能从ISR中选举新Leader，保证数据不丢失。

Kafka vs RocketMQ

特性KafkaRocketMQ
定位日志/大数据/流处理业务消息
单机吞吐量100万级10万级
消息延迟毫秒级毫秒级
可用性分布式架构主从架构

💡 选型建议：
  • Kafka：日志收集、大数据流处理、高吞吐场景、实时数据处理
  • RocketMQ：业务系统、事务消息、延迟消息、需要消息过滤的场景

最佳实践

生产者最佳实践

  • 合理设置acks（可靠性与性能平衡），业务场景建议acks=all
  • 启用重试机制，设置适当的重试次数
  • 使用批量发送提高吞吐，配合linger.ms使用
  • 开启压缩（lz4/snappy）减少网络传输
  • 异步发送提高性能，注意处理发送失败

消费者最佳实践

  • 关闭自动提交（enable.auto.commit=false）
  • 业务处理成功后再手动提交offset
  • 实现消费幂等性，防止重复消费
  • 合理设置消费者线程数，一般等于分区数
  • 监控消费延迟（lag），及时处理积压

集群最佳实践

  • 分区数 >= 消费者数，保证消费者不空闲
  • 副本数 >= 3，保证高可用
  • 合理设置min.insync.replicas
  • 定期监控消费延迟和集群健康状态
  • 为重要Topic设置数据保留策略
