分布式追踪

DistributedTracing.java

在微服务架构下，一个请求经过多个服务，需要追踪完整的调用链路，以便快速定位问题、分析性能瓶颈。

为什么需要分布式追踪

微服务架构带来了系统解耦和独立部署的优势，但也带来了新的挑战：

  • 调用链复杂：一个请求可能经过数十个服务
  • 问题定位困难：错误可能发生在任意环节
  • 性能分析困难：难以确定性能瓶颈在哪个服务
  • 日志分散：各服务日志独立存储，难以关联

调用链问题场景
  • 订单创建失败，是库存不足还是余额不足？
  • 接口响应慢，是哪个服务耗时最长？
  • 服务报错，是从哪个服务传递过来的？
  • 需要查看一次请求的完整处理过程

核心概念

Trace（追踪）

一个请求的完整调用链路，包含所有经过的服务。每个Trace有唯一的TraceId标识。

Span（跨度）

一次调用单元，表示在某个服务中的处理过程。Span包含：

  • SpanId：当前Span的唯一标识
  • ParentId：父Span的ID，用于构建调用树
  • Operation Name：操作名称（如HTTP方法+URL）
  • Start Time / Duration：开始时间和持续时间
  • Tags：标签（如http.method, http.status）
  • Logs：日志事件（如数据库查询、缓存命中）

Annotation（标注）

用于标记Span中关键事件的时间点：

  • cs (Client Send)：客户端发送请求
  • sr (Server Receive)：服务端接收请求
  • ss (Server Send)：服务端发送响应
  • cr (Client Receive)：客户端接收响应

自定义Span

自定义Span和Tag 
@Service
public class OrderService {

    @Autowired
    private Tracer tracer;

    // 方式1：编程式创建Span
    public Order getOrder(Long orderId) {
        // 创建新Span
        Span span = tracer.nextSpan().name("db-query");
        try (Tracer.SpanInScope ws = tracer.withSpan(span.start())) {
            // 添加Tag
            span.tag("db.table", "order");
            span.tag("db.orderId", orderId.toString());

            // 执行业务
            return orderRepository.findById(orderId);
        } finally {
            span.end();
        }
    }

    // 方式2：使用注解
    @NewSpan("process-order")
    public void processOrder(@SpanTag("order.id") Long orderId) {
        // 自动创建Span，参数作为Tag
        // 方法执行时间自动记录
    }
}

追踪数据采样策略

全量采样在生产环境会带来巨大存储和性能开销，需要合理的采样策略。

常见采样策略

策略 说明 适用场景
概率采样 按固定比例采样（如1%） 大流量场景
限速采样 每秒固定采样数（如100条/秒） 稳定流量场景
自适应采样 根据流量自动调整采样率 流量波动大场景
错误优先采样 错误请求100%采样 问题排查
SkyWalking采样配置 
# agent/config/agent.config
# 采样率：10000表示100%，1000表示10%
sampler.percentage.sample_rate=10000

# 可选：使用自适应采样
sampler.adaptive.enabled=true
sampler.adaptive.sample_rate=10000
sampler.adaptive.minimal_sample_rate=100
采样策略建议
  • 开发环境：100%采样，便于调试
  • 测试环境：50%-100%采样
  • 生产环境：1%-10%采样，错误请求100%采样
  • 关键路径：支付、订单等核心链路可提高采样率

最佳实践

1. TraceId传递

确保TraceId在调用链中正确传递：

TraceId传递方式 
// 1. HTTP Header传递
@Component
public class TraceFilter implements Filter {
    private static final String TRACE_ID_HEADER = "X-Trace-Id";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                        FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;

        // 从Header获取或生成TraceId
        String traceId = httpRequest.getHeader(TRACE_ID_HEADER);
        if (StringUtils.isEmpty(traceId)) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }

        // 设置到MDC（日志中打印）
        MDC.put("traceId", traceId);

        // 设置响应Header
        httpResponse.setHeader(TRACE_ID_HEADER, traceId);

        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}

// 2. Feign传递TraceId
@Bean
public RequestInterceptor traceIdInterceptor() {
    return template -> {
        String traceId = MDC.get("traceId");
        if (traceId != null) {
            template.header("X-Trace-Id", traceId);
        }
    };
}

在日志中打印TraceId，便于问题定位：

3. 监控告警

建议配置的告警项
  • 慢调用告警：P99延迟超过阈值
  • 错误率告警：错误比例超过阈值
  • 服务不可用：服务调用失败率过高
  • 采集异常：Agent连接失败或上报异常

4. 性能优化

  • 异步上报：使用队列异步发送追踪数据，避免阻塞业务
  • 批量发送：将多条Span批量打包发送，减少网络开销
  • 合理设置Span数量：避免过度细粒度的Span导致数据膨胀
  • 采样率调整：生产环境适当降低采样率
