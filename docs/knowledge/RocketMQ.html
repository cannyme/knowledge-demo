RocketMQ核心概念与使用

阿里巴巴开源的分布式消息中间件，支持事务消息、顺序消息、延迟消息等丰富特性，适用于业务系统消息通信场景

顺序消息：全局顺序 vs 分区顺序

flowchart LR subgraph "全局顺序" G1[Producer] -->|所有消息| G2[Queue 0] G2 --> G3[Consumer] end subgraph "分区顺序" P1[Producer] -->|订单A| P2[Queue 0] P1 -->|订单B| P3[Queue 1] P1 -->|订单C| P4[Queue 2] P2 --> P5[Consumer A] P3 --> P6[Consumer B] P4 --> P7[Consumer C] end style G2 fill:#5c4015,color:#f0b429,stroke:#f0b429 style P2 fill:#0d3d3a,color:#4ecdc4,stroke:#4ecdc4 style P3 fill:#0d3d3a,color:#4ecdc4,stroke:#4ecdc4 style P4 fill:#0d3d3a,color:#4ecdc4,stroke:#4ecdc4
对比项全局顺序分区顺序（推荐）
实现方式单队列，单线程消费按分区键路由到同一队列
性能差（无法并行）好（多队列并行）
适用场景强一致性要求极低订单状态变更等业务场景
分区键无需订单ID、用户ID等
💡 顺序消息场景示例：订单A的消息：创建 → 支付 → 发货 → 完成。订单A的消息必须按顺序处理，但订单A和订单B可以并行处理。

生产者使用

Java
@Service
public class OrderProducer {
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    // 同步发送
    public void sendSync(Order order) {
        SendResult result = rocketMQTemplate.syncSend(
            "order-topic",
            MessageBuilder.withPayload(order).build()
        );
        System.out.println("发送结果：" + result.getSendStatus());
    }

    // 单向发送（不关心结果，性能最高）
    public void sendOneWay(Order order) {
        rocketMQTemplate.sendOneWay("order-topic",
            MessageBuilder.withPayload(order).build());
    }
}
Java
// 异步发送
public void sendAsync(Order order) {
    rocketMQTemplate.asyncSend(
        "order-topic",
        MessageBuilder.withPayload(order).build(),
        new SendCallback() {
            @Override
            public void onSuccess(SendResult result) {
                System.out.println("发送成功");
            }
            @Override
            public void onException(Throwable e) {
                System.err.println("发送失败：" + e.getMessage());
            }
        }
    );
}
Java
// 延迟消息 - 延迟级别16，即30分钟后消费
public void sendDelayMessage(Order order) {
    Message<Order> message = MessageBuilder.withPayload(order).build();
    SendResult result = rocketMQTemplate.syncSend(
        "order-cancel-topic",
        message,
        3000,  // 超时时间
        16     // 延迟级别
    );
}
Java
// 顺序消息 - 根据订单ID选择队列，保证同一订单的消息顺序
public void sendOrderly(Order order) {
    rocketMQTemplate.syncSendOrderly(
        "order-topic",
        MessageBuilder.withPayload(order).build(),
        order.getOrderId()  // 分区键，相同值的消息进入同一队列
    );
}
Java
@Service
public class TransactionProducer {
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void sendTransactionMessage(Order order) {
        // 发送半消息
        TransactionSendResult result = rocketMQTemplate.sendMessageInTransaction(
            "order-topic",
            MessageBuilder.withPayload(order).build(),
            order  // 本地事务参数
        );
        System.out.println("事务消息发送结果：" + result.getSendStatus());
    }
}

@RocketMQTransactionListener
@Component
public class OrderTransactionListener implements RocketMQLocalTransactionListener {
    @Autowired
    private OrderService orderService;

    // 执行本地事务
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            Order order = (Order) arg;
            orderService.createOrder(order);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    // 回查本地事务状态
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        String orderId = (String) msg.getHeaders().get("orderId");
        Order order = orderService.getById(orderId);
        return order != null
            ? RocketMQLocalTransactionState.COMMIT
            : RocketMQLocalTransactionState.ROLLBACK;
    }
}

消息可靠性保障

阶段可能丢失点解决方案
生产者发送网络异常、Broker宕机同步发送+重试、多Broker发送
Broker存储写入失败、主从同步失败同步刷盘(SYNC_FLUSH)、同步复制(SYNC_MASTER)
消费者消费处理异常、ACK失败重试机制、消费确认
YAML
# 生产者配置
rocketmq:
  producer:
    retry-times-when-send-failed: 3  # 发送失败重试次数
    retry-times-when-send-async-failed: 3

# Broker配置
broker:
  flushDiskType: SYNC_FLUSH    # 同步刷盘
  brokerRole: SYNC_MASTER      # 同步复制

消费幂等

消息重复消费的原因：

  • 生产者重复发送（网络重试）
  • 消费者处理成功但ACK失败
  • Rebalance导致重复消费
Java
@Service
public class IdempotentConsumer {
    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    public void consume(String messageId, Consumer<Void> businessLogic) {
        // 使用Redis实现幂等
        String key = "mq:consumed:" + messageId;
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", 1, TimeUnit.DAYS);

        if (Boolean.TRUE.equals(success)) {
            // 首次消费，执行业务逻辑
            businessLogic.accept(null);
        } else {
            // 重复消费，直接返回
            log.info("消息已消费，跳过：{}", messageId);
        }
    }
}

Pull模式（拉取模式）由消费者主动拉取消息，适合需要精细控制消费节奏的场景。

Java
@Service
public class PullConsumerService {
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void pullMessage() {
        // 创建Pull消费者
        DefaultLitePullConsumer consumer = new DefaultLitePullConsumer("pull-consumer-group");
        consumer.setNamesrvAddr("localhost:9876");
        consumer.subscribe("order-topic", "*");
        consumer.setPullBatchSize(10);  // 每次拉取消息数

        try {
            consumer.start();

            while (true) {
                // 主动拉取消息
                List<MessageExt> messages = consumer.poll(3000);
                for (MessageExt msg : messages) {
                    System.out.println(new String(msg.getBody()));
                    // 处理完成后手动确认消费位置
                    consumer.commitSync();
                }
            }
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            consumer.shutdown();
        }
    }
}

批量消费可以一次处理多条消息，提高吞吐量。

Java
// 批量消费器
@RocketMQMessageListener(
    topic = "log-topic",
    consumerGroup = "log-batch-consumer",
    consumeMode = ConsumeMode.CONCURRENTLY,
    consumeMessageBatchMaxSize = 50  // 批量消费大小
)
public class BatchConsumer implements RocketMQListener<List<MessageExt>> {

    @Override
    public void onMessage(List<MessageExt> messages) {
        System.out.println("批次消费消息数: " + messages.size());

        // 批量处理，例如批量写入数据库
        List<LogEntry> logs = messages.stream()
            .map(msg -> parseLog(msg.getBody()))
            .collect(Collectors.toList());

        // 批量插入数据库
        logRepository.batchInsert(logs);
    }

    private LogEntry parseLog(byte[] body) {
        return JSON.parseObject(new String(body), LogEntry.class);
    }
}

顺序消费保证同一队列的消息按顺序处理。

Java
// 顺序消费：使用ORDERLY模式
@RocketMQMessageListener(
    topic = "order-status-topic",
    consumerGroup = "order-status-consumer",
    consumeMode = ConsumeMode.ORDERLY  // 顺序消费模式
)
public class OrderStatusConsumer implements RocketMQListener<MessageExt> {

    @Override
    public void onMessage(MessageExt message) {
        String orderId = message.getKeys();
        String status = new String(message.getBody());

        // 同一订单的消息会被分配到同一个队列，按顺序处理
        System.out.println("订单" + orderId + " 状态变更: " + status);
        updateOrderStatus(orderId, status);
    }
}

// 顺序消费注意事项：
// 1. 生产者发送时需要指定shardingKey（如orderId）
// 2. 消费者使用ORDERLY模式
// 3. 同一队列只能被一个消费线程处理

消费者重试策略

Java
@RocketMQMessageListener(
    topic = "order-topic",
    consumerGroup = "order-consumer",
    consumeMode = ConsumeMode.CONCURRENTLY,
    maxReconsumeTimes = 3  // 消费失败重试次数，默认16次
)
public class OrderConsumerWithRetry implements RocketMQListener<Order> {

    @Override
    public void onMessage(Order order) {
        try {
            processOrder(order);
        } catch (Exception e) {
            // 业务异常，抛出异常触发重试
            throw new RuntimeException("处理失败", e);
        }
    }
}

// 消费失败后的处理策略：
// 1. 重试消费：默认间隔10秒，最多16次
// 2. 进入死信队列：超过重试次数进入DLQ
// 3. 异步处理：记录到异步任务表，人工干预处理
