观察者模式

定义对象间一对多依赖，当主题状态改变时，所有观察者自动收到通知并更新。Spring 事件机制、MQ 消息推送、GUI 事件驱动均是观察者模式的体现。

📖

模式结构与角色

sequenceDiagram participant Client participant Subject as Subject(主题) participant ObsA as ObserverA participant ObsB as ObserverB Client->>Subject: attach(ObsA) Client->>Subject: attach(ObsB) Client->>Subject: publishEvent("新消息") Subject->>ObsA: update("新消息") Subject->>ObsB: update("新消息") Client->>Subject: detach(ObsB) Client->>Subject: publishEvent("又来了") Subject->>ObsA: update("又来了")

四个角色：

  • Subject（主题）：被观察者，维护观察者列表，提供注册/注销/通知方法
  • Observer（观察者）：定义接收通知的接口
  • ConcreteSubject：具体主题，状态变化时触发通知
  • ConcreteObserver：具体观察者，实现更新逻辑
💻

基础实现

Java
// 观察者接口
interface Observer {
    void update(String message);
}

// 主题接口
interface Subject {
    void attach(Observer observer);
    void detach(Observer observer);
    void notifyObservers(String message);
}

// 具体主题：公众号
class OfficialAccount implements Subject {
    private final List<Observer> observers = new ArrayList<>();
    private final String name;

    public OfficialAccount(String name) { this.name = name; }

    @Override public void attach(Observer o) { observers.add(o); }
    @Override public void detach(Observer o) { observers.remove(o); }

    @Override
    public void notifyObservers(String message) {
        System.out.println("[" + name + "] 发布：" + message);
        for (Observer o : observers) {
            o.update(message); // 同步通知所有观察者
        }
    }

    public void publish(String title) { notifyObservers(title); }
}

// 具体观察者：用户
class User implements Observer {
    private final String name;
    public User(String name) { this.name = name; }

    @Override
    public void update(String message) {
        System.out.println(name + " 收到推送：" + message);
    }
}

// 测试
OfficialAccount account = new OfficialAccount("技术分享");
account.attach(new User("张三"));
account.attach(new User("李四"));
account.publish("观察者模式详解");  // 两人都收到
// 输出：张三 收到推送：观察者模式详解
//       李四 收到推送：观察者模式详解

Spring 事件机制完整示例

Spring 事件机制是观察者模式的企业级实现，ApplicationContext 是事件发布者（Subject），@EventListener 是观察者。

Java - 自定义事件
// Step 1：定义事件（继承 ApplicationEvent）
public class OrderCreatedEvent extends ApplicationEvent {
    private final Long orderId;
    private final String userId;
    private final double amount;

    public OrderCreatedEvent(Object source, Long orderId, String userId, double amount) {
        super(source);
        this.orderId = orderId;
        this.userId = userId;
        this.amount = amount;
    }
    // getters...
}

// Step 2：发布事件（主题/发布者）
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    @Transactional
    public void createOrder(Long orderId, String userId, double amount) {
        // 创建订单业务逻辑...
        orderRepository.save(new Order(orderId, userId, amount));

        // 发布事件（同步，在同一个事务中）
        eventPublisher.publishEvent(new OrderCreatedEvent(this, orderId, userId, amount));
    }
}
Java - 事件监听器（观察者）
@Component
public class OrderEventHandler {

    // 方式1：基础监听
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        System.out.println("收到订单创建事件：" + event.getOrderId());
        // 发送站内通知
        notificationService.notify(event.getUserId(), "订单创建成功");
    }

    // 方式2：异步处理（不阻塞主流程，需 @EnableAsync）
    @Async
    @EventListener
    public void handleOrderCreatedAsync(OrderCreatedEvent event) {
        // 发送短信（耗时操作，异步执行）
        smsService.send(event.getUserId(), "您的订单 " + event.getOrderId() + " 已创建");
    }

    // 方式3：条件监听（只处理金额 > 1000 的订单）
    @EventListener(condition = "#event.amount > 1000")
    public void handleHighValueOrder(OrderCreatedEvent event) {
        System.out.println("大额订单预警：" + event.getOrderId());
        alertService.alert("大额订单：" + event.getAmount());
    }

    // 方式4：事务提交后才执行（最常用！避免事务回滚后还发了短信）
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handleAfterCommit(OrderCreatedEvent event) {
        // 事务成功提交后才发短信，避免事务回滚后的误操作
        smsService.send(event.getUserId(), "订单支付成功通知");
    }

    // 方式5：事务回滚时执行
    @TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
    public void handleAfterRollback(OrderCreatedEvent event) {
        log.error("订单创建失败，回滚：" + event.getOrderId());
    }
}
💡 @TransactionalEventListener 是实际项目中最常用的，它解决了"事务提交前发通知，但事务回滚后通知已发出"的问题。

面试高频考点

Q1：观察者模式 vs 发布-订阅模式的区别？
维度观察者模式发布-订阅模式
耦合度Subject 和 Observer 直接引用，有耦合通过消息中间件解耦，完全解耦
中间层无（直接调用）有 EventBus/MQ（Kafka、RocketMQ）
通信方式同步调用通常异步
典型实现Spring ApplicationEvent、Java ObservableKafka、RocketMQ、Guava EventBus
观察者模式是发布-订阅模式的一种简化版，两者都体现"松耦合"思想，但实现粒度不同。
Q2：推模型 vs 拉模型的区别？
  • 推模型（Push）：Subject 主动将变化的数据推送给 Observer（如 update(data)）。优点：Observer 无需知道 Subject；缺点：可能推送 Observer 不需要的数据
  • 拉模型（Pull）：Subject 通知 Observer 有变化，Observer 主动去取（如 update(Subject)）。优点：Observer 按需获取；缺点：Observer 需要了解 Subject 结构
  • 实践：Spring 事件机制是推模型，将完整事件对象推给监听器
Q3：观察者执行过慢会影响主流程吗？如何解决？
同步调用时会阻塞主流程。解决方案：
  • @Async：Spring 异步执行，Observer 在线程池中异步运行
  • 线程池：Subject 通知时用 ExecutorService 异步提交
  • 消息队列：将事件发到 MQ（Kafka/RocketMQ），消费者异步处理，彻底解耦
  • @TransactionalEventListener：事务提交后异步执行，不影响主事务
⚖️

优缺点分析

✅ 优点

  • 主题与观察者松耦合
  • 支持广播通信（一对多）
  • 符合开闭原则，可动态增删观察者
  • 事件驱动架构的基础

❌ 缺点

  • 同步通知可能阻塞主流程
  • 观察者过多时通知耗时长
  • 循环依赖可能导致死循环
  • 执行顺序不确定

手写轻量级 EventBus

理解观察者模式最好的方式是自己实现一个事件总线，这也是 Guava EventBus、Spring ApplicationEventPublisher 的核心原理。

Java
// 事件基类
public abstract class Event {}

// 具体事件
public class UserRegisteredEvent extends Event {
    private final String userId;
    private final String email;
    public UserRegisteredEvent(String userId, String email) {
        this.userId = userId;
        this.email = email;
    }
    public String getUserId() { return userId; }
    public String getEmail() { return email; }
}

// 事件监听器接口（函数式接口，支持 Lambda）
@FunctionalInterface
public interface EventListener {
    void onEvent(T event);
}

// 轻量级 EventBus
public class EventBus {
    // key: 事件类型，value: 监听器列表
    private final Map, List>> listeners =
        new ConcurrentHashMap<>();

    // 订阅
    @SuppressWarnings("unchecked")
    public  void subscribe(Class eventType, EventListener listener) {
        listeners.computeIfAbsent(eventType, k -> new CopyOnWriteArrayList<>()).add(listener);
    }

    // 取消订阅
    public  void unsubscribe(Class eventType, EventListener listener) {
        List> list = listeners.get(eventType);
        if (list != null) list.remove(listener);
    }

    // 发布事件（同步）
    @SuppressWarnings({"unchecked", "rawtypes"})
    public  void publish(T event) {
        List> list = listeners.get(event.getClass());
        if (list != null) {
            for (EventListener listener : list) {
                listener.onEvent(event); // 同步依次调用
            }
        }
    }
}

// 使用示例
EventBus bus = new EventBus();

// Lambda 订阅（函数式接口优势）
bus.subscribe(UserRegisteredEvent.class, e ->
    System.out.println("发送欢迎邮件：" + e.getEmail()));

bus.subscribe(UserRegisteredEvent.class, e ->
    System.out.println("发送短信验证码：" + e.getUserId()));

// 发布事件
bus.publish(new UserRegisteredEvent("u001", "test@example.com"));
// 输出：发送欢迎邮件：test@example.com
//       发送短信验证码：u001
⚠️

实战陷阱

陷阱1：忘记取消订阅导致内存泄漏
Observer 持有 Subject 引用，Subject 持有 Observer 引用。若 Observer 生命周期结束但未调用 detach()，Subject 仍持有其引用，GC 无法回收（内存泄漏）。使用 WeakReference 或 Java Observable 时尤其注意。
陷阱2：事务提交前触发事件，事务回滚后消息已发
Java
// ❌ 错误：事务内发布事件，可能事务回滚但短信已发
@Transactional
public void createOrder(Order order) {
    orderRepo.save(order);
    eventPublisher.publishEvent(new OrderCreatedEvent(order)); // 同步触发，立即发短信
    // 假设后续抛异常，事务回滚，但短信已发出！
    throw new RuntimeException("回滚测试");
}

// ✅ 正确：用 @TransactionalEventListener 确保事务提交后才执行
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void sendSms(OrderCreatedEvent event) {
    smsService.send(event.getUserId(), "订单创建成功"); // 只有事务提交了才执行
}
最佳实践：观察者模式 + MQ = 分布式事件驱动
单体应用内用 Spring 事件机制；跨服务用 Kafka/RocketMQ。两者都是观察者模式思想的体现，只是中间层从内存 List 变成了消息队列。

与其他模式的关系

相关模式关系说明
中介者模式中介者集中处理对象间通信（多对多→一对多）；观察者是发布者对订阅者的一对多广播
责任链模式责任链中请求只被一个处理者处理；观察者通知所有观察者
