观察者模式

定义对象间的一对多依赖关系，当一个对象状态改变时，所有依赖它的对象都会收到通知并自动更新。观察者模式是事件驱动编程的基础。

📖

模式定义与意图

定义：观察者模式（Observer Pattern）是一种行为型设计模式，它定义了对象之间的一对多依赖关系，当一个对象状态发生改变时，所有依赖于它的对象都得到通知并被自动更新。

核心思想：将观察者和被观察者解耦，被观察者（主题）维护一个观察者列表，状态变化时主动通知所有观察者。

角色说明：

  • Subject（主题/被观察者）：维护观察者列表，提供注册和注销方法
  • Observer（观察者）：定义更新接口，接收主题的通知
  • ConcreteSubject（具体主题）：实现主题接口，维护具体状态
  • ConcreteObserver（具体观察者）：实现观察者接口，更新自身状态
🎯

适用场景

以下场景适合使用观察者模式：

  • 事件驱动系统：GUI事件处理（按钮点击、鼠标移动）
  • 消息推送：公众号文章推送、群聊消息通知
  • 日志监控：日志变更时触发告警通知
  • 数据同步：缓存更新通知、数据库变更同步
  • Spring事件机制：ApplicationEvent事件发布与监听
  • 响应式编程：RxJava、Reactor中的发布-订阅模型

代码实现

基础实现：公众号订阅系统示例

Java
// 观察者接口
interface Observer {
    void update(String message);
}

// 主题接口
interface Subject {
    void attach(Observer observer);
    void detach(Observer observer);
    void notifyObservers(String message);
}

// 具体主题：公众号
class OfficialAccount implements Subject {
    private final List<Observer> observers = new ArrayList<>();
    private final String name;

    public OfficialAccount(String name) {
        this.name = name;
    }

    @Override
    public void attach(Observer observer) {
        observers.add(observer);
        System.out.println("新用户关注了：" + name);
    }

    @Override
    public void detach(Observer observer) {
        observers.remove(observer);
        System.out.println("用户取关了：" + name);
    }

    @Override
    public void notifyObservers(String message) {
        System.out.println("\n=== " + name + " 发布新文章 ===");
        for (Observer observer : observers) {
            observer.update(message);
        }
    }

    public void publishArticle(String title) {
        notifyObservers(title);
    }
}

// 具体观察者：用户
class User implements Observer {
    private final String name;

    public User(String name) {
        this.name = name;
    }

    @Override
    public void update(String message) {
        System.out.println(name + " 收到推送：" + message);
    }
}

// 使用
OfficialAccount account = new OfficialAccount("技术分享");
User user1 = new User("张三");
User user2 = new User("李四");

account.attach(user1);
account.attach(user2);
account.publishArticle("观察者模式详解");

account.detach(user2);
account.publishArticle("Spring事件机制");

推模型 vs 拉模型

Java
// ============ 推模型：主题主动推送数据 ============
interface PushObserver {
    void update(String message, Object data);
}

// 推模型实现
class PushUser implements PushObserver {
    private final String name;

    public PushUser(String name) {
        this.name = name;
    }

    @Override
    public void update(String message, Object data) {
        // 直接接收主题推送的数据
        System.out.println(name + " 收到推送：" + message + " 数据：" + data);
    }
}

// ============ 拉模型：观察者主动获取数据 ============
interface PullObserver {
    void update(Subject subject);
}

// 拉模型实现
class PullUser implements PullObserver {
    private final String name;

    public PullUser(String name) {
        this.name = name;
    }

    @Override
    public void update(Subject subject) {
        // 观察者主动从主题获取需要的数据
        // String data = ((OfficialAccount) subject).getLatestArticle();
        System.out.println(name + " 主动获取数据");
    }
}
特性 推模型 拉模型
数据传递 主题主动推送完整数据 只通知状态变化，观察者按需获取
灵活性 观察者被动接收，不够灵活 观察者按需获取，更灵活
性能 可能推送无用数据 减少不必要的数据传输

Spring事件机制 - 观察者模式的典型应用

Java
// 1. 定义自定义事件
public class OrderCreatedEvent extends ApplicationEvent {
    private final Long orderId;
    private final String userId;

    public OrderCreatedEvent(Object source, Long orderId, String userId) {
        super(source);
        this.orderId = orderId;
        this.userId = userId;
    }

    public Long getOrderId() { return orderId; }
    public String getUserId() { return userId; }
}

// 2. 方式一：实现ApplicationListener接口
@Component
public class OrderEventListener implements ApplicationListener<OrderCreatedEvent> {
    @Override
    public void onApplicationEvent(OrderCreatedEvent event) {
        System.out.println("收到订单创建事件：" + event.getOrderId());
    }
}

// 3. 方式二：使用@EventListener注解（推荐）
@Component
public class OrderEventHandler {

    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        System.out.println("处理订单：" + event.getOrderId());
    }

    // 异步处理
    @Async
    @EventListener
    public void handleAsync(OrderCreatedEvent event) {
        System.out.println("异步处理订单：" + event.getOrderId());
    }

    // 条件监听
    @EventListener(condition = "#event.orderId > 1000")
    public void handleHighValueOrder(OrderCreatedEvent event) {
        System.out.println("处理大额订单：" + event.getOrderId());
    }

    // 事务提交后监听
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handleAfterCommit(OrderCreatedEvent event) {
        System.out.println("事务提交后处理：" + event.getOrderId());
    }
}

// 4. 发布事件
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    public void createOrder(Long orderId, String userId) {
        // 创建订单逻辑...

        // 发布事件
        eventPublisher.publishEvent(new OrderCreatedEvent(this, orderId, userId));
    }
}

简易EventBus实现

Java
// 简易事件总线
public class EventBus {
    private final Map<Class<?>, List<Consumer<?>>> listeners = new HashMap<>();

    public <T> void register(Class<T> eventType, Consumer<T> listener) {
        listeners.computeIfAbsent(eventType, k -> new ArrayList<>()).add(listener);
    }

    @SuppressWarnings("unchecked")
    public <T> void post(T event) {
        List<Consumer<?>> consumers = listeners.get(event.getClass());
        if (consumers != null) {
            for (Consumer<?> consumer : consumers) {
                ((Consumer<T>) consumer).accept(event);
            }
        }
    }
}

// 使用
EventBus eventBus = new EventBus();
eventBus.register(String.class, msg -> System.out.println("收到消息：" + msg));
eventBus.register(Integer.class, num -> System.out.println("收到数字：" + num));

eventBus.post("Hello World");
eventBus.post(123);

⚖️

优缺点分析

优点

  • 松耦合：观察者和主题之间解耦，可以独立变化
  • 开闭原则：新增观察者无需修改主题代码
  • 广播通信：支持一对多的消息广播
  • 动态关系：可以在运行时添加或删除观察者

缺点

  • 循环依赖风险：观察者和主题可能形成循环依赖
  • 通知顺序不确定：观察者接收通知的顺序不确定
  • 性能问题：观察者过多时，通知开销大
  • 内存泄漏：忘记移除观察者可能导致内存泄漏

与其他模式的关系

相关模式 关系说明
中介者模式 观察者模式通过观察者通信，中介者模式通过中介者协调通信
单例模式 事件总线通常设计为单例，确保全局唯一
发布-订阅模式 发布-订阅是观察者模式的变体，引入事件总线作为中间层
MVC模式 MVC中的模型变更通知视图更新使用观察者模式

观察者模式 vs 发布-订阅模式：观察者模式中观察者和主题直接通信，发布-订阅模式引入事件通道作为中间层，发布者和订阅者完全解耦。
