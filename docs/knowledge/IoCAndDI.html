Spring IoC与DI

控制反转(IoC)是Spring框架的核心，依赖注入(DI)是IoC的具体实现方式，实现对象解耦和生命周期管理

📖

核心概念

IoC（控制反转）：将对象创建和管理的控制权从程序代码转移到Spring容器。

  • 传统方式：对象自己创建和管理依赖（new）
  • IoC方式：由容器创建和管理对象，对象被动接收依赖

DI（依赖注入）：IoC的具体实现方式，容器负责将依赖注入到对象中。

IoC容器特点适用场景
BeanFactory基础容器，延迟加载资源受限环境
ApplicationContext高级容器，立即加载，支持AOP、国际化标准应用（推荐）
🏷️

Bean的注册方式

方式1：@Component 及其衍生注解

java
@Component      // 通用组件
@Repository     // 数据访问层
@Service        // 业务层
@Controller     // 控制层
@Configuration  // 配置类

方式2：@Configuration + @Bean

适用于第三方库的类、需要复杂初始化逻辑的场景。

java
@Configuration
public class AppConfig {
    @Bean(initMethod = "init", destroyMethod = "destroy")
    public UserService userService() {
        return new UserService();
    }
}

方式3：@Import

快速导入配置类或组件。

java
@Import({AppConfig.class, UserService.class})
public class MainConfig {
}

依赖注入方式对比

构造器注入（推荐）✅

💡
优点：依赖明确、不可变（final）、便于单元测试、确保对象完整、循环依赖启动时报错
java
@Service
public class UserService {
    private final UserRepository userRepository;

    // 单构造器可省略@Autowired
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

Setter注入

java
@Service
public class UserService {
    private UserRepository userRepository;

    @Autowired
    public void setUserRepository(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

字段注入（不推荐）❌

⚠️
缺点：无法设置final、单元测试困难、依赖不透明、容易隐藏循环依赖

循环依赖问题

问题：A依赖B，B依赖A，形成循环。

解决方案

方案说明
构造器注入无法解决，启动时报错（推荐，强制避免循环）
Setter/字段注入Spring通过三级缓存解决
@Lazy延迟加载打破循环
重构设计最佳方案，避免循环依赖
java
@Service
public class ServiceA {
    public ServiceA(@Lazy ServiceB serviceB) {
        // 使用@Lazy延迟注入
    }
}
💡
三级缓存原理：
  • singletonObjects：完整的Bean
  • earlySingletonObjects：提前暴露的Bean（未完成属性注入）
  • singletonFactories：Bean工厂，用于生成代理对象

自动装配规则

@Autowired 装配规则：

  1. 首先按类型查找
  2. 如果有多个，按名称匹配
  3. 如果仍无法确定，抛出异常

解决多实现类问题

java
// 方式1：@Primary 设置默认实现
@Service
@Primary
public class AlipayService implements PaymentService {}

// 方式2：@Qualifier 指定名称
public PaymentController(@Qualifier("wechatPay") PaymentService service) {}

// 方式3：注入所有实现

BeanDefinition与BeanPostProcessor

BeanDefinition 是 Spring 中描述 Bean 元数据的核心接口，包含 class、scope、lazyInit、initMethod、dependsOn 等信息。

java
// 自定义BeanPostProcessor示例（Spring AOP底层原理）
@Component
public class MyBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        // Bean初始化前：可修改或替换Bean
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // Bean初始化后：AOP代理在此生成
        if (bean instanceof UserService) {
            return Proxy.newProxyInstance(...); // 返回代理对象
        }
        return bean;
    }
}

// ApplicationContext事件监听
@Component
public class AppStartListener implements ApplicationListener<ContextRefreshedEvent> {
    @Override
    public void onApplicationEvent(ContextRefreshedEvent event) {
        System.out.println("Spring容器启动完成，Bean总数: " +
            event.getApplicationContext().getBeanDefinitionCount());
    }
}
💡
BeanFactoryPostProcessor vs BeanPostProcessor BeanFactoryPostProcessor 在 BeanDefinition 注册后、Bean 实例化前执行（修改元数据）；BeanPostProcessor 在 Bean 实例化后、初始化前后执行（修改实例）。

面试高频考点

Q1: IoC 和 DI 的区别是什么？Spring 如何实现 IoC？
IoC（控制反转）是设计思想，将对象创建和依赖管理的控制权从应用代码转移到容器；DI（依赖注入）是 IoC 的具体实现方式，容器在运行时将依赖对象注入到目标对象。Spring 通过 BeanFactory/ApplicationContext 容器实现 IoC，底层基于反射机制创建对象并注入依赖。
Q2: @Autowired 和 @Resource 的区别？
@Autowired 是 Spring 注解，先按类型匹配，有多个时按名称匹配，可配合 @Qualifier 指定 Bean 名称；@Resource 是 Java 标准注解（JSR-250），先按名称匹配，找不到再按类型匹配。推荐使用 @Autowired + @Qualifier 组合，语义更清晰。
Q3: Spring 三级缓存如何解决循环依赖？
三级缓存：①singletonObjects（完整Bean）②earlySingletonObjects（早期暴露Bean，未完成属性注入）③singletonFactories（Bean工厂，生成早期引用）。A依赖B时：A实例化→放入三级缓存→注入B→B实例化→从三级缓存获取A的早期引用→B初始化完成→A完成注入→A初始化完成。注意：构造器注入无法解决循环依赖。
Q4: singleton 单例 Bean 是线程安全的吗？
不一定安全。singleton 只保证容器中只有一个实例，若 Bean 中有成员变量（状态），多线程并发修改会有线程安全问题。解决方案：①无状态设计（推荐）②使用 ThreadLocal 保存线程状态③改用 prototype 作用域④使用同步机制。
⚡

实战陷阱

⚠️
陷阱1：prototype Bean 注入到 singleton Bean 失效 singleton Bean 只初始化一次，其中注入的 prototype Bean 也只创建一次，不会每次请求都创建新实例。解决方案：注入 ApplicationContext，每次调用 getBean()；或使用 @Lookup 方法注入；或使用 ObjectProvider<PrototypeBean>。
⚠️
陷阱2：@Autowired 字段注入导致 NPE 使用字段注入时，若在构造函数中使用注入的字段，会得到 null（Spring 先调用构造函数，再注入字段）。改用构造器注入可彻底避免此问题。
⚠️
陷阱3：@PostConstruct 中调用事务方法无效 @PostConstruct 在 BeanPostProcessor 之前执行，此时事务代理尚未生成，直接调用会绕过事务。应将初始化逻辑移至 ApplicationListener<ContextRefreshedEvent> 中，此时代理已生成。
