单例模式

确保一个类只有一个实例，并提供一个全局访问点。广泛应用于配置管理、数据库连接池、日志记录等场景，是面试必考的高频知识点。

📖

模式定义与意图

定义：单例模式（Singleton Pattern）是一种创建型设计模式，确保一个类只有一个实例，并提供全局访问点。

核心思想：私有化构造方法，控制实例创建过程。

classDiagram class Singleton { -instance: Singleton$ -Singleton() +getInstance()$ Singleton +doSomething() void } note for Singleton "私有构造器\n静态实例引用\n全局访问点"

三个关键要素：

  • 私有化构造器：防止外部通过 new 创建实例
  • 静态实例引用：持有唯一实例的静态字段
  • 全局访问点：公共静态方法 getInstance()
🎯

适用场景

  • 配置管理器：应用配置信息只需加载一次，全局共享
  • 数据库连接池：管理有限的数据库连接资源（HikariCP、Druid）
  • 日志记录器：统一处理日志输出，避免重复创建
  • 缓存管理器：全局缓存数据的统一管理
  • 线程池：Executors 工厂方法返回的线程池通常全局唯一
  • Spring Bean：默认 scope="singleton"，容器内全局唯一
💻

五种实现方式

饿汉式（Eager Initialization） — 类加载时创建，线程安全，不支持懒加载

Java
public class EagerSingleton {
    // 类加载时立即初始化，JVM 保证线程安全
    private static final EagerSingleton INSTANCE = new EagerSingleton();

    // 私有构造器，防止外部实例化
    private EagerSingleton() {}

    public static EagerSingleton getInstance() {
        return INSTANCE;
    }

    public void doSomething() {
        System.out.println("单例方法执行");
    }
}
⚠️ 注意
不管是否用到，类加载时就创建实例。如果实例初始化代价很大且可能不被使用，会浪费资源。

懒汉式（Lazy Initialization） — 支持懒加载，但线程不安全（不推荐单独使用）

Java
public class LazySingleton {
    private static LazySingleton instance;

    private LazySingleton() {}

    // ❌ 线程不安全：多线程可能同时进入 if 判断，创建多个实例
    public static LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }

    // ✅ 线程安全版（同步方法）— 但每次调用都加锁，性能差
    public static synchronized LazySingleton getInstanceSafe() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}

双重检查锁（Double Checked Locking） — 线程安全 + 懒加载 + 高性能，推荐

Java
public class DCLSingleton {
    // volatile 关键字：禁止指令重排序 + 保证可见性
    private static volatile DCLSingleton instance;

    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        // 第一次检查：instance 已存在则直接返回，无需加锁（性能优化）
        if (instance == null) {
            synchronized (DCLSingleton.class) {
                // 第二次检查：防止多线程同时通过第一次检查
                if (instance == null) {
                    instance = new DCLSingleton(); // 非原子操作！
                }
            }
        }
        return instance;
    }
}

为什么 volatile 必不可少？

instance = new DCLSingleton() 在字节码层面分为三步：

  1. memory = allocate() — 分配内存空间
  2. initObject(memory) — 初始化对象
  3. instance = memory — 引用指向内存地址

JIT 编译器可能将步骤 2 和 3 重排序，导致另一个线程拿到未初始化完成的对象（instance 非 null 但对象字段未初始化）。volatile 禁止这种重排序。

静态内部类（Static Inner Class） — 最推荐的实现方式，优雅且高效 强烈推荐

Java
public class StaticInnerSingleton {
    private StaticInnerSingleton() {}

    // 静态内部类：只有在被引用时才会被 JVM 加载（懒加载）
    private static class Holder {
        // JVM 保证类初始化是线程安全的（CLInitLock）
        private static final StaticInnerSingleton INSTANCE = new StaticInnerSingleton();
    }

    public static StaticInnerSingleton getInstance() {
        return Holder.INSTANCE; // 触发 Holder 类加载，仅此一次
    }
}

// 使用示例
StaticInnerSingleton s1 = StaticInnerSingleton.getInstance();
StaticInnerSingleton s2 = StaticInnerSingleton.getInstance();
System.out.println(s1 == s2); // true

实现原理：利用 JVM 类初始化机制（Class Loading）保证线程安全，外部类加载时 Holder 不会被加载，只有首次调用 getInstance() 才触发 Holder 加载，且 JVM 用锁保证只初始化一次。

枚举（Enum） — 最安全的实现，Effective Java 第三版推荐 最安全

Java
public enum EnumSingleton {
    INSTANCE;

    // 可以添加状态和方法
    private int count = 0;

    public void doSomething() {
        count++;
        System.out.println("单例执行第 " + count + " 次");
    }

    public int getCount() {
        return count;
    }
}

// 使用
EnumSingleton.INSTANCE.doSomething();
EnumSingleton.INSTANCE.doSomething();
System.out.println(EnumSingleton.INSTANCE.getCount()); // 2

枚举的三大优势：

  • 防反射攻击：枚举构造器私有且 JVM 禁止通过反射调用枚举构造器，抛出 IllegalArgumentException
  • 防序列化破坏：枚举序列化时只写入枚举名称，反序列化时通过 Enum.valueOf() 返回已有实例
  • 线程安全：枚举常量在类加载时初始化，JVM 保证线程安全

面试高频考点

Q1：DCL 中 volatile 为什么必须加？不加会有什么问题？
new 操作分三步：①分配内存 ②初始化对象 ③引用赋值。JIT 可能将②③重排序为①③②。线程A刚执行③（引用非null但对象未初始化），线程B通过第一次 null 检查，直接使用了一个"半成品"对象，导致 NPE 或数据异常。volatile 通过内存屏障禁止这种重排序。
Q2：序列化会破坏单例吗？如何解决？
会破坏。对象序列化后再反序列化，会通过 ObjectInputStream 创建新实例，不走构造器。

解决方案：在单例类中添加 readResolve() 方法：
Java
// 反序列化时 JVM 会调用此方法，返回值替代新建的对象
protected Object readResolve() throws ObjectStreamException {
    return getInstance(); // 返回现有单例
}
枚举天然防止序列化破坏，无需额外处理。
Q3：反射能破坏单例吗？如何防止？
能。通过反射可以绕过私有构造器：Constructor.setAccessible(true)。

防御方案：在构造器中检查实例是否已存在：
Java
private SafeSingleton() {
    // 防止反射攻击
    if (instance != null) {
        throw new RuntimeException("禁止通过反射创建单例实例");
    }
}
枚举是最彻底的防御：JVM 源码层面禁止反射调用枚举构造器。
Q4：单例模式 vs Spring @Scope("singleton") 的区别？

AOP 支持不支持自动代理

实战陷阱

陷阱1：多 ClassLoader 环境导致单例失效
在 Web 容器（Tomcat）、OSGi、热部署等场景下，同一个类可能被不同 ClassLoader 加载多次，产生多个"单例"。
解决：使用同一个 ClassLoader 加载，或通过父类委派机制（Bootstrap ClassLoader）。
陷阱2：集群/分布式环境下单例无效
单例只在单个 JVM 进程内唯一。多台服务器部署时，每台机器各有一个实例，不是全局唯一。
解决：使用分布式锁（Redis、Zookeeper）+ 共享存储实现分布式单例。
陷阱3：单例持有有状态对象的线程安全问题
单例本身是全局共享的，如果单例内部持有可变状态（如 HashMap），多线程并发访问时需要加锁，否则会有线程安全问题。
Java
// ❌ 危险：单例内部的 HashMap 是共享可变状态
public class CacheManager {
    private static final CacheManager INSTANCE = new CacheManager();
    private Map<String, Object> cache = new HashMap<>(); // 线程不安全！

    // ✅ 正确：使用线程安全的 ConcurrentHashMap
    private Map<String, Object> safeCache = new ConcurrentHashMap<>();
}
⚖️

优缺点分析

✅ 优点

  • 全局唯一：避免重复创建，节省资源
  • 统一访问：全局访问点，方便管理
  • 懒加载：静态内部类等方式按需创建
  • 全局状态：可以维护跨请求的全局状态

❌ 缺点

  • 违反 SRP：既管理实例又执行业务
  • 隐藏依赖：全局访问使依赖关系不透明
  • 难以测试：全局状态污染单元测试
  • 并发隐患：有状态单例需要额外同步
