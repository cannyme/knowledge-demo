Nacos与Apollo配置中心

Nacos和Apollo是微服务架构中主流的注册中心和配置中心解决方案，提供服务发现、动态配置管理、服务治理等核心能力。

❓

为什么需要配置中心

在传统应用中，配置通常写在代码或配置文件中，但随着微服务架构的演进，配置管理面临诸多挑战：

  • 配置分散：数百个微服务的配置分散在各处，难以统一管理
  • 环境隔离：开发、测试、生产环境需要不同的配置
  • 动态更新：修改配置需要重新打包部署，效率低下
  • 版本管理：配置变更历史难以追溯，回滚困难
  • 实时生效：需要配置变更后立即生效，无需重启服务
  • 灰度发布：配置变更需要支持灰度验证

Nacos

Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的服务发现和配置管理平台，提供一站式解决方案。

架构设计

配置管理] F[Core Module
核心模块] end G[(存储
MySQL/内置)] A -->|注册/心跳| D B -->|服务发现| D C -->|配置管理| E D --> G E --> G

服务注册与发现

sequenceDiagram participant P as Provider participant N as Nacos Server participant C as Consumer P->>N: 注册服务(IP, Port, ServiceName) N-->>P: 注册成功 loop 每5秒 P->>N: 发送心跳 end C->>N: 订阅服务列表 N-->>C: 返回实例列表 Note over N,C: 实例变更时主动推送

健康检查机制

模式说明适用场景
临时实例客户端主动心跳，15秒无心跳标记不健康，30秒剔除微服务
持久实例服务端主动探测（TCP/HTTP）数据库、缓存等基础设施

配置管理模型

graph TB subgraph "Namespace" subgraph "Group" A[Data ID: application.yml] B[Data ID: redis.yml] C[Data ID: mysql.yml] end end
  • Namespace：命名空间，用于环境隔离（dev/test/prod）
  • Group：配置分组，默认DEFAULT_GROUP
  • Data ID：配置文件名，格式：${spring.application.name}-${profile}.${file-extension}

Nacos配置示例

YAML
spring:
  application:
    name: order-service
  cloud:
    nacos:
      # 注册中心配置
      discovery:
        server-addr: 127.0.0.1:8848
        namespace: dev
        group: DEFAULT_GROUP
        metadata:
          version: 1.0.0
          region: cn-hangzhou
      # 配置中心配置
      config:
        server-addr: 127.0.0.1:8848
        namespace: dev
        group: DEFAULT_GROUP
        file-extension: yaml
        # 共享配置
        shared-configs:
          - data-id: common.yaml
            refresh: true
        # 扩展配置
        extension-configs:
          - data-id: redis.yaml
            refresh: true

动态配置刷新

Java
@RefreshScope  // 支持配置动态刷新
@Service
public class ConfigService {

    @Value("${app.config.timeout:3000}")
    private int timeout;

    @Value("${app.config.feature.enabled:false}")
    private boolean featureEnabled;

    public void printConfig() {
        System.out.println("timeout: " + timeout);
        System.out.println("featureEnabled: " + featureEnabled);
    }
}

Apollo

Apollo是携程开源的分布式配置中心，专注于配置管理领域，功能比Nacos更加专业和完善。

架构设计

graph TB A[Portal
管理界面] -->|管理配置| B[Admin Service] B -->|存储配置| C[(Config DB)] D[Config Service] -->|读取配置| C E[Client] -->|长轮询| D B -->|推送变更| D D -->|推送变更| E

核心组件

组件说明
Portal配置管理界面，提供Web端配置操作
Admin Service配置管理服务，提供配置的增删改查接口
Config Service配置读取服务，客户端通过它获取配置
Client客户端SDK，嵌入应用程序中

核心概念

概念说明
AppId应用唯一标识
Environment环境（DEV/FAT/UAT/PRO）
Cluster集群，支持多数据中心
Namespace命名空间（配置文件）
Release配置发布版本

命名空间类型

类型说明
私有Namespace只有当前应用可使用
公共Namespace多应用共享，如公共数据库配置
关联Namespace继承公共Namespace，可覆盖配置

Apollo配置示例

Properties
# Apollo应用ID
app.id=order-service
# Apollo Meta Server地址
apollo.meta=http://127.0.0.1:8080
# 启用的命名空间
apollo.bootstrap.enabled=true
apollo.bootstrap.namespaces=application,common-db
# 集群名称
apollo.cluster=default

配置变更监听

Java
@Component
public class ApolloConfigChangeListener {

    @ApolloConfigChangeListener
    public void onChange(ConfigChangeEvent changeEvent) {
        for (String key : changeEvent.changedKeys()) {
            ConfigChange change = changeEvent.getChange(key);
            System.out.println(String.format(
                "配置变更: %s, 旧值: %s, 新值: %s",
                key, change.getOldValue(), change.getNewValue()
            ));
        }
    }
}

灰度发布

flowchart LR A[修改配置] --> B[创建灰度] B --> C[选择灰度IP] C --> D[灰度发布] D --> E{验证结果} E -->|成功| F[全量发布] E -->|失败| G[放弃灰度]

Apollo支持按IP列表或应用实例进行灰度发布，只有灰度机器会收到新配置，验证通过后再全量发布。

💡 选型建议：
- 需要同时使用注册中心和配置中心 → 选择Nacos
- 已有Eureka/Nacos做注册中心，需要专业配置中心 → 选择Apollo
- 需要灰度发布能力 → 选择Apollo
- 追求部署简单、运维成本低 → 选择Nacos

最佳实践

配置分类建议

  • 按环境隔离：DEV/FAT/UAT/PRO使用不同Namespace
  • 按应用隔离：每个微服务有自己的配置
  • 按模块隔离：db.yml/redis.yml/mq.yml分开管理

敏感配置处理

  • 数据库密码、API密钥等敏感信息加密存储
  • 生产环境配置只允许特定人员修改
  • 记录所有配置变更审计日志

配置变更规范

  1. 变更前：通知相关人员
  2. 变更中：灰度发布（Apollo）或小范围验证
  3. 变更后：验证配置生效
  4. 异常时：快速回滚
