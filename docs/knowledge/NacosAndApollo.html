Nacos与Apollo配置中心

Nacos和Apollo是微服务架构中主流的注册中心和配置中心解决方案，提供服务发现、动态配置管理、服务治理等核心能力。

❓

为什么需要配置中心

在传统应用中，配置通常写在代码或配置文件中，但随着微服务架构的演进，配置管理面临诸多挑战：

  • 配置分散：数百个微服务的配置分散在各处，难以统一管理
  • 环境隔离：开发、测试、生产环境需要不同的配置
  • 动态更新：修改配置需要重新打包部署，效率低下
  • 版本管理：配置变更历史难以追溯，回滚困难
  • 实时生效：需要配置变更后立即生效，无需重启服务
  • 灰度发布：配置变更需要支持灰度验证

Nacos

Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的服务发现和配置管理平台，提供一站式解决方案。

服务注册与发现

健康检查机制

模式说明适用场景
临时实例客户端主动心跳，15秒无心跳标记不健康，30秒剔除微服务
持久实例服务端主动探测（TCP/HTTP）数据库、缓存等基础设施

配置管理模型

graph TB subgraph "Namespace" subgraph "Group" A[Data ID: application.yml] B[Data ID: redis.yml] C[Data ID: mysql.yml] end end
  • Namespace：命名空间，用于环境隔离（dev/test/prod）
  • Group：配置分组，默认DEFAULT_GROUP
  • Data ID：配置文件名，格式：${spring.application.name}-${profile}.${file-extension}

Nacos配置示例

yaml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      # 注册中心配置
      discovery:
        server-addr: 127.0.0.1:8848
        namespace: dev
        group: DEFAULT_GROUP
        metadata:
          version: 1.0.0
          region: cn-hangzhou
      # 配置中心配置
      config:
        server-addr: 127.0.0.1:8848
        namespace: dev
        group: DEFAULT_GROUP
        file-extension: yaml
        # 共享配置
        shared-configs:
          - data-id: common.yaml
            refresh: true
        # 扩展配置
        extension-configs:
          - data-id: redis.yaml
            refresh: true

动态配置刷新

java
@RefreshScope  // 支持配置动态刷新
@Service
public class ConfigService {

    @Value("${app.config.timeout:3000}")
    private int timeout;

    @Value("${app.config.feature.enabled:false}")
    private boolean featureEnabled;

    public void printConfig() {
        System.out.println("timeout: " + timeout);
        System.out.println("featureEnabled: " + featureEnabled);
    }
}

Apollo

Apollo是携程开源的分布式配置中心，专注于配置管理领域，功能比Nacos更加专业和完善。

架构设计

graph TB A[Portal
管理界面] -->|管理配置| B[Admin Service] B -->|存储配置| C[(Config DB)] D[Config Service] -->|读取配置| C E[Client] -->|长轮询| D B -->|推送变更| D D -->|推送变更| E

核心组件

组件说明
Portal配置管理界面，提供Web端配置操作
Admin Service配置管理服务，提供配置的增删改查接口
Config Service配置读取服务，客户端通过它获取配置
Client客户端SDK，嵌入应用程序中

核心概念

概念说明
AppId应用唯一标识
Environment环境（DEV/FAT/UAT/PRO）
Cluster集群，支持多数据中心
Namespace命名空间（配置文件）
Release配置发布版本

命名空间类型

类型说明
私有Namespace只有当前应用可使用
公共Namespace多应用共享，如公共数据库配置
关联Namespace继承公共Namespace，可覆盖配置

Apollo配置示例

yaml
# Apollo应用ID
app.id=order-service
# Apollo Meta Server地址
apollo.meta=http://127.0.0.1:8080
# 启用的命名空间
apollo.bootstrap.enabled=true
apollo.bootstrap.namespaces=application,common-db
# 集群名称
apollo.cluster=default

配置变更监听

java
@Component
public class ApolloConfigChangeListener {

    @ApolloConfigChangeListener
    public void onChange(ConfigChangeEvent changeEvent) {
        for (String key : changeEvent.changedKeys()) {
            ConfigChange change = changeEvent.getChange(key);
            System.out.println(String.format(
                "配置变更: %s, 旧值: %s, 新值: %s",
                key, change.getOldValue(), change.getNewValue()
            ));
        }
    }
}

最佳实践

配置分类建议

  • 按环境隔离：DEV/FAT/UAT/PRO使用不同Namespace
  • 按应用隔离：每个微服务有自己的配置
  • 按模块隔离：db.yml/redis.yml/mq.yml分开管理

敏感配置处理

  • 数据库密码、API密钥等敏感信息加密存储
  • 生产环境配置只允许特定人员修改
  • 记录所有配置变更审计日志

配置变更规范

  1. 变更前：通知相关人员
  2. 变更中：灰度发布（Apollo）或小范围验证
  3. 变更后：验证配置生效
  4. 异常时：快速回滚

Nacos 动态配置刷新实战

java
// 方式1：@RefreshScope + @Value（自动刷新）
@RestController
@RefreshScope  // 配置变更时重新创建 Bean
public class RateLimitController {

    @Value("${rate.limit.qps:100}")
    private int qps;

    @GetMapping("/qps")
    public int getQps() {
        return qps;  // 返回最新值，无需重启
    }
}

// 方式2：@ConfigurationProperties（推荐，不需要 @RefreshScope）
@Component
@ConfigurationProperties(prefix = "app.feature")
@RefreshScope
public class FeatureProperties {
    private boolean darkMode = false;
    private int maxRetry = 3;
    private List<String> allowedIps = new ArrayList<>();
    // getters/setters
}

// 方式3：监听配置变更事件
@Component
public class ConfigChangeListener {

    @NacosConfigListener(dataId = "app-config.yaml", groupId = "DEFAULT_GROUP")
    public void onConfigChange(String newConfig) {
        log.info("配置已更新: {}", newConfig);
        // 手动解析并更新本地状态
    }
}

// 方式4：编程式获取配置（适合非 Spring Bean 场景）
@Autowired
private NacosConfigManager nacosConfigManager;

public String getConfig(String dataId) throws NacosException {
    return nacosConfigManager.getConfigService()
        .getConfig(dataId, "DEFAULT_GROUP", 3000);
}
💡
@RefreshScope 的刷新原理 @RefreshScope 创建的 Bean 存入独立的 scope 缓存。配置变更时，/actuator/refresh 端点（或 Nacos 推送）清空该缓存，下次访问 Bean 时重新创建并注入新配置值。注意：刷新会销毁并重建 Bean，若 Bean 中有资源（连接、线程）需在 @PreDestroy 中妥善清理。

面试高频考点

Q1: Nacos 配置中心的推送机制是什么？
Nacos 采用长轮询（Long Polling）机制：客户端发起请求到服务端，服务端挂起请求（默认 29.5 秒），若配置在此期间发生变更，立即响应返回变更数据；若超时未变更则返回空，客户端立即发起下一次长轮询。相比短轮询减少无效请求，相比 WebSocket 实现更简单，实时性接近推送（延迟约 1 秒内）。
Q2: Nacos 和 Apollo 的区别？如何选型？
Nacos：阿里开源，配置中心 + 注册中心二合一，部署简单，与 Spring Cloud Alibaba 深度集成，支持多格式（properties/yaml/json/xml/text），推荐新项目使用。Apollo：携程开源，纯配置中心，功能更完善（配置审计、灰度发布、权限管理、版本对比、发布历史），有完整的管理 UI，适合对配置管理要求极高的企业场景。二者都支持热更新，Apollo 的权限管控更强。
Q3: 配置变更如何保证安全性？
①权限控制：Nacos 支持命名空间隔离（开发/测试/生产），配置访问需 Token 认证；Apollo 有完善的角色权限系统（读/编辑/发布）。②灰度发布：Apollo 支持先发布到部分实例验证效果，再全量推送；Nacos 可通过 beta 标签实现。③配置加密：敏感配置（密码/Key）应加密存储，推荐使用 Jasypt 或自定义加密插件，配置中心仅存储密文。④变更审计：所有配置修改记录操作人和时间，支持回滚。
Q4: 多环境配置如何管理（开发/测试/生产）？
Nacos：通过 Namespace 隔离（每个环境一个 Namespace），DataId 和 Group 可保持一致，通过 spring.cloud.nacos.config.namespace 切换。Apollo：通过 Environment（DEV/FAT/UAT/PRO）隔离，每套环境独立部署 Apollo 服务，apollo.meta 配置对应环境地址。最佳实践：公共配置放 shared namespace（基础配置），应用私有配置放独立 namespace（覆盖公共值），实现配置继承与覆盖。

实战陷阱

⚠️
陷阱1：@Value 不配合 @RefreshScope 导致无法热更新 单独使用 @Value 注入的属性在配置变更后不会自动刷新，因为 Bean 已经实例化。必须在 Bean 类上加 @RefreshScope 才能在配置变更时重建 Bean。使用 @ConfigurationProperties 的 Bean 也需要 @RefreshScope 才能刷新（Spring Boot 2.4+ 中已改善）。
⚠️
陷阱2：配置刷新导致 Bean 重建引发并发问题 @RefreshScope Bean 刷新瞬间，旧 Bean 销毁、新 Bean 创建，期间可能有请求获取到 null 或半初始化的 Bean。对于数据库连接池、线程池等重资源 Bean，应避免使用 @RefreshScope，改用配置监听器手动更新参数（如 HikariPool 的 setMaximumPoolSize）。
⚠️
陷阱3：bootstrap.yml 与 application.yml 加载顺序混淆 Nacos/Apollo 配置需在 bootstrap.yml（Spring Cloud 早期加载）中配置 nacos.config.server-addr 等信息，否则应用启动时无法连接配置中心。Spring Boot 2.4+ 默认禁用 bootstrap context，需添加 spring-cloud-starter-bootstrap 依赖或设置 spring.cloud.bootstrap.enabled=true。
