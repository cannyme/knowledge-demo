三大缓存问题

缓存穿透（Cache Penetration）

问题：大量请求查询不存在的数据，绕过缓存直接打到数据库

⚠️ 典型场景
  • 恶意攻击：id=-1或随机不存在的ID
  • 业务bug：查询已被删除的数据

解决方案

Java
// 方案1：缓存空对象
public Object getWithNullCache(String key) {
    Object value = redis.get(key);
    if (value != null) {
        // 命中缓存（可能是空值标记）
        return "NULL".equals(value) ? null : value;
    }

    // 查询数据库
    value = db.query(key);

    if (value != null) {
        redis.set(key, value, 300);  // 正常过期时间
    } else {
        // 缓存空值，设置较短过期时间
        redis.set(key, "NULL", 60);
    }
    return value;
}

// 方案2：布隆过滤器
public Object getWithBloomFilter(String key) {
    // 先检查布隆过滤器
    if (!bloomFilter.mightContain(key)) {
        return null;  // 一定不存在
    }
    // 可能存在，继续查询
    return getFromCache(key);
}
布隆过滤器原理：位图 + 多个哈希函数
  • 判断可能存在：可能误判（假阳性）
  • 判断一定不存在：一定准确
  • 适用：数据量大、允许少量误判的场景

缓存击穿（Cache Breakdown）

问题：热点Key过期，大量请求同时穿透到数据库

⚠️ 与穿透的区别

穿透：Key不存在 | 击穿：Key存在但过期了

解决方案

Java
// 方案1：互斥锁
public Object getWithMutex(String key) {
    Object value = redis.get(key);
    if (value != null) {
        return value;
    }

    String lockKey = "lock:" + key;
    try {
        boolean locked = redis.setnx(lockKey, "1", 10);
        if (locked) {
            // 获取锁成功，查询数据库
            value = db.query(key);
            redis.set(key, value, 300);
        } else {
            // 获取锁失败，等待重试
            Thread.sleep(50);
            return getWithMutex(key);
        }
    } finally {
        redis.del(lockKey);
    }
    return value;
}

// 方案2：逻辑过期（永不过期）
public Object getWithLogicalExpire(String key) {
    CacheData cacheData = redis.get(key);
    if (cacheData == null) {
        return null;
    }
    // 检查逻辑过期时间
    if (cacheData.getExpireTime() > System.currentTimeMillis()) {
        return cacheData.getData();
    }
    // 已过期，异步刷新
    if (redis.setnx("lock:" + key, "1", 10)) {
        executor.submit(() -> refreshCache(key));
    }
    // 返回旧数据
    return cacheData.getData();
}

缓存雪崩（Cache Avalanche）

问题：大量Key同时过期，或Redis宕机

解决方案

Java
// 方案1：随机过期时间
public void setWithRandomExpire(String key, Object value, int baseExpire) {
    // 基础过期时间 + 随机时间（0-300秒）
    int randomExpire = baseExpire + ThreadLocalRandom.current().nextInt(0, 300);
    redis.set(key, value, randomExpire);
}

// 方案2：多级缓存
public Object getWithMultiCache(String key) {
    // L1: 本地缓存（Caffeine）
    Object value = localCache.get(key);
    if (value != null) {
        return value;
    }
    // L2: Redis缓存
    value = redis.get(key);
    if (value != null) {
        localCache.put(key, value);
        return value;
    }
    // L3: 数据库
    value = db.query(key);
    if (value != null) {
        redis.set(key, value);
        localCache.put(key, value);
    }
    return value;
}
💡 其他方案
  • 熔断降级：数据库压力过大时，降级返回默认值
  • Redis高可用：主从复制 + 哨兵/Cluster
  • 预热机制：系统启动时提前加载热点数据

分布式ID生成

分布式ID要求

  • 全局唯一：不同节点生成的ID不重复
  • 趋势递增：利于数据库索引（B+树）
  • 高性能：低延迟、高QPS
  • 高可用：不依赖单点

雪花算法（Snowflake）- 推荐

graph LR A[1位符号] --> B[41位时间戳] B --> C[10位机器ID] C --> D[12位序列号] style A fill:#3a2e15,stroke:#f0b429,color:#f0b429 style B fill:#0d3d3a,stroke:#4ecdc4,color:#4ecdc4 style C fill:#1a3a1a,stroke:#7ee787,color:#7ee787 style D fill:#15153a,stroke:#82aaff,color:#82aaff
部分位数说明
符号位1位固定为0（正数）
时间戳41位毫秒级，约69年
机器ID10位5位数据中心 + 5位机器
序列号12位每毫秒4096个ID
Java
public class SnowflakeIdGenerator {
    private final long twepoch = 1640995200000L;  // 起始时间
    private final long workerIdBits = 5L;
    private final long datacenterIdBits = 5L;
    private final long sequenceBits = 12L;

    private final long workerId;
    private final long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();

        // 时钟回拨处理
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("时钟回拨");
        }

        if (lastTimestamp == timestamp) {
            // 同一毫秒内，序列号自增
            sequence = (sequence + 1) & 4095;  // 12位最大值
            if (sequence == 0) {
                timestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }

        lastTimestamp = timestamp;
        return ((timestamp - twepoch) << 22)
            | (datacenterId << 17)
            | (workerId << 12)
            | sequence;
    }
}
💡 雪花算法优点
  • 本地生成，无需网络IO，高性能
  • 趋势递增，适合数据库索引
  • 不依赖外部系统
⚠️ 注意事项

时钟回拨问题：可通过序列号等待或使用NTP时间同步解决

号段模式（Leaf）

批量获取ID段，本地缓存在内存中，用完后再次从数据库获取

SQL
CREATE TABLE id_segment (
    biz_tag VARCHAR(64) PRIMARY KEY,
    max_id BIGINT NOT NULL,
    step INT NOT NULL,
    version INT NOT NULL DEFAULT 0,
    update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 插入业务标记
INSERT INTO id_segment (biz_tag, max_id, step) VALUES ('order', 0, 1000);

-- 获取号段（乐观锁）
UPDATE id_segment SET max_id = max_id + step, version = version + 1
WHERE biz_tag = 'order' AND version = #{version};
号段模式流程：
  1. 应用启动时从数据库获取一个号段（如1-1000）
  2. 本地缓存，内存中自增分配ID
  3. 剩余20%时异步获取下一个号段
  4. 用完切换到新号段

Redis原子扣减库存

-- 秒杀库存扣减Lua脚本
local stockKey = KEYS[1]
local userId = ARGV[1]

-- 1. 检查用户是否已购买（一人一单）
if redis.call('sismember', stockKey .. ':users', userId) == 1 then
    return -1  -- 已购买
end

-- 2. 获取库存
local stock = redis.call('get', stockKey)
if stock == nil or tonumber(stock) <= 0 then
    return 0  -- 库存不足
end

-- 3. 扣减库存
redis.call('decr', stockKey)
redis.call('sadd', stockKey .. ':users', userId)

return 1  -- 抢购成功
Java
@Service
public class SeckillService {

    public Result seckill(Long userId, Long seckillId) {
        // 1. 参数校验
        if (!validateParams(userId, seckillId)) {
            return Result.fail("非法请求");
        }

        // 2. Redis原子扣减库存
        Long result = redisTemplate.execute(
            STOCK_DEDUCT_SCRIPT,
            Collections.singletonList("seckill:stock:" + seckillId),
            userId.toString()
        );

        if (result == 0) {
            return Result.fail("库存不足");
        }
        if (result == -1) {
            return Result.fail("已购买过");
        }

        // 3. 发送MQ异步创建订单
        OrderMessage message = new OrderMessage(userId, seckillId);
        rocketMQTemplate.asyncSend("seckill-order-topic", message, new SendCallback() {
            @Override
            public void onSuccess(SendResult sendResult) {
                log.info("订单消息发送成功");
            }
            @Override
            public void onException(Throwable e) {
                // 失败回滚库存
                rollbackStock(seckillId, userId);
            }
        });

        // 4. 返回排队中（异步处理）
        return Result.success("排队中", orderToken);
    }
}

// 方案1：Hash后Base62编码
public String shortUrlV1(String longUrl) {
    // MD5哈希取前8位
    String hash = DigestUtils.md5Hex(longUrl).substring(0, 8);
    long num = Long.parseLong(hash, 16);
    return base62Encode(num);
}

// 方案2：发号器（推荐，无Hash冲突）
public String shortUrlV2() {
    long id = snowflake.nextId();
    return base62Encode(id);
}

// Base62编码（0-9, a-z, A-Z）
private String base62Encode(long num) {
    String chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    StringBuilder sb = new StringBuilder();
    while (num > 0) {
        sb.append(chars.charAt((int)(num % 62)));
        num /= 62;
    }
    return sb.reverse().toString();
}
容量计算：
  • 6位Base62：62^6 ≈ 568亿
  • 7位Base62：62^7 ≈ 3.5万亿
  • 一般6位足够，可根据需求选择

设计原则总结

高可用设计

  • 冗余部署：多实例、多可用区
  • 故障转移：自动切换、熔断降级
  • 限流保护：防止系统过载
  • 监控告警：及时发现问题

高性能设计

  • 缓存优化：多级缓存、本地缓存
  • 异步处理：消息队列削峰
  • 数据库优化：索引、分库分表、读写分离
  • 连接池：复用连接资源

可扩展设计

  • 无状态服务：水平扩展基础
  • 服务拆分：微服务架构
  • 数据分片：水平切分
  • 弹性伸缩：自动扩缩容
