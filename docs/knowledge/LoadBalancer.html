负载均衡策略

负载均衡是将请求分发到多个服务实例的技术，用于提高系统吞吐量和可用性。本文详解各类负载均衡算法及Spring Cloud中的实现方案。

❓

为什么需要负载均衡

随着业务增长，单机服务无法满足高并发需求，需要通过水平扩展部署多个服务实例。负载均衡器作为流量入口，负责将请求合理分配到各实例。

flowchart TD A[客户端请求] --> B[负载均衡器] B --> C[服务实例1] B --> D[服务实例2] B --> E[服务实例3] B --> F[服务实例N] C --> G[(数据库)] D --> G E --> G F --> G

负载均衡的核心价值：

  • 高可用：单点故障自动剔除，请求转发到健康实例
  • 高并发：多实例分担流量，提升系统整体处理能力
  • 高性能：根据服务器负载动态分配，避免单点过载
  • 横向扩展：无缝添加新实例，应对业务增长
📂

根据实现位置和方式，负载均衡可分为以下几类：

分类 代表产品 特点 适用场景

负载均衡算法详解

负载均衡算法决定了请求如何分配到各个服务实例，不同算法适用于不同场景。

1. 轮询（Round Robin）
按顺序依次将请求分发到每个实例，循环往复。实现简单，适用于服务器性能相近的场景。
Java
/**
 * 轮询负载均衡算法
 * 按顺序依次分发请求
 */
public class RoundRobinLoadBalancer {
    private final List<ServiceInstance> instances;
    private final AtomicInteger position = new AtomicInteger(0);

    public RoundRobinLoadBalancer(List<ServiceInstance> instances) {
        this.instances = instances;
    }

    public ServiceInstance choose() {
        if (instances.isEmpty()) return null;
        int pos = Math.abs(position.getAndIncrement() % instances.size());
        return instances.get(pos);
    }
}
2. 加权轮询（Weighted Round Robin）
根据服务器权重比例分配请求，权重高的服务器获得更多请求。适用于服务器性能差异较大的场景。
Java
/**
 * 加权轮询负载均衡
 * 按权重比例分发请求
 */
public class WeightedRoundRobinLoadBalancer {
    private final List<ServiceInstance> instances;
    private final AtomicInteger position = new AtomicInteger(0);
    private List<ServiceInstance> expandedList;

    public WeightedRoundRobinLoadBalancer(List<ServiceInstance> instances) {
        this.instances = instances;
        this.expandedList = expandInstances();
    }

    // 按权重展开实例列表
    private List<ServiceInstance> expandInstances() {
        List<ServiceInstance> expanded = new ArrayList<>();
        for (ServiceInstance instance : instances) {
            for (int i = 0; i < instance.getWeight(); i++) {
                expanded.add(instance);
            }
        }
        return expanded;
    }

    public ServiceInstance choose() {
        if (expandedList.isEmpty()) return null;
        int pos = Math.abs(position.getAndIncrement() % expandedList.size());
        return expandedList.get(pos);
    }
}
3. 平滑加权轮询（Smooth Weighted Round Robin）
Nginx默认算法，解决加权轮询请求分布不均的问题。通过动态调整当前权重，使请求分布更加平滑均匀。
Java
/**
 * 平滑加权轮询 - Nginx默认算法
 * 原理：currentWeight += weight，选最大者，然后减去totalWeight
 */
public class SmoothWeightedRoundRobinLoadBalancer {
    private final List<ServiceInstance> instances;
    private final Map<ServiceInstance, Integer> currentWeights = new ConcurrentHashMap<>();

    public SmoothWeightedRoundRobinLoadBalancer(List<ServiceInstance> instances) {
        this.instances = instances;
        for (ServiceInstance instance : instances) {
            currentWeights.put(instance, 0);
        }
    }

    public ServiceInstance choose() {
        if (instances.isEmpty()) return null;

        int totalWeight = instances.stream()
            .mapToInt(ServiceInstance::getWeight).sum();
        ServiceInstance selected = null;
        int maxWeight = Integer.MIN_VALUE;

        // currentWeight += weight，找出最大值
        for (ServiceInstance instance : instances) {
            int currentWeight = currentWeights.get(instance) + instance.getWeight();
            currentWeights.put(instance, currentWeight);

            if (currentWeight > maxWeight) {
                maxWeight = currentWeight;
                selected = instance;
            }
        }

        // 被选中的实例currentWeight -= totalWeight
        currentWeights.put(selected, currentWeights.get(selected) - totalWeight);
        return selected;
    }
}
4. 随机（Random）
随机选择一个服务实例。实现简单，在大量请求下分布均匀，适用于服务器性能相近的场景。
Java
/**
 * 随机负载均衡算法
 */
public class RandomLoadBalancer {
    private final List<ServiceInstance> instances;
    private final Random random = new Random();

    public RandomLoadBalancer(List<ServiceInstance> instances) {
        this.instances = instances;
    }

    public ServiceInstance choose() {
        if (instances.isEmpty()) return null;
        int pos = random.nextInt(instances.size());
        return instances.get(pos);
    }
}
5. 加权随机（Weighted Random）
根据权重随机选择实例，权重高的实例被选中的概率更大。适用于服务器性能不均的场景。
Java
/**
 * 加权随机负载均衡
 */
public class WeightedRandomLoadBalancer {
    private final List<ServiceInstance> instances;
    private final Random random = new Random();
    private final int totalWeight;
    private final List<Integer> weights;

    public WeightedRandomLoadBalancer(List<ServiceInstance> instances) {
        this.instances = instances;
        this.weights = new ArrayList<>();
        int sum = 0;
        for (ServiceInstance instance : instances) {
            sum += instance.getWeight();
            weights.add(sum);
        }
        this.totalWeight = sum;
    }

    public ServiceInstance choose() {
        if (instances.isEmpty()) return null;

        int randomWeight = random.nextInt(totalWeight);
        for (int i = 0; i < weights.size(); i++) {
            if (randomWeight < weights.get(i)) {
                return instances.get(i);
            }
        }
        return instances.get(0);
    }
}
6. 最少连接（Least Connections）
选择当前连接数最少的服务器。适合长连接场景（如WebSocket），能够动态感知服务器负载。
Java
/**
 * 最少连接数负载均衡
 * 适合长连接场景
 */
public class LeastConnectionsLoadBalancer {
    private final Map<ServiceInstance, AtomicInteger> connections = new ConcurrentHashMap<>();

    public ServiceInstance choose(List<ServiceInstance> instances) {
        if (instances.isEmpty()) return null;

        ServiceInstance selected = null;
        int minConnections = Integer.MAX_VALUE;

        for (ServiceInstance instance : instances) {
            int conn = connections.computeIfAbsent(
                instance, k -> new AtomicInteger(0)).get();
            if (conn < minConnections) {
                minConnections = conn;
                selected = instance;
            }
        }

        connections.get(selected).incrementAndGet();
        return selected;
    }

    public void release(ServiceInstance instance) {
        connections.get(instance).decrementAndGet();
    }
}
7. IP哈希（IP Hash）
根据客户端IP地址计算哈希值，选择对应服务器。同一IP的请求总是路由到同一服务器，适合需要会话保持的场景。
Java
/**
 * IP Hash负载均衡
 * 相同IP总是路由到同一服务器
 */
public class IpHashLoadBalancer {
    private final List<ServiceInstance> instances;

    public IpHashLoadBalancer(List<ServiceInstance> instances) {
        this.instances = instances;
    }

    public ServiceInstance choose(String clientIp) {
        if (instances.isEmpty()) return null;

        int hash = clientIp.hashCode();
        int index = Math.abs(hash) % instances.size();
        return instances.get(index);
    }
}
8. 一致性哈希（Consistent Hash）
使用一致性哈希环，相同请求Key总是路由到同一服务器。当服务器增减时，只有少量请求需要重新路由。适合缓存等有状态场景。
Java
/**
 * 一致性哈希负载均衡
 * 适合有状态场景（缓存、会话）
 */
public class ConsistentHashLoadBalancer {
    // 虚拟节点数，解决数据倾斜
    private static final int VIRTUAL_NODES = 150;
    private final TreeMap<Integer, ServiceInstance> ring = new TreeMap<>();

    public ConsistentHashLoadBalancer(List<ServiceInstance> instances) {
        for (ServiceInstance instance : instances) {
            // 每个实例创建多个虚拟节点
            for (int i = 0; i < VIRTUAL_NODES; i++) {
                String virtualNode = instance.getHost() + ":"
                    + instance.getPort() + "#" + i;
                int hash = hash(virtualNode);
                ring.put(hash, instance);
            }
        }
    }

    public ServiceInstance choose(String key) {
        if (ring.isEmpty()) return null;

        int hash = hash(key);
        // 找到第一个大于等于该hash的节点
        Map.Entry<Integer, ServiceInstance> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            // 环形结构，回到第一个节点
            entry = ring.firstEntry();
        }
        return entry.getValue();
    }

    private int hash(String key) {
        // 可使用MurmurHash等更好的哈希算法
        return key.hashCode() & Integer.MAX_VALUE;
    }
}
算法 优点 缺点 适用场景
轮询 简单公平 不考虑服务器差异 服务器性能相近
加权轮询 考虑权重 分布可能不均 性能差异明显
平滑加权轮询 分布均匀 实现稍复杂 推荐使用
随机 简单 短时间可能不均 大量请求场景
最少连接 动态感知负载 需维护连接数 长连接场景
IP哈希 会话保持 可能分布不均 需要会话保持
一致性哈希 节点变化影响小 实现复杂 缓存等有状态场景

Ribbon负载均衡（Spring Cloud Netflix）

⚠️ 维护模式提示
Ribbon 已进入维护模式，Spring Cloud 2020+ 版本建议使用 Spring Cloud LoadBalancer 替代。

Ribbon 是 Netflix 开源的客户端负载均衡器，与 Eureka、Feign 集成，实现服务间调用的负载均衡。

flowchart LR A[服务消费者] -->|1. 从Eureka获取服务列表| B[Eureka注册中心] A -->|2. 本地选择实例| C[实例1] A --> D[实例2] A --> E[实例3]

Ribbon内置负载均衡策略：

策略类 说明
RoundRobinRule 轮询（默认）
RandomRule 随机
WeightedResponseTimeRule 根据响应时间加权
RetryRule 重试机制
BestAvailableRule 选择最空闲实例
AvailabilityFilteringRule 过滤故障实例
ZoneAvoidanceRule 根据区域和可用性选择
YAML
# Ribbon配置示例
user-service:
  ribbon:
    # 负载均衡策略
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
    # 连接超时
    ConnectTimeout: 1000
    # 读取超时
    ReadTimeout: 3000
    # 同一实例重试次数
    MaxAutoRetries: 1
    # 切换实例重试次数
    MaxAutoRetriesNextServer: 2

Spring Cloud LoadBalancer

        initial-delay: 0
        interval: 25s

自定义负载均衡策略

通过继承 AbstractLoadBalancerRule 实现自定义 Ribbon 负载均衡策略：

Java
/**
 * 自定义Ribbon负载均衡策略
 * 基于版本号的灰度路由
 */
public class VersionAwareRule extends AbstractLoadBalancerRule {

    @Override
    public Server choose(Object key) {
        ILoadBalancer lb = getLoadBalancer();
        List<Server> servers = lb.getAllServers();

        // 从请求头获取目标版本
        String targetVersion = GrayContextHolder.getVersion();

        // 筛选匹配版本的服务
        List<Server> matched = servers.stream()
            .filter(server -> {
                Map<String, String> metadata =
                    ((NacosServer) server).getMetadata();
                return targetVersion.equals(metadata.get("version"));
            })
            .collect(Collectors.toList());

        if (matched.isEmpty()) {
            // 降级到默认版本
            matched = servers;
        }

        // 随机选择
        return matched.get(ThreadLocalRandom.current()
            .nextInt(matched.size()));
    }

    @Override
    public void initWithNiwsConfig(IClientConfig clientConfig) {}
}
Java
/**
 * Ribbon配置类
 * 注意：此类不能放在主启动类同包或子包下
 * 避免被@ComponentScan扫描到全局生效
 */
@Configuration
public class RibbonConfig {

    @Bean
    public IRule ribbonRule() {
        // 使用自定义灰度路由规则
        return new VersionAwareRule();
    }
}

实现 ReactorServiceInstanceLoadBalancer 接口自定义 Spring Cloud LoadBalancer 策略：

Java
/**
 * 自定义Spring Cloud LoadBalancer
 * 基于权重的随机选择
 */
public class WeightedRandomLoadBalancer
        implements ReactorServiceInstanceLoadBalancer {

    private final ObjectProvider<ServiceInstanceListSupplier>
        serviceInstanceSupplier;
    private final String serviceId;

    public WeightedRandomLoadBalancer(
            ObjectProvider<ServiceInstanceListSupplier> supplier,
            String serviceId) {
        this.serviceInstanceSupplier = supplier;
        this.serviceId = serviceId;
    }

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return serviceInstanceSupplier.getIfAvailable()
            .get(request)
            .next()
            .map(this::getInstanceResponse);
    }

    private Response<ServiceInstance> getInstanceResponse(
            List<ServiceInstance> instances) {
        if (instances.isEmpty()) {
            return new EmptyResponse();
        }

        // 按权重随机选择
        int totalWeight = instances.stream()
            .mapToInt(this::getWeight)
            .sum();
        int random = ThreadLocalRandom.current().nextInt(totalWeight);

        for (ServiceInstance instance : instances) {
            random -= getWeight(instance);
            if (random < 0) {
                return new DefaultResponse(instance);
            }
        }
        return new DefaultResponse(instances.get(0));
    }

    private int getWeight(ServiceInstance instance) {
        String weight = instance.getMetadata().get("weight");
        return weight != null ? Integer.parseInt(weight) : 1;
    }
}

完整的灰度发布负载均衡实现，支持基于用户ID、Header等多种路由策略：

Java
/**
 * 灰度发布负载均衡策略
 * 支持：Header版本号、用户ID白名单、百分比流量
 */
@Component
public class GrayReleaseLoadBalancer
        implements ReactorServiceInstanceLoadBalancer {

    @Autowired
    private GrayRuleProperties grayProperties;

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        DefaultRequest<RequestDataContext> ctx =
            (DefaultRequest<RequestDataContext>) request;
        HttpHeaders headers = ctx.getContext()
            .getClientRequest().getHeaders();

        String grayVersion = headers.getFirst("X-Gray-Version");
        String userId = headers.getFirst("X-User-Id");

        return serviceInstanceSupplier.getIfAvailable()
            .get(request)
            .next()
            .map(instances -> {
                List<ServiceInstance> grayInstances = instances.stream()
                    .filter(inst -> grayVersion.equals(
                        inst.getMetadata().get("version")))
                    .collect(Collectors.toList());

                // 白名单用户强制走灰度
                if (grayProperties.getWhiteList().contains(userId)) {
                    return new DefaultResponse(grayInstances.get(0));
                }

                // 按比例分配流量
                if (ThreadLocalRandom.current().nextInt(100)
                        < grayProperties.getPercent()) {
                    return new DefaultResponse(grayInstances.get(0));
                }

                // 返回稳定版本
                return new DefaultResponse(instances.get(0));
            });
    }
}

最佳实践

1. 选择合适的算法
服务器性能相近用轮询，性能差异大用加权轮询，有状态服务用一致性哈希，长连接用最少连接。
2. 健康检查
配合健康检查机制，自动剔除故障实例。Ribbon 默认集成 Ping 检查，Spring Cloud LoadBalancer 支持 reactive 健康检查。
3. 服务列表缓存
启用本地缓存避免频繁查询注册中心，但注意设置合理的过期时间，保证服务上下线能及时感知。
4. 重试机制
配置合理的重试策略，区分可重试异常（网络超时）和不可重试异常（业务异常），避免无效重试。
5. 监控与调优
监控各实例的 QPS、响应时间、错误率，根据实际数据调整权重和策略，持续优化负载分布。
6. 平滑升级
使用灰度发布逐步切换流量，配合自定义负载均衡策略实现金丝雀发布、A/B测试等场景。
💡 生产环境建议
外部网关使用 Nginx（轮询/权重），微服务间调用使用 Spring Cloud LoadBalancer（推荐）或 Ribbon，配合熔断器实现完整的服务治理方案。
